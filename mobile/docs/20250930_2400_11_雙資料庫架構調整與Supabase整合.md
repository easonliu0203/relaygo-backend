# Relay GO 雙資料庫架構調整與 Supabase 整合

**開發時間**: 2025/09/30 24:00  
**開發者**: Augment Agent  
**功能編號**: 11  
**主要任務**: 實作雙資料庫架構，將業務邏輯從 Firebase 遷移到 Supabase/PostgreSQL

## 📋 **任務概述**

本次開發實作了重大的技術架構調整，從單一 Firebase 架構改為雙資料庫架構：
- **Firebase**: 負責認證、推播通知、即時聊天、檔案儲存、定位追蹤
- **Supabase/PostgreSQL**: 負責訂單管理、金額計算、結帳處理、退款管理、獎金計算、報表生成

## 🎯 **架構調整需求**

### **原有架構問題**
1. 所有資料都存儲在 Firebase Firestore
2. 價目表配置從 Firestore 或模擬資料獲取
3. 模擬支付邏輯在客戶端處理
4. 缺乏完整的業務邏輯和財務管理

### **新架構設計**
1. **Firebase**: 專注於即時性和認證功能
2. **Supabase**: 處理所有業務邏輯和資料持久化
3. **管理後台**: 控制封測階段設定和自動支付
4. **API 層**: 提供統一的資料訪問介面

## 📊 **資料庫結構設計**

### **Supabase 資料表**

**vehicle_pricing 表**（價格配置）
```sql
CREATE TABLE vehicle_pricing (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  vehicle_type VARCHAR(10) NOT NULL CHECK (vehicle_type IN ('A', 'B', 'C', 'D')),
  duration_hours INTEGER NOT NULL CHECK (duration_hours IN (6, 8)),
  base_price DECIMAL(10,2) NOT NULL,
  overtime_rate DECIMAL(10,2) NOT NULL,
  is_active BOOLEAN DEFAULT true,
  effective_from TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  effective_until TIMESTAMP WITH TIME ZONE
);
```

**bookings 表**（訂單管理）
```sql
CREATE TABLE bookings (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  customer_id UUID NOT NULL REFERENCES users(id),
  driver_id UUID REFERENCES users(id),
  booking_number VARCHAR(20) UNIQUE NOT NULL,
  status VARCHAR(20) NOT NULL DEFAULT 'pending',
  pickup_location TEXT NOT NULL,
  pickup_latitude DECIMAL(10, 8),
  pickup_longitude DECIMAL(11, 8),
  destination TEXT,
  base_price DECIMAL(10,2) NOT NULL,
  total_amount DECIMAL(10,2) NOT NULL,
  deposit_amount DECIMAL(10,2) NOT NULL
);
```

**payments 表**（支付記錄）
```sql
CREATE TABLE payments (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  booking_id UUID NOT NULL REFERENCES bookings(id),
  payment_type VARCHAR(20) NOT NULL CHECK (payment_type IN ('deposit', 'balance', 'tip', 'refund')),
  amount DECIMAL(10,2) NOT NULL,
  status VARCHAR(20) NOT NULL DEFAULT 'pending',
  payment_provider VARCHAR(20) NOT NULL DEFAULT 'mock',
  is_test_mode BOOLEAN DEFAULT true,
  transaction_id VARCHAR(100),
  processed_at TIMESTAMP WITH TIME ZONE,
  confirmed_at TIMESTAMP WITH TIME ZONE
);
```

## 💡 **開發心得與經驗**

### **架構設計心得**
1. **職責分離**: Firebase 專注即時性，Supabase 處理業務邏輯
2. **向後兼容**: 保留原有 Firestore 方法，確保平滑遷移
3. **降級機制**: API 不可用時自動使用模擬資料
4. **統一介面**: 通過 Provider 層統一資料訪問邏輯

### **技術實作經驗**
1. **車型映射**: 資料庫使用 A/B/C/D，客戶端使用 large/small
2. **封測設計**: 管理後台控制自動支付，客戶端不顯示設定
3. **錯誤處理**: 完整的異常捕獲和用戶友好的錯誤提示
4. **狀態同步**: 支付成功後正確更新訂單狀態

### **遇到的困難**
1. **模型不匹配**: BookingRequest 缺少新欄位，需要擴展
2. **API 設計**: 需要考慮客戶端和資料庫的資料格式差異
3. **狀態管理**: 雙 API 支援需要仔細處理狀態轉換
4. **依賴管理**: 需要添加 http 依賴並重新生成代碼

### **解決方法**
1. **漸進式遷移**: 先添加新方法，再逐步替換舊方法
2. **充分測試**: 每個階段都進行編譯和功能測試
3. **文檔參考**: 深入研究 Supabase 和 PostgreSQL 文檔
4. **錯誤預防**: 使用 try-catch 和降級機制確保穩定性
