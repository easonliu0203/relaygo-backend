## 🔧 根本原因

### 問題 1: Supabase JOIN 語法不正確

原始代碼使用的 JOIN 語法：
```typescript
.select(`
  *,
  user_profiles (
    first_name,
    last_name,
    phone,
    avatar_url
  )
`)
```

這個語法在 Supabase PostgREST 中需要正確的外鍵關係配置才能工作。

### 問題 2: 分離查詢時 `.in()` 查詢失敗

修改後的代碼使用分離查詢：
```typescript
// 1. 查詢用戶
const { data: users } = await db.supabase
  .from('users')
  .select('*')
  .eq('role', 'customer');

// 2. 查詢 profiles
const userIds = users?.map(u => u.id) || [];
const { data: profiles } = await db.supabase
  .from('user_profiles')
  .select('*')
  .in('user_id', userIds);  // ❌ 返回空數組
```

**可能的原因**:
1. `user_id` 欄位類型不匹配（UUID vs 字符串）
2. Supabase 客戶端配置問題
3. RLS 政策阻止查詢

---

## 💡 解決方案

### 方案 1: 使用 Service Role Key（推薦）

確保 API 使用 Service Role Key 而不是 Anon Key：

```typescript
// web-admin/src/lib/supabase.ts
const supabaseServiceKey = process.env.SUPABASE_SERVICE_ROLE_KEY!;
const supabase = createClient(supabaseUrl, supabaseServiceKey);
```

### 方案 2: 檢查 RLS 政策

確認 `user_profiles` 表的 RLS 政策允許 service_role 訪問：

```sql
-- 檢查 RLS 狀態
SELECT relrowsecurity FROM pg_class WHERE relname = 'user_profiles';

-- 檢查政策
SELECT * FROM pg_policies WHERE tablename = 'user_profiles';
```

### 方案 3: 使用原生 SQL 查詢

如果 PostgREST 查詢有問題，可以使用原生 SQL：

```typescript
const { data, error } = await db.supabase.rpc('get_customers_with_profiles', {
  p_limit: limit,
  p_offset: offset,
});
```

創建 SQL 函數：
```sql
CREATE OR REPLACE FUNCTION get_customers_with_profiles(
  p_limit INT DEFAULT 100,
  p_offset INT DEFAULT 0
)
RETURNS TABLE (
  id UUID,
  email TEXT,
  first_name TEXT,
  last_name TEXT,
  phone TEXT,
  ...
) AS $$
BEGIN
  RETURN QUERY
  SELECT
    u.id,
    u.email,
    up.first_name,
    up.last_name,
    up.phone,
    ...
  FROM users u
  LEFT JOIN user_profiles up ON u.id = up.user_id
  WHERE u.role = 'customer'
  ORDER BY u.created_at DESC
  LIMIT p_limit
  OFFSET p_offset;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```
