## 📋 架構設計

### 安全架構

```
客戶端/司機端 App (Firebase Auth)
    ↓ 發送 Firebase UID
中台 API (Next.js /api/profile/upsert)
    ↓ 驗證 Token + 使用 Service Role Key
Supabase (啟用 RLS，只允許 service_role)
```

### 為什麼這個架構是安全的？

1. **RLS 保護**: 
   - `user_profiles` 表啟用 RLS
   - 只有 `service_role` 可以訪問
   - 客戶端使用 `anon key` 無法直接訪問

2. **Service Role Key 保護**:
   - Service Role Key 只存在於後端 `.env.local`
   - 永遠不會暴露給客戶端
   - 客戶端無法獲取或使用

3. **Firebase Auth 驗證**:
   - 用戶必須先通過 Firebase Auth 登入
   - 中台 API 可以驗證 Firebase Token（可選）
   - 確保請求來自已認證的用戶

4. **API 層控制**:
   - 所有資料操作通過中台 API
   - API 可以實作額外的業務邏輯和驗證
   - 可以記錄日誌、限流等

## 🔧 實作內容

### 1. Supabase RLS 政策

**文件**: `supabase/migrations/20251010_fix_user_profiles_rls_for_service_role.sql`

**內容**:
```sql
-- 啟用 RLS
ALTER TABLE user_profiles ENABLE ROW LEVEL SECURITY;

-- 只允許 service_role 訪問
CREATE POLICY "service_role_full_access"
ON user_profiles
FOR ALL
TO service_role
USING (true)
WITH CHECK (true);
```


## 💡 開發心得

### 1. 安全第一

**教訓**: 永遠不要為了便利而犧牲安全性。

**正確做法**:
- ✅ 保留 RLS
- ✅ 使用 Service Role Key（只在後端）
- ✅ 通過 API 層控制訪問

**錯誤做法**:
- ❌ 禁用 RLS
- ❌ 暴露 Service Role Key 給客戶端
- ❌ 客戶端直接訪問資料庫

### 2. 分層架構

**優點**:
- 清晰的職責分離
- 易於維護和擴展
- 可以添加額外的業務邏輯

**架構層次**:
1. **客戶端層**: UI 和用戶交互
2. **API 層**: 業務邏輯和驗證
3. **資料層**: 資料存儲和 RLS

### 3. 漸進式遷移

**策略**:
- 先實作核心功能（個人資料）
- 驗證架構可行性
- 逐步遷移其他功能

**下一步**:
