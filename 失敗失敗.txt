🎛 為什麼需要這個「即時同步開關」

在實務環境中：

Trigger + pg_net 即時通知 適合低延遲。
Cron 補償任務 適合穩定容錯。

但即時機制有時會遇到：
Edge Function 維護 / Deploy 中斷
API Rate 限制
Service Role Key 變更失效
pg_net HTTP 連線 timeout 導致交易延遲

因此，做一個**公司端開關（管理後台切換）**能讓你：
緊急關掉 trigger（退回只用 cron 模式）
維護完再一鍵啟用，不需 DROP / CREATE trigger

✅ 整體架構演化
階段	機制	功能	延遲	可靠性
初期	只用 Cron	每分鐘掃描	60 秒	高
完整	Trigger + Cron	即時觸發 + 補償	1~3 秒	非常高
緊急維護	關閉 Trigger	暫時退回 Cron	60 秒	高


🧭 專案任務指令：建立「即時 + 補償」雙保險通知架構

目標
在現有 Supabase → Firestore 同步機制中，新增即時通知（Trigger + pg_net HTTP），並保留 Cron 作為補償機制。

🪜 任務步驟

1.確認現有 Cron 運作正常

確認 sync-orders-to-firestore 任務每分鐘執行成功。

執行次數與最後執行時間需正常更新。

2.新增即時通知機制

在 Supabase 建立 Trigger，當 bookings 或 outbox 資料更新時，
透過 pg_net 直接發送 HTTP 請求到 Edge Function。

HTTP 目標：sync-to-firestore Edge Function URL。

目標：在 1–3 秒內將變動即時同步至 Firestore。

3.確認即時與補償可並行運作

Cron 任務保留（每分鐘跑一次），作為備援補償。

Trigger 進行即時同步，降低延遲。

4.實現退場機制

建立一個「一鍵停用 Trigger」腳本（DROP TRIGGER）。

當 Edge Function 維護、Rate 限制或服務異常時，可快速退回只用 Cron 模式。

保留可重新啟用的腳本（CREATE TRIGGER）。

5.公司端開關控制

在管理後台新增「即時同步開關」：

開啟 → 啟用 Trigger + pg_net 即時通知

關閉 → 停用 Trigger，只保留 Cron

開關狀態寫入 system_settings 或 admin_config 資料表。

開關可以放在公司端>派單設定頁面
但是該派單設定頁面目前暫無實際功能與頁面

6.Edge Function 冪等與重試

Edge Function sync-to-firestore 必須可安全重複執行（冪等）。

若 HTTP 請求失敗或超時，可由 Cron 自動補上。

完整	Trigger + Cron	即時通知 + 補償	1–3 秒	非常高
維護	關閉 Trigger	退回 Cron 模式	約 60 秒	高
