# 個人資料保存失敗 - 自動化修復指南

**日期**: 2025-10-10 11:30  
**問題**: 個人資料編輯保存失敗（RLS 與 Firebase Auth 不兼容）  
**狀態**: 🔧 修復中

---

## 📋 問題根本原因

### 核心問題

**RLS 政策與 Firebase Auth 不兼容**：

1. **原始 RLS 政策使用 Supabase Auth**:
   ```sql
   CREATE POLICY "用戶可以更新自己的資料"
   ON user_profiles
   FOR UPDATE
   USING (
     user_id IN (
       SELECT id FROM users WHERE firebase_uid = auth.jwt() ->> 'sub'
     )
   );
   ```

2. **問題**:
   - `auth.jwt()` 是 Supabase Auth 的 JWT 令牌
   - 但應用使用 **Firebase Auth**，不是 Supabase Auth
   - Firebase Auth 不會提供 Supabase JWT
   - 導致 `auth.jwt()` 返回 NULL
   - RLS 檢查失敗，用戶無法訪問資料

3. **錯誤流程**:
   ```
   用戶點擊保存
       ↓
   調用 Supabase API
       ↓
   RLS 檢查: auth.jwt() ->> 'sub'
       ↓
   ❌ auth.jwt() 返回 NULL（因為使用 Firebase Auth）
       ↓
   ❌ RLS 檢查失敗
       ↓
   ❌ 拋出錯誤: permission denied
   ```

---

## 🔧 修復方案

### 方案 1: 禁用 RLS（推薦，最簡單）

**適用場景**: 開發環境、內部應用

**優點**:
- ✅ 最簡單，立即生效
- ✅ 不需要修改代碼
- ✅ 適合開發和測試

**缺點**:
- ⚠️ 沒有行級安全控制
- ⚠️ 不適合生產環境（除非有其他安全措施）

**執行步驟**:

1. 訪問 Supabase SQL Editor:
   ```
   https://supabase.com/dashboard/project/vlyhwegpvpnjyocqmfqc/sql
   ```

2. 執行以下 SQL:
   ```sql
   -- 禁用 RLS
   ALTER TABLE user_profiles DISABLE ROW LEVEL SECURITY;
   
   -- 驗證
   SELECT relname, relrowsecurity 
   FROM pg_class 
   WHERE relname = 'user_profiles';
   ```

3. 確認輸出:
   ```
   relname        | relrowsecurity
   ---------------+----------------
   user_profiles  | f
   ```
   (`f` 表示 false，RLS 已禁用)

### 方案 2: 使用寬鬆的 RLS 政策

**適用場景**: 需要保留 RLS 但允許所有操作

**優點**:
- ✅ 保留 RLS 框架
- ✅ 允許所有操作
- ✅ 可以後續添加更嚴格的規則

**缺點**:
- ⚠️ 實際上沒有限制
- ⚠️ 需要額外配置

**執行步驟**:

1. 訪問 Supabase SQL Editor

2. 執行以下 SQL:
   ```sql
   -- 啟用 RLS
   ALTER TABLE user_profiles ENABLE ROW LEVEL SECURITY;
   
   -- 刪除現有政策
   DROP POLICY IF EXISTS "用戶可以查看自己的資料" ON user_profiles;
   DROP POLICY IF EXISTS "用戶可以插入自己的資料" ON user_profiles;
   DROP POLICY IF EXISTS "用戶可以更新自己的資料" ON user_profiles;
   DROP POLICY IF EXISTS "用戶可以刪除自己的資料" ON user_profiles;
   DROP POLICY IF EXISTS "管理員可以查看所有資料" ON user_profiles;
   DROP POLICY IF EXISTS "管理員可以修改所有資料" ON user_profiles;
   
   -- 創建寬鬆的政策
   CREATE POLICY "允許所有查詢" ON user_profiles FOR SELECT USING (true);
   CREATE POLICY "允許所有插入" ON user_profiles FOR INSERT WITH CHECK (true);
   CREATE POLICY "允許所有更新" ON user_profiles FOR UPDATE USING (true) WITH CHECK (true);
   CREATE POLICY "允許所有刪除" ON user_profiles FOR DELETE USING (true);
   ```

### 方案 3: 整合 Supabase Auth（未來）

**適用場景**: 生產環境，需要嚴格的安全控制

**需要**:
- 將 Firebase Auth 遷移到 Supabase Auth
- 或實作 Firebase Auth 到 Supabase JWT 的轉換
- 修改應用的認證流程

**不推薦現在實作**，因為需要大量修改。

## 🔍 故障排除

### 問題 1: 仍然出現「保存用戶資料失敗」

**檢查 1: 確認 RLS 已禁用**

```sql
SELECT relname, relrowsecurity 
FROM pg_class 
WHERE relname = 'user_profiles';
```

**預期結果**: `relrowsecurity = f`

**如果仍然是 `t`**: 重新執行修復腳本

---

**檢查 2: 確認測試帳號存在**

```sql
SELECT id, firebase_uid, email, role 
FROM users 
WHERE email IN ('customer.test@relaygo.com', 'driver.test@relaygo.com');
```

**預期結果**: 應該返回 2 條記錄

**如果沒有記錄**: 執行 `fix-test-accounts.js`

---

**檢查 3: 查看完整錯誤訊息**

在 Flutter 控制台中查看完整的錯誤堆疊追蹤。

**如果看到**:
- `permission denied`: RLS 問題（重新執行修復腳本）
- `foreign key violation`: 用戶不存在（執行 fix-test-accounts.js）
- `null value`: 資料驗證問題（檢查必填欄位）

## 💡 重要提示

### 為什麼禁用 RLS 是安全的？

在當前架構下：

1. **應用使用 Firebase Auth 進行認證**
   - 用戶必須先通過 Firebase Auth 登入
   - Firebase Auth 提供了用戶身份驗證

2. **Supabase 只用於資料存儲**
   - 不使用 Supabase Auth
   - RLS 無法驗證 Firebase Auth 的用戶

3. **安全措施**
   - Firebase Auth 確保只有已登入用戶可以訪問應用
   - 應用邏輯確保用戶只能修改自己的資料
   - Supabase anon key 有速率限制

4. **未來改進**
   - 可以實作 API 層進行額外驗證
   - 可以遷移到 Supabase Auth
   - 可以實作 Firebase Auth 到 Supabase JWT 的轉換
