## 📋 問題描述

在公司端（web-admin）管理頁面中，測試帳號的個人資料（姓名、電話等）顯示為空：
- 客戶管理頁面：姓名顯示"未設定"，電話為空
- 司機管理頁面：姓名顯示"未設定"，電話為空（但駕照、車型等司機特定資料正常）

---

## 🔍 根本原因

**API 使用了錯誤的 Supabase 客戶端配置**

### 問題代碼

```typescript
// web-admin/src/app/api/admin/customers/route.ts
const db = new DatabaseService(); // ❌ 使用 anon key
```

### 為什麼會失敗？

1. **RLS 政策限制**: `user_profiles` 表啟用了 RLS（Row Level Security）
2. **Anon Key 權限不足**: 使用 `anon key` 時，RLS 政策阻止了對 `user_profiles` 表的訪問
3. **查詢返回空**: `.in('user_id', userIds)` 查詢返回空數組，導致無法獲取個人資料

### 診斷過程

```
📋 查詢到 0 個 profiles  // ❌ 使用 anon key 時
   User IDs: ['c03f0310-d3c8-44ab-8aec-1a4a858c52cb', ...]
   Profiles: []  // 空數組！

📋 查詢到 3 個 profiles  // ✅ 使用 service_role key 後
   User IDs: ['c03f0310-d3c8-44ab-8aec-1a4a858c52cb', ...]
   Profiles: [{first_name: '測試', last_name: '客戶', ...}, ...]
```

---

## 🔧 修復方案

### 修復 1: 使用 Service Role Key

**文件**: `web-admin/src/app/api/admin/customers/route.ts`

```typescript
// 修改前
const db = new DatabaseService(); // 使用 anon key

// 修改後
const db = new DatabaseService(true); // 使用 service_role key
```

**文件**: `web-admin/src/app/api/admin/drivers/route.ts`

```typescript
// 修改前
const db = new DatabaseService(); // 使用 anon key

// 修改後
const db = new DatabaseService(true); // 使用 service_role key
```

### 修復 2: 改進查詢邏輯

將原本的 JOIN 查詢改為分離查詢，以更好地處理關聯資料：

```typescript
// 1. 查詢用戶
const { data: users } = await db.supabase
  .from('users')
  .select('*')
  .eq('role', 'customer');

// 2. 查詢 profiles
const userIds = users?.map(u => u.id) || [];
const { data: profiles } = await db.supabase
  .from('user_profiles')
  .select('*')
  .in('user_id', userIds);

// 3. 合併數據
const profileMap = new Map();
profiles?.forEach(p => {
  profileMap.set(p.user_id, p);
});

const customers = users?.map(user => ({
  ...user,
  user_profiles: profileMap.get(user.id) || null,
}));
```

## 💡 開發心得

### 1. Service Role Key vs Anon Key

**教訓**: 管理端 API 必須使用 Service Role Key

**原因**:
- Anon Key 受 RLS 政策限制
- Service Role Key 可以繞過 RLS
- 管理端需要完整的資料訪問權限

**最佳實踐**:
```typescript
// 前端（客戶端）
const db = new DatabaseService(); // 使用 anon key

// 後端 API（管理端）
const db = new DatabaseService(true); // 使用 service_role key
```

### 2. RLS 政策的影響

**教訓**: RLS 政策會影響所有查詢，包括 JOIN 和 `.in()` 查詢

**診斷方法**:
1. 檢查查詢返回的資料
2. 添加調試日誌
3. 確認使用正確的 Key

### 3. 分離查詢 vs JOIN 查詢

**教訓**: 分離查詢更容易調試和控制

**優點**:
- 更清晰的錯誤處理
- 更容易添加調試日誌
- 更靈活的資料處理

**缺點**:
- 需要多次查詢
- 需要手動合併資料
