## 📋 問題演進和修復歷程

### 問題 1：401 認證錯誤 ✅ 已修復

**症狀**：
- Edge Function 返回：`success: 0, failure: 7`
- 錯誤：`ACCESS_TOKEN_TYPE_UNSUPPORTED`

**根本原因**：
- 使用 Firebase Web API Key
- Firestore REST API 需要 OAuth 2.0 Access Token

**修復方案**：
- 使用 Service Account JSON
- 實現 JWT 簽名和 OAuth 2.0 Token 交換
- 添加 Token 緩存機制

**文檔**：`修復完成-下一步操作.md`

---

### 問題 2：訂單不存在（欄位映射錯誤）✅ 已修復

**症狀**：
- Edge Function 返回：`success: 7, failure: 0`
- Firestore 有資料
- 客戶端顯示：「訂單不存在」

**根本原因**：
- 欄位名稱不匹配：`destination` vs `dropoffAddress`
- 時間格式不匹配：分開的 `startDate` + `startTime` vs 組合的 `bookingTime`
- 缺少必需欄位：`passengerCount`、`estimatedFare`、`depositPaid`

**修復方案**：
- 更新欄位映射：`destination` → `dropoffAddress`
- 組合時間：`startDate` + `startTime` → `bookingTime`
- 添加缺少的欄位

**文檔**：`docs/20251006_0023_06_Firestore欄位映射修復.md`

---

### 問題 3：載入訂單失敗（GeoPoint 格式錯誤）✅ 已修復

**症狀**：
- Firestore 有資料
- 客戶端從「訂單不存在」變成「載入訂單失敗」
- 錯誤：`type '_Map<String, dynamic>' is not a subtype of type 'GeoPoint'`

**根本原因**：
- Edge Function 寫入：`{ latitude: 0, longitude: 0 }` → Firestore 存儲為 **map** 類型
- 客戶端期望：**geopoint** 類型
- Dart 無法將 Map 轉換為 GeoPoint

**修復方案**：
- 使用 `{ _latitude: ..., _longitude: ... }` 格式
- 在 `convertToFirestoreFields` 中檢測並轉換為 `geoPointValue`
- Firestore 存儲為 **geopoint** 類型

**文檔**：`docs/20251006_0840_07_GeoPoint格式修復.md`

---

### 問題 4：載入訂單失敗（Timestamp 格式錯誤）✅ 已修復

**症狀**：
- GeoPoint 已修復
- 客戶端仍然顯示「載入訂單失敗」

**根本原因**：
- Edge Function 寫入：`"2025-10-06T05:12:00"` → Firestore 存儲為 **string** 類型
- 客戶端期望：**timestamp** 類型
- Dart 無法將 string 轉換為 Timestamp

**修復方案**：
- 使用 `{ _timestamp: "..." }` 格式
- 在 `convertToFirestoreFields` 中檢測並轉換為 `timestampValue`
- Firestore 存儲為 **timestamp** 類型

**文檔**：`docs/20251006_2047_08_Timestamp格式修復.md`

## 📊 資料流程（最終版本）

### 完整的資料轉換流程

```
1. Supabase Trigger 創建 Outbox 事件
   ↓
   payload: {
     destination: "台北101",
     startDate: "2025-10-06",
     startTime: "05:12:00",
     pickupLocation: { latitude: 25.033, longitude: 121.5654 },
     ...
   }

2. Edge Function 讀取並轉換
   ↓
   firestoreData: {
     dropoffAddress: "台北101",                    // ✅ 欄位映射
     bookingTime: { _timestamp: "2025-10-06T05:12:00" },  // ✅ 時間組合 + Timestamp 標記
     pickupLocation: { _latitude: 25.033, _longitude: 121.5654 },  // ✅ GeoPoint 標記
     ...
   }

3. convertToFirestoreFields 轉換為 Firestore REST API 格式
   ↓
   {
     dropoffAddress: { stringValue: "台北101" },
     bookingTime: { timestampValue: "2025-10-06T05:12:00.000Z" },
     pickupLocation: { geoPointValue: { latitude: 25.033, longitude: 121.5654 } },
     ...
   }

4. Firestore 存儲
   ↓
   dropoffAddress: "台北101" (string)
   bookingTime: October 6, 2025 at 5:12:00 AM UTC+8 (timestamp)
   pickupLocation: [25.033° N, 121.5654° E] (geopoint)

5. 客戶端讀取並解析
   ↓
   BookingOrder(
     dropoffAddress: "台北101",
     bookingTime: DateTime(2025, 10, 6, 5, 12),
     pickupLocation: LocationPoint(25.033, 121.5654),
     ...
   )
   ✅ 成功！
```

## 💡 經驗總結

### 1. Firestore 資料類型的重要性

**關鍵教訓**：
- Firestore 有特定的資料類型，不能隨意使用 JavaScript 值
- **GeoPoint** 必須是 geopoint 類型（不是 map）
- **Timestamp** 必須是 timestamp 類型（不是 string）
- 客戶端才能正確解析

---

### 2. 逐步診斷的價值

**診斷過程**：
1. 401 錯誤 → 認證問題
2. 訂單不存在 → 欄位映射問題
3. 載入訂單失敗（GeoPoint 錯誤）→ GeoPoint 格式問題
4. 載入訂單失敗（Timestamp 錯誤）→ Timestamp 格式問題

**經驗**：
- 每次修復後，錯誤訊息都會變化
- 錯誤訊息的變化表示修復成功，但還有其他問題
- 逐步排除問題，更容易診斷

---

### 3. 客戶端錯誤訊息的價值

**有效的診斷**：
- 「訂單不存在」→ 找不到訂單或 GeoPoint 解析失敗
- 「載入訂單失敗」→ 找到訂單但其他欄位解析失敗
- 錯誤訊息的變化幫助定位問題
