## 🏗️ 系統架構

### 雙保險機制

```
┌─────────────────────────────────────────────────────────┐
│              即時 + 補償雙保險通知架構                    │
└─────────────────────────────────────────────────────────┘

1️⃣ 即時通知（1-3 秒延遲）
   bookings 表變更
        ↓
   Trigger 觸發
        ↓
   notify_edge_function_realtime()
        ↓
   pg_net HTTP POST
        ↓
   Edge Function: sync-to-firestore
        ↓
   Firestore 同步 (orders_rt + bookings)

2️⃣ Cron 補償（5-30 秒延遲）
   Cron Job 每 30 秒執行
        ↓
   查詢 outbox 未處理事件
        ↓
   調用 Edge Function
        ↓
   Firestore 同步
```

## 🎯 關鍵特性

### 1. 智能狀態檢測 ✅

**問題**: 不同環境的數據庫可能有不同的 status CHECK 約束

**解決**: 從現有訂單中動態獲取有效的 status 值

**優點**:
- ✅ 自動適配任何數據庫約束
- ✅ 不需要硬編碼 status 值
- ✅ 100% 保證使用有效的 status

### 2. 雙保險機制 ✅

**即時通知**:
- 延遲: 1-3 秒
- 觸發: bookings 表 INSERT/UPDATE
- 方式: PostgreSQL Trigger + pg_net

**Cron 補償**:
- 延遲: 5-30 秒
- 觸發: 每 30 秒執行
- 方式: pg_cron + outbox 模式

### 3. 完整的監控 ✅

**HTTP 請求記錄**:
- 表: `net._http_response`
- 記錄: 所有 HTTP 請求和響應

**Outbox 事件**:
- 表: `outbox`
- 記錄: 所有訂單變更事件

**性能統計**:
- 表: `realtime_sync_stats`
- 記錄: 延遲、成功率等指標

### 3. 監控性能

**查看 HTTP 請求記錄**:
```sql
SELECT 
  id,
  created,
  status_code,
  CASE
    WHEN status_code = 200 THEN '✅ 成功'
    WHEN status_code IS NULL THEN '⏳ 處理中'
    ELSE '❌ 失敗'
  END AS "狀態"
FROM net._http_response
ORDER BY created DESC
LIMIT 10;
```

**查看性能統計**:
```sql
WITH recent_events AS (
  SELECT 
    created_at,
    processed_at,
    EXTRACT(EPOCH FROM (processed_at - created_at)) AS delay_seconds
  FROM outbox
  WHERE created_at >= NOW() - INTERVAL '1 hour'
    AND processed_at IS NOT NULL
)
SELECT 
  COUNT(*) AS "處理事件數",
  ROUND(AVG(delay_seconds)::NUMERIC, 2) AS "平均延遲（秒）",
  ROUND(MIN(delay_seconds)::NUMERIC, 2) AS "最小延遲（秒）",
  ROUND(MAX(delay_seconds)::NUMERIC, 2) AS "最大延遲（秒）"
FROM recent_events;
```
