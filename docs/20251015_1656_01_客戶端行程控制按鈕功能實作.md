## 📋 需求概述

在客戶端訂單詳情頁面添加「開始行程」和「結束行程」按鈕，讓客戶能夠控制行程的開始和結束，並自動更新所有端（公司端、客戶端、司機端）的訂單狀態。

### 核心功能

1. **開始行程按鈕**
   - 顯示條件：訂單狀態為 `driver_arrived`（司機已到達）或 `matched`（司機已確認接單）
   - 點擊後：更新訂單狀態為 `trip_started`，發送系統訊息
   - 備註提示：「⚠️ 請務必在與司機見面時按下，避免金額計算錯誤」

2. **結束行程按鈕**
   - 顯示條件：訂單狀態為 `inProgress`（進行中）
   - 點擊後：更新訂單狀態為 `trip_ended`，發送系統訊息
   - 備註提示：「⚠️ 請務必在確定結束行程時按下，避免金額計算錯誤」

## 🔍 問題診斷過程

### 1. 現有架構分析

**發現的架構特點**:
- Backend 使用 Node.js/Express + Supabase (PostgreSQL)
- Flutter 使用 Riverpod 狀態管理
- Supabase 作為主數據庫，Firestore 用於實時同步
- 已有司機端狀態按鈕功能（可參考實作模式）

**狀態流程**:
```
Supabase: driver_confirmed → driver_departed → driver_arrived → trip_started → trip_ended → pending_balance → completed
Firestore: matched → inProgress → inProgress → inProgress → completed → completed → completed
```

### 2. 技術挑戰識別

**挑戰 1: 狀態映射複雜性**
- Supabase 有詳細的狀態（trip_started, trip_ended, pending_balance）
- Firestore 狀態較簡化（pending, matched, inProgress, completed）
- Flutter BookingStatus 枚舉更簡化，無法區分細微狀態

**挑戰 2: 按鈕顯示邏輯**
- 需要根據訂單狀態動態顯示不同按鈕
- 同一個 Firestore 狀態（inProgress）可能對應多個 Supabase 狀態
- 需要避免按鈕重複顯示或錯誤顯示

**挑戰 3: 權限驗證**
- 只有訂單的客戶才能操作這些按鈕
- 需要驗證 Firebase UID 與訂單客戶的對應關係

## 💡 解決方案設計

### 1. Backend API 設計

**API 1: 開始行程**
```typescript
POST /api/booking-flow/bookings/:bookingId/start-trip
Body: { customerUid: string }

流程:
1. 查詢訂單資料
2. 驗證客戶權限（customer_id 匹配）
3. 檢查訂單狀態（必須為 driver_confirmed 或 driver_arrived）
4. 更新狀態為 trip_started
5. 記錄 actual_start_time
6. 發送系統訊息：「客戶已開始行程 🚀」
7. 返回成功響應
```

**API 2: 結束行程**
```typescript
POST /api/booking-flow/bookings/:bookingId/end-trip
Body: { customerUid: string }

流程:
1. 查詢訂單資料
2. 驗證客戶權限
3. 檢查訂單狀態（必須為 trip_started）
4. 更新狀態為 trip_ended
5. 記錄 actual_end_time
6. 發送系統訊息：「行程已結束，請支付尾款 💰」
7. 返回成功響應
```

### 2. Flutter Service 設計

**新增方法**:
```dart
// BookingService
Future<Map<String, dynamic>> startTrip(String bookingId)
Future<Map<String, dynamic>> endTrip(String bookingId)
```

**實作要點**:
- 使用 Firebase Auth 獲取當前用戶 UID
- 調用 Backend API
- 完整的錯誤處理
- Debug 日誌輸出

### 3. Flutter UI 設計

**按鈕樣式**:
- 開始行程：綠色 (#4CAF50) + 🚀 圖標
- 結束行程：橙色 (#FF9800) + 🏁 圖標
- 備註文字：灰色小字，顯示在按鈕下方

**交互流程**:
1. 點擊按鈕 → 顯示確認對話框
2. 確認後 → 顯示載入對話框
3. API 成功 → 關閉載入對話框 → 顯示成功提示 → 刷新訂單
4. API 失敗 → 關閉載入對話框 → 顯示錯誤提示

### 4. Firestore 同步更新

**狀態映射更新**:
```typescript
'trip_started': 'inProgress',    // 行程開始 → 進行中
'trip_ended': 'completed',       // 行程結束 → 已完成
'pending_balance': 'completed',  // 待付尾款 → 已完成
```

```dart
Future<Map<String, dynamic>> startTrip(String bookingId) async {
  final user = _auth.currentUser;
  if (user == null) throw Exception('用戶未登入');
  
  final url = '$_baseUrl/booking-flow/bookings/$bookingId/start-trip';
  final response = await http.post(
    Uri.parse(url),
    headers: {'Content-Type': 'application/json'},
    body: json.encode({'customerUid': user.uid}),
  );
  
  if (response.statusCode == 200) {
    return json.decode(response.body);
  } else {
    throw Exception(json.decode(response.body)['error']);
  }
}
```

## 💭 開發心得與經驗總結

### 成功經驗

1. **參考現有實作**

   - 複用了相同的 API 結構、錯誤處理、UI 交互流程
   - 大幅減少了開發時間和錯誤

2. **完整的錯誤處理**
   - 權限驗證：確保只有訂單客戶才能操作
   - 狀態檢查：嚴格檢查訂單狀態，避免重複操作
   - 用戶提示：清晰的確認對話框和錯誤訊息

3. **良好的用戶體驗**
   - 確認對話框：避免誤觸
   - 載入指示器：提供視覺反饋
   - 成功/錯誤提示：明確的操作結果
   - 備註文字：提醒用戶注意事項

### 遇到的困難

1. **狀態映射複雜性**
   - **問題**: Supabase、Firestore、Flutter 三層狀態不一致
   - **解決**: 在 Edge Function 中統一映射，暫時將 trip_ended 映射為 completed
   - **改進**: 長期應擴展 Flutter BookingStatus 枚舉以支援所有狀態

2. **按鈕顯示邏輯**
   - **問題**: inProgress 狀態包含多個 Supabase 狀態，難以精確判斷
   - **解決**: 暫時在 inProgress 狀態下同時顯示「開始行程」和「結束行程」按鈕
   - **改進**: 在 BookingOrder 模型中添加原始 Supabase 狀態欄位

3. **時間戳記欄位**
   - **問題**: 需要記錄 actual_start_time 和 actual_end_time
   - **解決**: 在 API 中直接更新這些欄位
   - **注意**: 需要確保數據庫表結構支援這些欄位

### 犯過的錯誤與教訓

1. **忘記導入 BookingService**
   - **錯誤**: 在 order_detail_page.dart 中使用 BookingService 但忘記導入
   - **教訓**: 修改文件時要檢查所有依賴是否已導入
   - **修復**: 添加 `import '../../../../core/services/booking_service.dart';`

2. **狀態映射不完整**
   - **錯誤**: 最初忘記在 sync-to-firestore 中添加新狀態映射
   - **教訓**: 修改狀態流程時要同步更新所有相關配置
   - **修復**: 添加 trip_started, trip_ended, pending_balance 映射

3. **按鈕顯示邏輯過於簡化**
   - **錯誤**: 最初計劃根據 Firestore 狀態顯示按鈕，但無法區分細微狀態
   - **教訓**: 需要考慮狀態映射的限制，設計更靈活的顯示邏輯
   - **修復**: 添加 TODO 註釋，提醒未來改進
