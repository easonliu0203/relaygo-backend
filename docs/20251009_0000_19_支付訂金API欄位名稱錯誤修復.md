## 🔍 問題診斷過程

### 步驟 1: 檢查 Supabase Schema

**文件**: `supabase/fix-schema-complete.sql` (第 54-76 行)

```sql
CREATE TABLE IF NOT EXISTS payments (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  transaction_id VARCHAR(100) UNIQUE NOT NULL,
  booking_id UUID REFERENCES bookings(id) NOT NULL,
  customer_id UUID REFERENCES users(id) NOT NULL,
  type VARCHAR(20) NOT NULL CHECK (type IN ('deposit', 'balance', 'refund')),  -- ✅ 欄位名稱是 'type'
  amount DECIMAL(10, 2) NOT NULL,
  currency VARCHAR(3) DEFAULT 'TWD',
  status VARCHAR(20) NOT NULL DEFAULT 'pending' CHECK (status IN (
    'pending', 'processing', 'completed', 'failed', 'cancelled', 'expired', 'refunded'
  )),
  payment_provider VARCHAR(50) NOT NULL,
  payment_method VARCHAR(50),
  is_test_mode BOOLEAN DEFAULT false,
  external_transaction_id VARCHAR(255),
  payment_url TEXT,
  instructions TEXT,
  confirmed_by UUID REFERENCES users(id),
  admin_notes TEXT,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  expires_at TIMESTAMP WITH TIME ZONE
);
```

**發現**: Supabase 實際使用的 schema 中，欄位名稱是 **`type`**，不是 `payment_type`。

## 💡 開發心得與經驗

### 1. Schema 一致性的重要性

**教訓**: 
- 資料庫 schema 和後端代碼必須保持一致
- 不同文件中的 schema 定義可能不同，需要確認實際使用的是哪一個

**最佳實踐**:
- 使用單一的 schema 定義文件作為真實來源（Single Source of Truth）
- 定期檢查代碼和 schema 的一致性
- 使用 TypeScript 類型定義來強制類型安全

---

### 2. 欄位命名規範

**發現的問題**:
- 不同文件使用不同的欄位名稱（`payment_type` vs `type`）
- 導致混淆和錯誤

**建議**:
- 統一欄位命名規範
- 避免使用過於通用的名稱（如 `type`），考慮使用更具描述性的名稱（如 `payment_type`）
- 但如果已經確定使用某個名稱，就要在所有地方保持一致

---

### 3. 錯誤訊息的價值

**這次的錯誤訊息非常清晰**:
```
Could not find the 'payment_type' column of 'payments' in the schema cache
```

**從錯誤訊息中可以直接得知**:
- 問題出在 `payments` 表
- 缺少 `payment_type` 欄位
- 是 schema cache 的問題（即資料庫 schema）

**教訓**: 仔細閱讀錯誤訊息，它們通常會直接指出問題所在。

---

### 4. 診斷流程

**有效的診斷流程**:
1. ✅ 閱讀錯誤訊息，確定問題範圍
2. ✅ 檢查資料庫 schema 定義
3. ✅ 檢查後端代碼使用的欄位名稱
4. ✅ 對比兩者，找出不匹配的地方
5. ✅ 修復代碼或 schema
6. ✅ 測試驗證

## 🚨 遇到的困難和解決方法

### 困難 1: 多個 Schema 定義文件

**問題**: 
- `database/schema.sql` 使用 `payment_type`
- `supabase/fix-schema-complete.sql` 使用 `type`
- 不確定實際使用的是哪一個

**解決方法**:
- 檢查 Supabase 的部署歷史
- 確認 `supabase/fix-schema-complete.sql` 是最新的
- 查看文件的修改時間和內容

---

### 困難 2: 缺少必填欄位

**問題**: 
- 修復 `payment_type` → `type` 後，可能還會遇到 `customer_id` 缺失的錯誤

**解決方法**:
- 仔細檢查 schema 定義，找出所有 NOT NULL 欄位
- 確保後端代碼提供所有必填欄位
- 添加 `customer_id` 和 `currency` 欄位

---
