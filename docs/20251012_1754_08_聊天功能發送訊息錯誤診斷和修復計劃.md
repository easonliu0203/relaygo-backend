## 🔍 根本原因分析

### 問題 1 根本原因：架構不匹配

**Flutter APP 的實現** (`mobile/lib/core/services/chat_service.dart`):
```dart
// 調用 HTTP API 發送訊息
final response = await http.post(
  Uri.parse('$_baseUrl/api/chat/send-message'),  // ❌ 這個端點不存在
  headers: {'Content-Type': 'application/json'},
  body: jsonEncode({
    'bookingId': bookingId,
    'senderId': currentUser.uid,
    'receiverId': receiverId,
    'messageText': messageText,
  }),
);
```

**Backend 的實現** (`backend/src/services/chat/ChatService.ts`):
- Backend ChatService 是基於 Socket.IO 的內存服務
- **沒有 HTTP API 端點** `/api/chat/send-message`
- 只有 Socket.IO 事件處理

**正確的架構** (根據 Supabase migration):
```
Flutter APP 
  → Supabase chat_messages 表 (INSERT)
  → Supabase trigger (chat_messages_to_outbox)
  → outbox 表
  → Edge Function (sync-to-firestore)
  → Firestore chat_rooms/{bookingId}/messages
  → Flutter APP (實時監聽 Firestore)
```

**結論**: Flutter APP 應該直接寫入 Supabase `chat_messages` 表，而不是調用不存在的 HTTP API。

## 🛠️ 修復方案

### 方案 1: 修改 Flutter ChatService 使用 Supabase（推薦）

**優點**:
- 符合現有架構設計
- 利用 Supabase RLS 策略保護數據
- 自動觸發 trigger 同步到 Firestore
- 不需要修改 Backend

**缺點**:
- 需要修改 Flutter 代碼
- 需要處理 Firebase UID 到 Supabase UUID 的轉換

**實現步驟**:

1. **修改 `mobile/lib/core/services/chat_service.dart`**:
   - 移除 `http` 依賴
   - 添加 `supabase_flutter` 依賴
   - 修改 `sendMessage()` 方法直接寫入 Supabase

2. **發送訊息流程**:
   ```dart
   // 1. 獲取發送者的 Supabase UUID（通過 Firebase UID 查詢）
   final senderResponse = await _supabase
       .from('users')
       .select('id')
       .eq('firebase_uid', currentUser.uid)
       .single();
   
   // 2. 獲取接收者的 Supabase UUID
   final receiverResponse = await _supabase
       .from('users')
       .select('id')
       .eq('firebase_uid', receiverId)
       .single();
   
   // 3. 插入訊息到 Supabase
   await _supabase
       .from('chat_messages')
       .insert({
         'booking_id': bookingId,
         'sender_id': senderId,
         'receiver_id': receiverSupabaseId,
         'message_text': messageText,
       });
   ```

3. **Supabase 自動處理**:
   - RLS 策略驗證發送者權限
   - Trigger 寫入 outbox 表
   - Edge Function 同步到 Firestore

4. **Flutter APP 讀取**:
   - 從 Firestore 實時監聽訊息（已實現）

### 方案 2: 創建 Backend HTTP API（不推薦）

**優點**:
- Flutter 代碼改動較小

**缺點**:
- 需要創建新的 Backend API 路由和控制器
- 需要處理認證和授權
- 增加系統複雜度
- 與現有架構不一致

  b.id as booking_id,

  COALESCE(CONCAT(cp.first_name, ' ', cp.last_name), c.email) as customer_name,

LEFT JOIN users d ON b.driver_id = d.id

ORDER BY b.created_at DESC

## 💡 技術要點

### Supabase RLS 策略

**查看訊息** (`chat_messages` 表):
```sql
CREATE POLICY "Users can view their own messages"
  ON chat_messages
  FOR SELECT
  USING (
    sender_id IN (
      SELECT id FROM users WHERE firebase_uid = auth.jwt() ->> 'sub'
    )
    OR
    receiver_id IN (
      SELECT id FROM users WHERE firebase_uid = auth.jwt() ->> 'sub'
    )
  );
```

**發送訊息** (`chat_messages` 表):
```sql
CREATE POLICY "Users can send messages"
  ON chat_messages
  FOR INSERT
  WITH CHECK (
    sender_id IN (
      SELECT id FROM users WHERE firebase_uid = auth.jwt() ->> 'sub'
    )
  );
```

### Firebase UID 到 Supabase UUID 轉換

**問題**: Flutter APP 使用 Firebase UID，Supabase 使用 UUID

**解決方案**: 通過 `users` 表的 `firebase_uid` 欄位查詢對應的 `id` (UUID)

```dart
// 查詢 Supabase UUID
final response = await _supabase
    .from('users')
    .select('id')
    .eq('firebase_uid', firebaseUid)
    .single();

final supabaseId = response['id'] as String;
```
