# Cron Job 執行頻率調整指南

**日期**: 2025-10-14  
**時間**: 16:30  
**主旨**: 將 Cron Job 執行頻率從 30 秒改為 5 秒（測試環境）  
**狀態**: ✅ 已完成

---

## 📋 需求背景

### 當前問題

測試流程需要經過多個步驟：
1. 創建訂單 → `pending_payment`
2. 支付訂金 → `paid_deposit`
3. 手動派單 → `matched`
4. 司機確認 → `driver_confirmed`

**問題**: Cron Job 每 30 秒執行一次，導致每個步驟都需要等待 30 秒才能看到 Firestore 同步結果，測試效率低。

### 解決方案

將 Cron Job 執行頻率從「每 30 秒」改為「每 5 秒」：
- ✅ 加快測試速度（等待時間從 30 秒降至 5 秒）
- ✅ 快速驗證每個步驟的同步結果
- ⚠️ 測試完成後需改回 30 秒（避免資源浪費）

---

## 🔧 修改步驟

### 步驟 1: 更新配置文件（已完成）

**文件**: `supabase/functions/sync-to-firestore/cron.yaml`

**修改前**:
```yaml
# 每 30 秒執行一次（高頻率，確保即時性）
- name: sync-orders-to-firestore
  schedule: "*/30 * * * * *"  # 每 30 秒
  function: sync-to-firestore
  description: "同步訂單變更到 Firestore（Outbox Pattern）"
```

**修改後**:
```yaml
# ⚠️ 測試環境配置：每 5 秒執行一次（快速驗證同步結果）
# 🚨 正式環境請改回：*/30 * * * * *（每 30 秒）
- name: sync-orders-to-firestore
  schedule: "*/5 * * * * *"  # 每 5 秒（測試用）
  function: sync-to-firestore
  description: "同步訂單變更到 Firestore（Outbox Pattern）- 測試環境高頻率"
```

### 步驟 2: 更新 Supabase 中的 Cron Job

**執行 SQL**:
```sql
-- 前往 Supabase Dashboard > SQL Editor
-- 執行以下命令

-- 更新執行頻率（從 30 秒改為 5 秒）
UPDATE cron.job
SET schedule = '*/5 * * * * *'
WHERE jobname = 'sync-orders-to-firestore';

-- 驗證更新結果
SELECT 
  jobname as 任務名稱,
  schedule as 執行頻率,
  CASE 
    WHEN schedule = '*/5 * * * * *' THEN '✅ 已更新為每 5 秒'
    ELSE '❌ 更新失敗'
  END as 更新狀態
FROM cron.job
WHERE jobname = 'sync-orders-to-firestore';
```

**預期結果**:
```
任務名稱: sync-orders-to-firestore
執行頻率: */5 * * * * *
更新狀態: ✅ 已更新為每 5 秒
```

### 步驟 3: 驗證修改是否生效

#### 3.1 等待 10 秒後檢查執行記錄

```sql
-- 等待 10 秒後執行此查詢
SELECT 
  runid as 執行ID,
  status as 狀態,
  start_time as 開始時間,
  CASE 
    WHEN status = 'succeeded' THEN '✅ 成功'
    WHEN status = 'failed' THEN '❌ 失敗'
    ELSE '⏳ 執行中'
  END as 執行結果
FROM cron.job_run_details
WHERE jobid = (
  SELECT jobid FROM cron.job WHERE jobname = 'sync-orders-to-firestore'
)
  AND start_time >= NOW() - INTERVAL '1 minute'
ORDER BY start_time DESC
LIMIT 10;
```

**預期結果**:
- 1 分鐘內應該有約 12 次執行記錄（60 秒 / 5 秒 = 12 次）

#### 3.2 測試同步速度

```bash
# 創建測試訂單
curl -X POST "http://localhost:3000/api/bookings" \
  -H "Content-Type: application/json" \
  -d '{
    "customerUid": "hUu4fH5dTlW9VUYm6GojXvRLdni2",
    "pickupAddress": "測試 5 秒同步",
    "destination": "測試目的地",
    "pickupLatitude": 25.033,
    "pickupLongitude": 121.5654,
    "bookingTime": "2025-10-15T15:00:00.000",
    "passengerCount": 1
  }'
```

```sql
-- 創建訂單後，立即執行此查詢檢查同步時間
SELECT 
  id as 事件ID,
  payload->>'bookingNumber' as 訂單編號,
  created_at as 創建時間,
  processed_at as 處理時間,
  EXTRACT(EPOCH FROM (processed_at - created_at)) as 同步耗時_秒,
  CASE 
    WHEN processed_at IS NOT NULL THEN '✅ 已同步'
    ELSE '⏳ 待同步'
  END as 狀態
FROM outbox
WHERE created_at >= NOW() - INTERVAL '5 minutes'
ORDER BY created_at DESC
LIMIT 5;
```

**預期結果**:
- 同步耗時應該在 **5 秒以內**（最多等待一個 Cron Job 週期）

| 環境 | Cron 表達式 | 執行頻率 | 每小時執行次數 | 資源消耗 |

| **正式環境（30 秒）** | 30 秒 | 15 秒 | ⏳ 較慢 |

## 🚨 重要提醒

### ⚠️ 測試完成後必須改回 30 秒

**原因**:
1. **資源消耗**: 5 秒頻率是 30 秒的 6 倍，長期運行會浪費資源
2. **成本考量**: Supabase Edge Function 按執行次數計費
3. **實際需求**: 正式環境 30 秒延遲已足夠（用戶不會頻繁刷新）

### 改回正式環境配置

**執行 SQL**:
```sql
-- 改回正式環境配置（每 30 秒）
UPDATE cron.job
SET schedule = '*/30 * * * * *'
WHERE jobname = 'sync-orders-to-firestore';

-- 驗證更新
SELECT 
  jobname,
  schedule,
  CASE 
    WHEN schedule = '*/30 * * * * *' THEN '✅ 已改回正式環境配置'
    ELSE '❌ 仍是測試環境配置'
  END as 配置狀態
FROM cron.job
WHERE jobname = 'sync-orders-to-firestore';
```

**同時更新配置文件**:
```yaml
# 正式環境配置：每 30 秒執行一次
- name: sync-orders-to-firestore
  schedule: "*/30 * * * * *"  # 每 30 秒
  function: sync-to-firestore
  description: "同步訂單變更到 Firestore（Outbox Pattern）"
```

    "passengerCount": 1
  }'

**預期**: 5 秒內同步到 Firestore（狀態：`pending`）

#### 2. 支付訂金

```bash
curl -X POST "http://localhost:3000/api/bookings/<BOOKING_ID>/pay-deposit" \
  -H "Content-Type: application/json" \
  -d '{
    "paymentMethod": "credit_card",
    "customerUid": "hUu4fH5dTlW9VUYm6GojXvRLdni2"
  }'
```

**預期**: 5 秒內同步到 Firestore（狀態：`pending`）

#### 3. 手動派單

```sql
-- 在 Supabase Table Editor 中執行
UPDATE bookings
SET 
  driver_id = '416556f9-adbf-4c2e-920f-164d80f5307a',
  status = 'matched'
WHERE id = '<BOOKING_ID>';
```

**預期**: 5 秒內同步到 Firestore（狀態：`pending`）

#### 4. 司機確認接單

```bash
curl -X POST "http://localhost:3000/api/booking-flow/bookings/<BOOKING_ID>/accept" \
  -H "Content-Type: application/json" \
  -d '{
    "driverUid": "CMfTxhJFlUVDkosJPyUoJvKjCQk1"
  }'
```

**預期**: 5 秒內同步到 Firestore（狀態：`matched`）

### 驗證同步結果

```sql
-- 檢查所有步驟的同步時間
SELECT 
  payload->>'bookingNumber' as 訂單編號,
  event_type as 事件類型,
  payload->>'status' as 訂單狀態,
  created_at as 創建時間,
  processed_at as 處理時間,
  EXTRACT(EPOCH FROM (processed_at - created_at)) as 同步耗時_秒
FROM outbox
WHERE aggregate_id = '<BOOKING_ID>'
ORDER BY created_at;
```

**預期結果**:
- 所有步驟的同步耗時都在 **5 秒以內**
