# Firestore 權限錯誤修復

**日期**：2025-10-08 07:38  
**問題**：訂單創建後跳轉到「預約成功」頁面時出現 `permission-denied` 錯誤  
**狀態**：✅ 已修復，待部署

---

## 📋 問題描述

### 錯誤訊息
```
[cloud_firestore/permission-denied] 
The caller does not have permission to execute the specified operation.
```

### 發生時機
1. 用戶完成支付流程
2. 調用 `payDepositWithSupabase()` 成功
3. 跳轉到「預約成功」頁面（`BookingSuccessPage`）
4. 頁面嘗試從 Firestore 讀取訂單資料
5. ❌ 拋出 `permission-denied` 錯誤

### 背景資訊
- 剛完成 CQRS 架構修復
- 修改了 Firestore 規則禁止客戶端寫入 `bookings` 和 `orders_rt` 集合
- 目的是強制所有寫入操作通過 Supabase API

---

````dart
Stream<BookingOrder?> watchBooking(String orderId) {
  // 從 orders_rt 鏡像集合讀取（Read-Only）
  return _firestore
      .collection('orders_rt')
      .doc(orderId)
      .snapshots()
      .map((doc) {
        if (!doc.exists) return null;
        return BookingOrder.fromFirestore(doc);
      });
}
````
</augment_code_snippet>

**發現**：
- 從 Firestore `orders_rt` 集合讀取訂單

### 步驟 4：檢查 Firestore 規則

<augment_code_snippet path="firebase/firestore.rules" mode="EXCERPT">
````javascript
match /orders_rt/{orderId} {
  // 允許用戶讀取自己的訂單
  allow read: if request.auth != null
              && resource.data.customerId == request.auth.uid;

  // 禁止客戶端寫入（由 Supabase 寫入）
  allow write: if false;
}
````
</augment_code_snippet>

**發現問題**：
- 規則使用 `resource.data.customerId` 檢查權限
- `resource.data` 只在文檔**已經存在**時才有值
- 如果文檔不存在，`resource.data` 為 null
- 導致權限檢查失敗 → `permission-denied`

---

### 步驟 5：分析時序問題

**正常流程**：
```
1. 用戶支付訂金
   ↓
2. payDepositWithSupabase() → Supabase API
   ↓
3. Supabase 更新 bookings 表
   ↓
4. Trigger 寫入 outbox 表
   ↓
5. 返回成功 → 跳轉到「預約成功」頁面 ⭐
   ↓
6. 頁面嘗試讀取 Firestore orders_rt/{orderId}
   ↓
7. ❌ 文檔還不存在（Edge Function 還沒執行）
   ↓
8. ❌ resource.data 為 null
   ↓
9. ❌ 權限檢查失敗 → permission-denied
   ↓
10. (等待 Cron Job 執行，最多 30 秒)
   ↓
11. Edge Function 同步到 Firestore
   ↓
12. 文檔現在存在了（但用戶已經看到錯誤）
```

**根本原因**：
- CQRS 架構的同步延遲
- Supabase → Outbox → Cron Job → Edge Function → Firestore
- 整個流程可能需要 1-30 秒
- 客戶端在同步完成前嘗試讀取文檔
- Firestore 規則不允許讀取不存在的文檔

---

## 🔧 修復方案

### 解決思路

**問題**：
- `resource.data` 在文檔不存在時為 null
- 無法檢查 `resource.data.customerId`

**解決方案**：
- 使用 `exists()` 函數檢查文檔是否存在
- 文檔不存在時允許讀取（返回 null，不報錯）
- 文檔存在時再檢查 `customerId`

---

### 修改 Firestore 規則

#### 修改 1：orders_rt 集合

**修改前**（錯誤）：
```javascript
match /orders_rt/{orderId} {
  allow read: if request.auth != null
              && resource.data.customerId == request.auth.uid;
  allow write: if false;
}
```

**修改後**（正確）：
```javascript
match /orders_rt/{orderId} {
  // 允許用戶讀取自己的訂單
  // 注意：使用 exists() 來允許讀取不存在的文檔（返回 null）
  // 這樣可以避免在 Edge Function 同步延遲時出現權限錯誤
  allow read: if request.auth != null
              && (
                // 文檔不存在時允許讀取（會返回 null）
                !exists(/databases/$(database)/documents/orders_rt/$(orderId))
                ||
                // 文檔存在時檢查是否為用戶自己的訂單
                resource.data.customerId == request.auth.uid
              );

  // 禁止客戶端寫入（由 Supabase 寫入）
  allow write: if false;
}
```

---

#### 修改 2：bookings 集合

**修改前**（錯誤）：
```javascript
match /bookings/{bookingId} {
  allow read: if request.auth != null &&
    (request.auth.uid == resource.data.customerId ||
     request.auth.uid == resource.data.driverId);
  allow write: if false;
}
```

**修改後**（正確）：
```javascript
match /bookings/{bookingId} {
  // 允許用戶讀取自己的訂單
  // 注意：使用 exists() 來允許讀取不存在的文檔（返回 null）
  // 這樣可以避免在 Edge Function 同步延遲時出現權限錯誤
  allow read: if request.auth != null &&
    (
      // 文檔不存在時允許讀取（會返回 null）
      !exists(/databases/$(database)/documents/bookings/$(bookingId))
      ||
      // 文檔存在時檢查是否為用戶自己的訂單
      (request.auth.uid == resource.data.customerId ||
       request.auth.uid == resource.data.driverId)
    );

  // 禁止客戶端寫入（由 Supabase Edge Function 寫入）
  allow write: if false;
  }
}
```

---

### 修復效果

**修復前**：
```
讀取不存在的文檔 → resource.data 為 null → 權限檢查失敗 → permission-denied
```

**修復後**：
```
讀取不存在的文檔 → exists() 返回 false → 允許讀取 → 返回 null（不報錯）
讀取存在的文檔 → exists() 返回 true → 檢查 customerId → 允許讀取
```

## 💡 開發心得

### 1. Firestore 規則的細節很重要

**教訓**：
- `resource.data` 只在文檔存在時有值
- 讀取不存在的文檔時，`resource.data` 為 null
- 必須使用 `exists()` 函數處理這種情況

**改進**：
- 先檢查文檔是否存在
- 文檔不存在時允許讀取（返回 null）
- 文檔存在時再檢查權限

---

### 2. CQRS 架構的同步延遲

**問題**：
- Supabase → Outbox → Cron Job → Edge Function → Firestore
- 整個流程可能需要 1-30 秒
- 客戶端可能在同步完成前嘗試讀取

**解決方案**：
- 允許讀取不存在的文檔
- 使用 Stream 監聽文檔變化
- 顯示「載入中」狀態
- 等待同步完成後自動更新

---

### 3. 安全性不受影響

**重要**：
- 修復後仍然禁止客戶端寫入（`allow write: if false`）
- 只允許讀取不存在的文檔（返回 null）
- 文檔存在時仍然檢查 `customerId`
- 不違反 CQRS 架構原則
- 不降低安全性

---

### 4. exists() 函數的正確使用

**語法**：
```javascript
exists(/databases/$(database)/documents/collection/$(docId))
```

**作用**：
- 檢查文檔是否存在
- 返回 `true` 或 `false`
- 不會拋出權限錯誤

**使用場景**：
- 允許讀取不存在的文檔
- 避免同步延遲導致的權限錯誤
- 處理文檔創建前的讀取請求

**問題**：
- 不清楚 `resource.data` 在文檔不存在時的行為
- 以為只要登入就可以讀取

**解決方法**：

## ❌ 犯過的錯誤和教訓

### 錯誤 1：過度限制讀取權限

**錯誤做法**：
```javascript
allow read: if request.auth != null
            && resource.data.customerId == request.auth.uid;
```

**問題**：
- 文檔不存在時無法讀取
- 導致同步延遲時出現權限錯誤

**教訓**：
- 必須考慮文檔不存在的情況
- 使用 `exists()` 函數處理

---

### 錯誤 2：沒有考慮同步延遲

**問題**：
- 以為 Supabase 寫入後 Firestore 會立即同步
- 沒有考慮 Cron Job 的執行週期（30 秒）

**教訓**：
- CQRS 架構有同步延遲
- 必須在規則和 UI 中處理這種情況
- 使用 Stream 監聽文檔變化

---

### 錯誤 3：沒有充分測試規則修改

**問題**：
- 修改規則後沒有測試讀取不存在的文檔
- 直到用戶測試時才發現問題

**教訓**：
- 修改規則後必須充分測試
- 測試各種邊界情況
- 包括文檔不存在、同步延遲等
