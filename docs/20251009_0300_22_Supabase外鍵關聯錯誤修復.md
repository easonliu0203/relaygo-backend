## 📋 問題描述

### 問題現象

**發生位置**:
- 公司端（web-admin）訂單管理頁面
- 訪問 http://localhost:3001/orders 時

**錯誤訊息**:
```
❌ 獲取訂單失敗: {
  code: 'PGRST200',
  details: "Searched for a foreign key relationship between 'users' and 'user_profiles' in the schema 'public', but no matches were found.",
  hint: null,
  message: "Could not find a relationship between 'users' and 'user_profiles' in the schema cache"
}
```

**錯誤類型**:
- Supabase PostgREST 錯誤（PGRST200）
- 外鍵關聯（Foreign Key Relationship）找不到
- 涉及的表：`users` 和 `user_profiles`

**影響範圍**:
- 訂單管理主頁面無法載入
- 所有訂單相關頁面（待處理、進行中、已完成）都受影響
- 客戶和司機資訊無法顯示

## 🔍 問題診斷過程

### 1. 分析錯誤訊息

**錯誤關鍵信息**:
- `PGRST200`: PostgREST 無法找到表之間的關聯
- `Could not find a relationship between 'users' and 'user_profiles'`
- 在 schema cache 中找不到外鍵關聯

**初步判斷**:
1. 可能是 API 查詢語法錯誤
2. 可能是資料庫 schema 缺少表或外鍵
3. 可能是 Supabase schema cache 問題

### 2. 檢查 API 查詢代碼

**文件**: `web-admin/src/app/api/admin/bookings/route.ts`

**問題代碼** (第 40-54 行):
```typescript
let query = db.supabase
  .from('bookings')
  .select(`
    *,
    customer:customer_id (
      id,
      firebase_uid,
      email,
      user_profiles (first_name, last_name, phone)  // ❌ 問題在這裡
    ),
    driver:driver_id (
      id,
      firebase_uid,
      email,
      user_profiles (first_name, last_name, phone),  // ❌ 問題在這裡
      drivers (vehicle_type, vehicle_plate)          // ❌ 問題在這裡
    )
  `, { count: 'exact' })
```

**問題分析**:
- 查詢嘗試從 `users` 表關聯到 `user_profiles` 表
- 但是沒有指定正確的關聯路徑
- PostgREST 無法自動推斷關聯方式

### 4. 根本原因總結

**問題 1: 缺少資料表**
- Supabase migrations 中沒有創建 `user_profiles` 表
- Supabase migrations 中沒有創建 `drivers` 表
- 只有 `users` 和 `bookings` 表

**問題 2: API 查詢語法不正確**
- 查詢語法沒有指定正確的關聯路徑
- `users` → `user_profiles` 是反向關聯（one-to-one）
- 需要使用 `user_profiles!user_id` 語法明確指定

**關聯關係**:
```
bookings.customer_id → users.id
users.id ← user_profiles.user_id  (反向關聯)

bookings.driver_id → users.id
users.id ← drivers.user_id  (反向關聯)
```

```sql
-- 創建 user_profiles 表
CREATE TABLE IF NOT EXISTS user_profiles (

CREATE INDEX IF NOT EXISTS idx_drivers_vehicle_type ON drivers(vehicle_type);

```

**關鍵設計**:
- ✅ `user_profiles.user_id` 關聯到 `users.id`
- ✅ `drivers.user_id` 關聯到 `users.id`
- ✅ `UNIQUE(user_id)` 確保一對一關聯
- ✅ `ON DELETE CASCADE` 確保資料一致性

#### 步驟 2: 修改 API 查詢語法

**文件**: `web-admin/src/app/api/admin/bookings/route.ts`

**修改前**:
```typescript
let query = db.supabase
  .from('bookings')
  .select(`
    *,
    customer:customer_id (
      id,
      firebase_uid,
      email,
      user_profiles (first_name, last_name, phone)  // ❌ 錯誤
    ),
    driver:driver_id (
      id,
      firebase_uid,
      email,
      user_profiles (first_name, last_name, phone),  // ❌ 錯誤
      drivers (vehicle_type, vehicle_plate)          // ❌ 錯誤
    )
  `, { count: 'exact' })
```

**修改後**:
```typescript
let query = db.supabase
  .from('bookings')
  .select(`
    *,
    customer:customer_id (
      id,
      firebase_uid,
      email,
      user_profiles!user_id (first_name, last_name, phone)  // ✅ 正確
    ),
    driver:driver_id (
      id,
      firebase_uid,
      email,
      user_profiles!user_id (first_name, last_name, phone),  // ✅ 正確
      drivers!user_id (vehicle_type, vehicle_plate)          // ✅ 正確
    )
  `, { count: 'exact' })
```

**語法說明**:
- `user_profiles!user_id` - 使用 `user_id` 欄位進行反向關聯
- `!` 符號表示明確指定關聯欄位
- 這告訴 PostgREST 使用 `user_profiles.user_id = users.id` 進行關聯

```sql
SELECT
    tc.table_name,
    kcu.column_name,
    ccu.table_name AS foreign_table_name,

JOIN information_schema.key_column_usage AS kcu
  ON tc.constraint_name = kcu.constraint_name
JOIN information_schema.constraint_column_usage AS ccu
```

## 💡 開發心得和經驗總結

### 1. Supabase PostgREST 關聯查詢

**學習要點**:
- PostgREST 使用外鍵自動推斷關聯
- 反向關聯需要明確指定欄位名
- 使用 `!` 符號指定關聯欄位

**語法模式**:
```typescript
// 正向關聯（表 A 有外鍵指向表 B）
tableA.select('*, tableB(*)')

// 反向關聯（表 B 有外鍵指向表 A）
tableA.select('*, tableB!foreign_key_column(*)')
```

### 2. 資料庫 Schema 設計

**一對一關聯**:
```sql
CREATE TABLE users (
    id UUID PRIMARY KEY
);

CREATE TABLE user_profiles (
    id UUID PRIMARY KEY,
    user_id UUID NOT NULL REFERENCES users(id),
    UNIQUE(user_id)  -- 確保一對一
);
```

**好處**:
- 分離核心資料和擴展資料
- 提高查詢效能（不是所有查詢都需要 profile）
- 便於資料管理和隱私控制

### 3. Migration 管理

**最佳實踐**:
- ✅ 使用版本化的 migration 文件
- ✅ 每個 migration 只做一件事
- ✅ 包含驗證和回滾邏輯
- ✅ 添加詳細的註釋

**命名規範**:
```
YYYYMMDD_description.sql
例如: 20250104_create_user_profiles_and_drivers.sql
```

## ❌ 犯過的錯誤和教訓

### 錯誤 1: 最初使用錯誤的語法

**錯誤代碼**:
```typescript
user_profiles!user_profiles_user_id_fkey (...)
```

**問題**:
- 使用了外鍵約束名稱而不是欄位名
- 過於複雜

**正確代碼**:
```typescript
user_profiles!user_id (...)
```

**教訓**:
- 使用簡單明確的語法
- 欄位名比約束名更直觀

### 錯誤 2: 忘記添加 UNIQUE 約束

**問題**:
- 最初沒有添加 `UNIQUE(user_id)`
- 可能導致一對多關聯

**修復**:
```sql
CREATE TABLE user_profiles (
    ...
    user_id UUID NOT NULL REFERENCES users(id),
    UNIQUE(user_id)  -- ✅ 確保一對一
);
```

**教訓**:
- 一對一關聯必須有 UNIQUE 約束
- 資料庫層面的約束比應用層更可靠

### 錯誤 3: 沒有考慮現有資料

**問題**:
- 創建表時沒有考慮現有 users 資料
- user_profiles 表為空是正常的

**教訓**:
- 新表創建後可能為空
- 需要資料遷移或逐步填充
- 查詢時使用 LEFT JOIN 而不是 INNER JOIN
