### 診斷步驟 1: 檢查 Flutter ChatService

**問題 1**: 調用不存在的 API 端點
```dart

  Uri.parse('$_baseUrl/api/chat/send-message'),  // ❌ 端點不存在

  Uri.parse('$_baseUrl/api/chat/mark-read/$bookingId'),  // ❌ 端點不存在
```

**根本原因**: Backend 沒有實作這兩個 API 端點，導致返回 404 錯誤。

檢查 `backend/src/minimal-server.ts`，確認：

- ❌ `/api/chat` 路由**不存在**

### 診斷步驟 3: 檢查 Firestore 規則

檢查 `firebase/firestore.rules`，發現：

**問題 2**: 禁止客戶端寫入訊息

### 修復 2: 修改 Firestore 規則允許客戶端寫入訊息

**文件**: `firebase/firestore.rules`

**修改前** (Line 70-107):
```javascript

  allow read: if ...;
```

**修改後** (Line 70-120):
```javascript
match /chat_rooms/{roomId} {
  allow read: if ...;
  
  // ✅ 允許客戶端更新聊天室（lastMessage, unreadCount）
  allow update: if request.auth != null &&
    (resource.data.customerId == request.auth.uid ||
     resource.data.driverId == request.auth.uid) &&
    request.resource.data.diff(resource.data).affectedKeys()
      .hasOnly(['lastMessage', 'lastMessageTime', 'customerUnreadCount', 'driverUnreadCount', 'updatedAt']);

  // ❌ 禁止客戶端創建或刪除聊天室（由 Backend API 創建）
  allow create, delete: if false;

  match /messages/{messageId} {
    allow read: if ...;
    
    // ✅ 允許用戶創建訊息（發送訊息）
    allow create: if request.auth != null &&
      (get(/databases/$(database)/documents/chat_rooms/$(roomId)).data.customerId == request.auth.uid ||
       get(/databases/$(database)/documents/chat_rooms/$(roomId)).data.driverId == request.auth.uid) &&
      request.resource.data.senderId == request.auth.uid;

    // ❌ 禁止客戶端更新或刪除訊息
    allow update, delete: if false;
  }
}
```

## 📚 開發心得和經驗總結

### 1. 架構設計原則

**混合架構**:
- **聊天室創建**: Backend API（需要權限控制和資料驗證）
- **訊息發送**: 客戶端直接寫入 Firestore（性能和即時性）

**優點**:
- 聊天室創建有嚴格的權限控制
- 訊息發送無需經過 Backend，延遲更低
- Firestore 提供即時同步能力

### 2. UID 管理的重要性

**教訓**: 必須區分 Supabase UUID 和 Firebase UID

**Supabase UUID**:
- 格式: `xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx` (36 字符)
- 用途: Supabase 資料庫的主鍵
- 範例: `c03f0310-d3c8-44ab-8aec-1a4a858c52cb`

**Firebase UID**:
- 格式: 字母數字字符串 (28 字符)
- 用途: Firebase Authentication 的用戶 ID
- 範例: `hUu4fH5dTlW9VUYm6GojXvRLdni2`

**最佳實踐**:
```typescript
// ❌ 錯誤：直接使用 booking.customer_id
customerId: booking.customer_id  // Supabase UUID

// ✅ 正確：查詢對應的 Firebase UID
const { data: customer } = await supabase
  .from('users')
  .select('firebase_uid')
  .eq('id', booking.customer_id)
  .single();

customerId: customer.firebase_uid  // Firebase UID
```

### 3. Firestore 規則設計

**原則**: 最小權限原則

```javascript
// ✅ 好的規則：精確控制可修改的欄位
allow update: if request.auth != null &&
  (resource.data.customerId == request.auth.uid ||
   resource.data.driverId == request.auth.uid) &&
  request.resource.data.diff(resource.data).affectedKeys()
    .hasOnly(['lastMessage', 'lastMessageTime', 'customerUnreadCount', 'driverUnreadCount', 'updatedAt']);

// ❌ 壞的規則：全部禁止或全部允許
allow write: if false;  // 太嚴格
allow write: if true;   // 太寬鬆
```

## 🚨 遇到的困難和解決方法

### 困難 1: 客戶端看不到聊天室

**現象**: Firestore 查詢返回空列表

**原因**: `customerId` 是 Supabase UUID，但查詢用的是 Firebase UID

**解決方法**:
1. 創建診斷腳本檢查 UID 類型
2. 創建修復腳本更新現有聊天室
3. 修改 Backend 代碼使用正確的 UID

### 困難 2: Firestore 規則過於嚴格

**現象**: 客戶端寫入訊息時被拒絕

**原因**: 規則設置為 `allow write: if false`

**解決方法**:
1. 分析架構設計原則（聊天室創建 vs 訊息發送）
2. 修改規則允許客戶端創建訊息
3. 使用 `affectedKeys()` 精確控制可修改的欄位

### 困難 3: API 端點不存在

**現象**: Flutter 調用 API 返回 404

**原因**: Backend 沒有實作 `/api/chat` 路由

**解決方法**:
1. 分析是否真的需要 API 端點
2. 決定採用客戶端直接寫入 Firestore 的方案
3. 修改 Flutter 代碼移除 API 調用

## 💡 犯過的錯誤和教訓

### 錯誤 1: 沒有驗證 UID 類型

**錯誤代碼**:

**錯誤代碼**:
```javascript
allow write: if false;  // 禁止所有寫入
```

**教訓**:
- 理解架構設計原則
- 區分「創建」和「更新」權限
- 使用 `affectedKeys()` 精確控制

### 錯誤 3: 沒有清理未使用的代碼

**錯誤**: 保留了 API 調用代碼但沒有實作 API

**教訓**:
- 決定架構後立即修改代碼
- 移除未使用的導入和變數
- 保持代碼整潔
