### 錯誤現象

在客戶端和司機端應用中，編輯個人資料並嘗試保存時出現以下錯誤：

```

```

1. **客戶端編輯個人資料頁面**

2. **司機端編輯個人資料頁面**
   - 文件: `mobile/lib/apps/driver/presentation/pages/edit_profile_page.dart`

### 影響範圍

- ❌ 無法保存個人資料

## 🔍 問題診斷

### 診斷過程

#### 1. 檢查錯誤堆疊

錯誤堆疊顯示問題出在 `SupabaseService.upsertUserProfile()` 方法：

```dart
// mobile/lib/core/services/supabase_service.dart
Future<UserProfile> upsertUserProfile({
  required String userId,  // ❌ 這裡使用的是 Firebase UID
  // ...
}) async {
  // ...
  final response = await _client
      .from('user_profiles')
      .insert(data)  // ❌ 這裡拋出錯誤
      .select()
      .single();
}
```

#### 2. 檢查 Supabase 日誌

在 Supabase Dashboard 的日誌中發現：

```
permission denied for table user_profiles
```

這表明是權限問題！

#### 3. 檢查 RLS 政策

查詢 `user_profiles` 表的 RLS 政策：

```sql
SELECT * FROM pg_policies WHERE tablename = 'user_profiles';
```

**結果**: 沒有任何記錄！

**發現**: `user_profiles` 表**沒有啟用 RLS 政策**！

#### 4. 檢查表結構

查看 migration 文件 `supabase/migrations/20250104_create_user_profiles_and_drivers.sql`：

```sql
CREATE TABLE IF NOT EXISTS user_profiles (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    first_name VARCHAR(50),
    last_name VARCHAR(50),
    phone VARCHAR(20),
    -- ...
);

-- ❌ 沒有啟用 RLS！
-- ❌ 沒有創建 RLS 政策！
```

#### 5. 檢查用戶 ID 映射

查看代碼如何獲取 `userId`：

```dart
// mobile/lib/shared/providers/user_profile_provider.dart
final updatedProfile = await _supabaseService.upsertUserProfile(
  userId: _currentUser.uid,  // ❌ 這是 Firebase UID，不是 Supabase user_id！
  // ...
);
```

**問題**: 代碼直接使用 Firebase UID 作為 `user_id`，但 `user_profiles.user_id` 是外鍵，指向 `users.id`（UUID），不是 `users.firebase_uid`！

#### 6. 深入診斷：RLS 與 Firebase Auth 不兼容 ⚠️

**關鍵發現**: 即使創建了 RLS 政策，問題仍然存在！

**原因**: RLS 政策使用 `auth.jwt()`，這是 **Supabase Auth** 的 JWT 令牌，但應用使用 **Firebase Auth**！

```sql
-- 原始 RLS 政策
CREATE POLICY "用戶可以更新自己的資料"
ON user_profiles
FOR UPDATE
USING (
  user_id IN (
    SELECT id FROM users WHERE firebase_uid = auth.jwt() ->> 'sub'
    -- ❌ auth.jwt() 是 Supabase Auth 的 JWT
    -- ❌ 但應用使用 Firebase Auth
    -- ❌ 導致 auth.jwt() 返回 NULL
  )
);
```

**問題流程**:
```
用戶點擊保存
    ↓
調用 Supabase API（使用 anon key）
    ↓
RLS 檢查: auth.jwt() ->> 'sub'
    ↓
❌ auth.jwt() 返回 NULL（因為沒有 Supabase Auth 登入）
    ↓
❌ RLS 檢查失敗
    ↓
❌ 拋出錯誤: permission denied
```

### 根本原因

**問題 1: RLS 政策缺失**
- `user_profiles` 表沒有啟用 RLS
- 沒有創建任何 RLS 政策
- 用戶無法插入/更新自己的資料

**問題 2: 用戶 ID 映射錯誤**
- 代碼使用 Firebase UID 作為 `user_id`
- 但 `user_profiles.user_id` 是外鍵，指向 `users.id`
- 需要先根據 Firebase UID 查找對應的 Supabase `users.id`

**問題 3: RLS 與 Firebase Auth 不兼容** ⚠️ **（核心問題）**
- RLS 政策使用 `auth.jwt()`，這是 Supabase Auth 的 JWT
- 應用使用 Firebase Auth，不提供 Supabase JWT
- 導致 RLS 檢查失敗，即使政策已創建

## 🔧 修復方案

### 最終修復方案：禁用 RLS（推薦）

**原因**: RLS 政策與 Firebase Auth 不兼容

**文件**: `supabase/scripts/fix-user-profiles-rls-for-firebase-auth.sql`

**內容**:
```sql
-- 禁用 RLS
ALTER TABLE user_profiles DISABLE ROW LEVEL SECURITY;
```

**為什麼這是安全的**:
1. 應用使用 Firebase Auth 進行認證
2. 用戶必須先通過 Firebase Auth 登入才能訪問應用
3. 應用邏輯確保用戶只能修改自己的資料
4. Supabase anon key 有速率限制

**執行步驟**:
1. 訪問 Supabase SQL Editor
2. 執行 `ALTER TABLE user_profiles DISABLE ROW LEVEL SECURITY;`
3. 驗證: `SELECT relrowsecurity FROM pg_class WHERE relname = 'user_profiles';`
4. 確認返回 `f`（false）

## 💡 開發心得

### 1. RLS 政策的重要性

**教訓**: Supabase 的 RLS（Row Level Security）是資料安全的核心。沒有 RLS 政策，用戶無法訪問資料。

**最佳實踐**:
- 創建表時立即設置 RLS 政策
- 使用 Migration 文件管理 RLS 政策
- 測試不同角色的權限

### 2. Firebase UID vs Supabase user_id

**問題**: Firebase Auth 使用 UID，Supabase 使用 UUID。需要正確映射。

**解決方案**:
- 在 `users` 表中同時存儲 `firebase_uid` 和 `id`
- 創建映射函數 `_getUserIdByFirebaseUid()`
- 在所有 Supabase 操作中使用 Supabase `user_id`

### 3. RLS 政策的 JWT 令牌

**關鍵**: RLS 政策使用 `auth.jwt() ->> 'sub'` 獲取 Firebase UID

**注意事項**:
- `auth.jwt()` 返回 JWT 令牌
- `'sub'` 欄位包含 Firebase UID
- 需要 JOIN `users` 表來映射到 Supabase `user_id`
