## 系統架構概述

### 雙資料庫架構

本系統採用 **Supabase (PostgreSQL)** 和 **Firebase Firestore** 雙資料庫架構：

- **Supabase PostgreSQL**：作為 **單一真實來源 (Single Source of Truth)**
  - 儲存完整的訂單資料
  - 支援複雜查詢和事務處理
  - 所有寫入操作都通過 Backend API 進行

- **Firebase Firestore**：作為 **即時資料層 (Real-time Data Layer)**
  - 為移動端 APP 提供即時訂單更新
  - 支援離線同步
  - 只讀取，不直接寫入

### CQRS 架構

- **Command (寫入)**：所有寫入操作通過 Backend API → Supabase
- **Query (讀取)**：移動端 APP 從 Firestore 讀取即時資料
- **同步機制**：Outbox Pattern + Edge Function

## 訂單狀態定義

### Supabase 訂單狀態（詳細狀態）

Supabase 使用詳細的訂單狀態來追蹤訂單的每個階段：

| 狀態值 | 中文名稱 | 說明 | 觸發操作 | Firestore 映射 |
|--------|---------|------|---------|---------------|
| `pending_payment` | 待支付 | 訂單已創建，等待支付訂金 | 創建訂單 | `pending` |
| `paid_deposit` | 已付訂金 | 訂金已支付，等待派單 | 支付訂金 | `pending` |
| `matched` | 已分配司機 | 系統已分配司機，等待司機確認 | 手動派單 | `awaitingDriver` |
| `driver_confirmed` | 司機已確認 | 司機已確認接單 | 司機確認接單 | `matched` |
| `driver_departed` | 司機已出發 | 司機已出發前往載客地點 | 司機點擊「出發前往載客」 | `inProgress` |
| `driver_arrived` | 司機已到達 | 司機已到達載客地點 | 司機點擊「抵達上車地點」 | `inProgress` |
| `trip_started` | 行程進行中 | 客戶已上車，行程開始 | 客戶點擊「開始行程」 | `inProgress` |
| `trip_ended` | 行程已結束 | 行程已結束，等待支付尾款 | 客戶點擊「結束行程」 | `awaitingBalance` |
| `completed` | 訂單完成 | 尾款已支付，訂單完成 | 支付尾款 | `completed` |
| `cancelled` | 已取消 | 訂單已取消 | 取消訂單 | `cancelled` |

### Flutter APP 訂單狀態（簡化狀態）

Flutter APP 使用簡化的訂單狀態枚舉（`BookingStatus`）：

```dart
enum BookingStatus {
  @JsonValue('pending')
  pending,        // 待配對（待付訂金或待派單）

  @JsonValue('awaitingDriver')
  awaitingDriver, // 待司機確認

  @JsonValue('matched')
  matched,        // 已配對

  @JsonValue('inProgress')
  inProgress,     // 進行中

  @JsonValue('awaitingBalance')
  awaitingBalance, // 待付尾款

  @JsonValue('completed')
  completed,      // 已完成

  @JsonValue('cancelled')
  cancelled,      // 已取消
}
```

**狀態說明**：
- `pending`：訂單已創建但尚未派單（包含待付訂金和已付訂金等待派單）
- `awaitingDriver`：已分配司機，等待司機確認接單
- `matched`：司機已確認接單，準備出發
- `inProgress`：行程進行中（包含司機已出發、已到達、行程開始）
- `awaitingBalance`：行程已結束，等待客戶支付尾款
- `completed`：訂單完成
- `cancelled`：訂單已取消

## 資料同步流程

### 完整訂單流程

```
1. 客戶創建訂單
   ↓ Backend API: POST /api/bookings
   ↓ Supabase: INSERT INTO bookings (status = 'pending_payment')
   ↓ Trigger: 寫入 outbox 表
   ↓ Edge Function: 同步到 Firestore (status = 'pending')

2. 客戶支付訂金
   ↓ Backend API: POST /api/bookings/:id/pay-deposit
   ↓ Supabase: UPDATE bookings SET status = 'paid_deposit'
   ↓ Trigger: 寫入 outbox 表
   ↓ Edge Function: 同步到 Firestore (status = 'pending')

3. 管理員手動派單
   ↓ Backend API: POST /api/bookings/:id/assign
   ↓ Supabase: UPDATE bookings SET status = 'matched', driver_id = ?
   ↓ Trigger: 寫入 outbox 表
   ↓ Edge Function: 同步到 Firestore (status = 'pending')

4. 司機確認接單
   ↓ Backend API: POST /api/booking-flow/bookings/:id/accept
   ↓ Supabase: UPDATE bookings SET status = 'driver_confirmed'
   ↓ Trigger: 寫入 outbox 表
   ↓ Edge Function: 同步到 Firestore (status = 'matched')
   ↓ 創建 Firestore 聊天室

5. 司機出發前往載客
   ↓ Backend API: POST /api/booking-flow/bookings/:id/depart
   ↓ Supabase: UPDATE bookings SET status = 'driver_departed'
   ↓ Trigger: 寫入 outbox 表
   ↓ Edge Function: 同步到 Firestore (status = 'inProgress')
   ↓ 發送系統訊息到聊天室

6. 司機抵達上車地點
   ↓ Backend API: POST /api/booking-flow/bookings/:id/arrive
   ↓ Supabase: UPDATE bookings SET status = 'driver_arrived'
   ↓ Trigger: 寫入 outbox 表
   ↓ Edge Function: 同步到 Firestore (status = 'inProgress')
   ↓ 發送系統訊息到聊天室

7. 客戶開始行程
   ↓ Backend API: POST /api/booking-flow/bookings/:id/start
   ↓ Supabase: UPDATE bookings SET status = 'trip_started', actual_start_time = NOW()
   ↓ Trigger: 寫入 outbox 表
   ↓ Edge Function: 同步到 Firestore (status = 'inProgress')
   ↓ 發送系統訊息到聊天室

8. 客戶結束行程
   ↓ Backend API: POST /api/booking-flow/bookings/:id/end
   ↓ Supabase: UPDATE bookings SET status = 'trip_ended', actual_end_time = NOW()
   ↓ Trigger: 寫入 outbox 表
   ↓ Edge Function: 同步到 Firestore (status = 'completed')
   ↓ 發送系統訊息到聊天室
   ↓ Flutter APP: 自動導航到支付尾款頁面

9. 客戶支付尾款
   ↓ Backend API: POST /api/booking-flow/bookings/:id/pay-balance
   ↓ Supabase: UPDATE bookings SET status = 'completed'
   ↓ Trigger: 寫入 outbox 表
   ↓ Edge Function: 同步到 Firestore (status = 'completed')
   ↓ 發送系統訊息到聊天室
   ↓ Flutter APP: 自動導航到訂單完成頁面
```

## Outbox Pattern 實作

### 什麼是 Outbox Pattern？

Outbox Pattern 是一種確保資料一致性的設計模式：

1. **原子性寫入**：在同一個資料庫事務中，同時寫入業務資料和事件記錄
2. **非同步處理**：由獨立的處理程序（Edge Function）消費事件佇列
3. **至少一次交付**：確保事件不會丟失，但可能重複處理（需要冪等性）

### Outbox 表結構

```sql
CREATE TABLE IF NOT EXISTS outbox (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  aggregate_type VARCHAR(50) NOT NULL,  -- 聚合類型（'booking', 'chat_message'）
  aggregate_id VARCHAR(255) NOT NULL,   -- 聚合 ID（訂單 ID、訊息 ID）
  event_type VARCHAR(50) NOT NULL,      -- 事件類型（'created', 'updated', 'deleted'）
  payload JSONB NOT NULL,                -- 事件資料（JSON 格式）
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  processed_at TIMESTAMP WITH TIME ZONE,  -- 處理時間（NULL = 未處理）
  retry_count INTEGER DEFAULT 0,         -- 重試次數
  error_message TEXT                     -- 錯誤訊息
);
```

### Trigger 函數

當 `bookings` 表發生 INSERT 或 UPDATE 時，自動寫入 `outbox` 表：

```sql
CREATE OR REPLACE FUNCTION bookings_to_outbox()
RETURNS TRIGGER AS $$
BEGIN
  INSERT INTO outbox (
    aggregate_type,
    aggregate_id,
    event_type,
    payload
  ) VALUES (
    'booking',
    NEW.id::TEXT,
    CASE
      WHEN TG_OP = 'INSERT' THEN 'created'
      WHEN TG_OP = 'UPDATE' THEN 'updated'
      WHEN TG_OP = 'DELETE' THEN 'deleted'
    END,
    jsonb_build_object(
      'id', NEW.id,
      'status', NEW.status,
      'customerId', (SELECT firebase_uid FROM users WHERE id = NEW.customer_id),
      'driverId', (SELECT firebase_uid FROM users WHERE id = NEW.driver_id),
      -- ... 其他欄位
    )
  );
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

### Edge Function 處理流程

1. **查詢未處理事件**：`WHERE processed_at IS NULL AND retry_count < 3`
2. **批次處理**：每次最多處理 10 個事件
3. **狀態映射**：將 Supabase 狀態轉換為 Firestore 狀態
4. **雙寫策略**：同時寫入 `orders_rt` 和 `bookings` 集合
5. **標記已處理**：`UPDATE outbox SET processed_at = NOW()`
6. **錯誤重試**：失敗時增加 `retry_count`，最多重試 3 次

## 常見問題

### Q1: 為什麼需要狀態映射？

**A**: Supabase 使用詳細狀態來追蹤訂單的每個階段，但 Flutter APP 只需要簡化的狀態來顯示 UI。狀態映射可以：
- 減少 APP 端的複雜度
- 統一 UI 顯示邏輯
- 方便未來擴展新狀態

### Q2: 如果 Edge Function 失敗怎麼辦？

**A**: Outbox Pattern 提供了重試機制：
- 事件會保留在 `outbox` 表中（`processed_at = NULL`）
- Edge Function 會自動重試（最多 3 次）
- 可以手動觸發 Edge Function 重新處理失敗的事件

### Q3: 如何確保資料一致性？

**A**: 
- **Supabase 是單一真實來源**：所有寫入操作都通過 Backend API → Supabase
- **Firestore 只讀取**：移動端 APP 不直接寫入 Firestore
- **Outbox Pattern**：確保事件不會丟失
- **冪等性設計**：重複處理同一事件不會產生副作用

### Q4: 如何部署 Edge Function？

**A**: 
```bash
npx supabase functions deploy sync-to-firestore
```

### Q5: 如何手動觸發同步？

**A**: 
```bash
curl -X POST https://your-project.supabase.co/functions/v1/sync-to-firestore \
  -H "Authorization: Bearer YOUR_ANON_KEY"
```
