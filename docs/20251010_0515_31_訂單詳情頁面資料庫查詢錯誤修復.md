# 訂單詳情頁面資料庫查詢錯誤修復

**日期**: 2025-10-10  
**時間**: 05:15

**主旨**: 修復訂單詳情頁面資料庫查詢中 full_name 欄位不存在的錯誤  
**狀態**: ✅ 已完成

---

## 📋 問題描述

### 當前狀況

1. 在訂單列表頁面點擊「查看詳情」按鈕後，訂單詳情頁面顯示「訂單不存在」
2. 實際上訂單存在於資料庫中（訂單 ID: `5fc778ac-a309-4d97-9814-28e6b931ddc3`）

### 錯誤訊息

**終端機顯示**:
```
📋 獲取訂單詳情: { bookingId: '5fc778ac-a309-4d97-9814-28e6b931ddc3' }
❌ 獲取訂單失敗: {
  code: '42703',
  details: null,
  hint: null,
  message: 'column user_profiles_2.full_name does not exist'
}
```

### 根本原因

- PostgreSQL 錯誤代碼 `42703` 表示「欄位不存在」
- 錯誤訊息指出 `user_profiles_2.full_name` 欄位不存在
- 這是 Supabase 關聯查詢時產生的別名問題
- 問題出在訂單詳情 API（`web-admin/src/app/api/admin/bookings/[id]/route.ts`）的 SQL 查詢語法

---

## 🔍 問題診斷過程

### 1. 分析錯誤訊息

**錯誤代碼**: `42703`  
**錯誤訊息**: "column user_profiles_2.full_name does not exist"

**分析**:
- PostgreSQL 錯誤代碼 42703 表示嘗試訪問不存在的欄位
- `user_profiles_2` 是 Supabase 在關聯查詢時自動生成的別名
- 錯誤明確指出 `full_name` 欄位在 `user_profiles` 表中不存在

### 2. 檢查資料庫結構

**查看遷移文件**: `supabase/migrations/20250104_create_user_profiles_and_drivers.sql`

**user_profiles 表結構**:
```sql
CREATE TABLE IF NOT EXISTS user_profiles (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    first_name VARCHAR(50),      -- ✅ 名字
    last_name VARCHAR(50),        -- ✅ 姓氏
    phone VARCHAR(20),
    avatar_url TEXT,
    date_of_birth DATE,
    gender VARCHAR(10) CHECK (gender IN ('male', 'female', 'other')),
    address TEXT,
    emergency_contact_name VARCHAR(100),
    emergency_contact_phone VARCHAR(20),
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    UNIQUE(user_id)
);
```

**發現問題**: ❌ 表中沒有 `full_name` 欄位，而是有 `first_name` 和 `last_name` 兩個欄位

### 3. 檢查 API 查詢語法

**訂單詳情 API** (`web-admin/src/app/api/admin/bookings/[id]/route.ts`):

**錯誤的查詢**:
```typescript
.select(`
  *,
  customer:users!customer_id (
    id,
    email,
    user_profiles!user_id (
      full_name,    // ❌ 欄位不存在
      phone
    )
  ),
  driver:users!driver_id (
    id,
    email,
    user_profiles!user_id (
      full_name,    // ❌ 欄位不存在
      phone
    ),
    drivers!user_id (
      vehicle_type,
      vehicle_plate,
      rating
    )
  )
`)
```

### 4. 根本原因總結

1. ❌ API 查詢使用了不存在的 `full_name` 欄位
2. ✅ 資料庫實際有 `first_name` 和 `last_name` 兩個欄位
3. ❌ 格式化代碼也使用了 `full_name`

## 🔧 修復方案

### 方案概述

1. 修改 Supabase 查詢，將 `full_name` 改為 `first_name` 和 `last_name`
2. 修改格式化代碼，組合 `first_name` 和 `last_name` 為完整姓名

### 修復步驟

#### 步驟 1: 修改查詢語法

**修改前**:
```typescript
user_profiles!user_id (
  full_name,
  phone
)
```

**修改後**:
```typescript
user_profiles!user_id (
  first_name,
  last_name,
  phone
)
```

#### 步驟 2: 修改客戶資料格式化

**修改前**:
```typescript
const formattedCustomer = booking.customer ? {
  id: booking.customer.id,
  email: booking.customer.email,
  name: customerProfile?.full_name || '未知客戶',
  phone: customerProfile?.phone || '無電話',
} : null;
```

**修改後**:
```typescript
const formattedCustomer = booking.customer ? {
  id: booking.customer.id,
  email: booking.customer.email,
  name: customerProfile 
    ? `${customerProfile.first_name || ''} ${customerProfile.last_name || ''}`.trim() || '未知客戶'
    : '未知客戶',
  phone: customerProfile?.phone || '無電話',
} : null;
```

#### 步驟 3: 修改司機資料格式化

**修改前**:
```typescript

  id: booking.driver.id,

  vehicleType: driverInfo?.vehicle_type || '-',
  vehiclePlate: driverInfo?.vehicle_plate || '-',
  rating: driverInfo?.rating || 0,

```

**修改後**:
```typescript

  id: booking.driver.id,
  email: booking.driver.email,
  name: driverProfile 
    ? `${driverProfile.first_name || ''} ${driverProfile.last_name || ''}`.trim() || '未知司機'
```

## 💡 開發心得和經驗總結

### 1. 資料庫欄位名稱的確認

**學習要點**:
- 開發前應該先檢查資料庫表結構
- 確認所有要查詢的欄位都存在於資料庫中
- 不要假設欄位名稱，應該查看遷移文件或資料庫結構

**最佳實踐**:
```typescript
// ❌ 不好的做法：假設欄位名稱
user_profiles!user_id (
  full_name,  // 假設有這個欄位
  phone
)

// ✅ 好的做法：確認實際欄位名稱
user_profiles!user_id (
  first_name,  // 確認存在的欄位
  last_name,   // 確認存在的欄位
  phone
)
```

### 2. PostgreSQL 錯誤代碼解讀

**常見錯誤代碼**:
- `42703` - 欄位不存在
- `42P01` - 表不存在
- `23503` - 外鍵約束違反
- `23505` - 唯一約束違反

**學習要點**:
- 錯誤代碼可以快速定位問題類型
- 錯誤訊息通常會明確指出問題所在
- 仔細閱讀錯誤訊息可以節省調試時間

### 3. 姓名欄位的處理

**設計考量**:
- 分開儲存 `first_name` 和 `last_name` 更靈活
- 可以支援不同的姓名格式（西方、東方等）
- 查詢時需要組合成完整姓名

**組合姓名的最佳實踐**:
```typescript
// ✅ 好的做法：處理空值並去除多餘空格
const name = customerProfile 
  ? `${customerProfile.first_name || ''} ${customerProfile.last_name || ''}`.trim() || '未知客戶'
  : '未知客戶';

// 說明：
// 1. 使用 || '' 處理 null/undefined
// 2. 使用 .trim() 去除首尾空格
// 3. 使用 || '未知客戶' 處理空字串
```

### 4. Supabase 關聯查詢

**學習要點**:
- Supabase 會自動為關聯表生成別名（如 `user_profiles_2`）
- 查詢的欄位必須存在於資料庫中
- 可以嵌套查詢多層關聯資料

**查詢語法**:
```typescript
.select(`
  *,
  customer:users!customer_id (
    id,
    email,
    user_profiles!user_id (
      first_name,
      last_name,
      phone
    )
  )
`)
```

## 🐛 遇到的困難和解決方法

### 困難 1: 確定正確的欄位名稱

**問題**: 不確定 `user_profiles` 表中的實際欄位名稱

**解決**:
1. 查看資料庫遷移文件
2. 確認表結構和欄位名稱
3. 使用實際存在的欄位

### 困難 2: 組合姓名時處理空值

**問題**: `first_name` 或 `last_name` 可能為 null

**解決**:
```typescript
// 使用 || '' 處理 null/undefined
`${customerProfile.first_name || ''} ${customerProfile.last_name || ''}`.trim()
```

### 困難 3: 處理組合後的空字串

**問題**: 如果 `first_name` 和 `last_name` 都為空，組合後會是空字串

**解決**:
```typescript
// 使用 || '未知客戶' 處理空字串
`${customerProfile.first_name || ''} ${customerProfile.last_name || ''}`.trim() || '未知客戶'
```
