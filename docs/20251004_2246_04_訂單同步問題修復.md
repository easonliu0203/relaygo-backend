# 訂單同步問題修復

**日期**：2025-10-04  
**時間**：22:46  
**編號**：04  
**主旨**：修復 Supabase 到 Firestore 的訂單同步問題

---

## 📋 問題描述

### 用戶報告

用戶在測試完整流程時遇到問題：

**症狀**：
- 手機 App 顯示「訂單不存在」錯誤
- 但在 Supabase Dashboard 中已確認：
  - ✅ `bookings` 表中有新訂單記錄
  - ✅ `outbox` 表中有對應的事件記錄

**這表示**：
- 訂單創建成功（之前的修復有效）
- Trigger 正常工作（事件已創建）
- 但同步到 Firestore 有問題

### 預期行為

**正常流程**：
```
1. 用戶創建訂單
2. API 寫入 Supabase bookings 表
3. Trigger 自動創建 outbox 事件
4. Cron Job 每 30 秒觸發 Edge Function
5. Edge Function 讀取 outbox 事件
6. Edge Function 同步到 Firestore orders_rt 集合
7. 客戶端從 Firestore 讀取訂單
8. 顯示訂單詳情
```

**實際情況**：
```
1-3. ✅ 正常
4. ❓ Cron Job 可能執行
5-6. ❌ Edge Function 沒有處理事件
7-8. ❌ Firestore 沒有資料，客戶端顯示「訂單不存在」
```

## 🔍 診斷過程

### 步驟 1：創建診斷查詢

創建了 `supabase/diagnose-sync-issue.sql`，包含 6 個檢查：

1. **最近的訂單和同步狀態** - 檢查訂單和對應事件的同步狀態
2. **Outbox 事件詳情** - 檢查事件的 payload 和錯誤訊息
3. **Cron Job 執行記錄** - 檢查 Cron Job 是否執行
4. **Cron Job 配置** - 檢查 Cron Job 設置
5. **未處理事件統計** - 統計積壓的事件
6. **錯誤事件** - 查找有錯誤的事件

### 步驟 2：檢查 Edge Function 代碼

檢查了 `supabase/functions/sync-to-firestore/index.ts`，發現關鍵問題：

**問題 1：aggregate_type 不匹配**

```typescript
// Edge Function 中的代碼（第 100 行）
if (event.aggregate_type === 'order') {  // ❌ 檢查 'order'
  await syncOrderToFirestore(event)
}
```

但檢查 Trigger 代碼（`supabase/migrations/20250101_create_outbox_table.sql`）：

```sql
-- Trigger 創建的事件（第 32 行）
INSERT INTO outbox (
  aggregate_type,
  ...
) VALUES (
  'booking',  -- ✅ 實際是 'booking'
  ...
);
```

**結果**：Edge Function 永遠不會處理任何事件，因為類型不匹配！

**問題 2：payload 欄位不匹配**

Edge Function 期望的欄位：
```typescript
const firestoreData = {
  dropoffAddress: orderData.dropoffAddress,  // ❌ Trigger 沒有這個欄位
  bookingTime: orderData.bookingTime,        // ❌ Trigger 沒有這個欄位
  passengerCount: orderData.passengerCount,  // ❌ Trigger 沒有這個欄位
  ...
}
```

Trigger 實際提供的欄位：
```sql
jsonb_build_object(
  'bookingNumber', NEW.booking_number,  -- ✅ 實際欄位
  'destination', NEW.destination,       -- ✅ 實際欄位
  'startDate', NEW.start_date,          -- ✅ 實際欄位
  'startTime', NEW.start_time,          -- ✅ 實際欄位
  ...
)
```

**結果**：即使 Edge Function 處理事件，也會因為欄位缺失而失敗！

### 步驟 3：確認根本原因

**根本原因**：
1. Edge Function 和 Trigger 的代碼不同步
2. Edge Function 使用舊的欄位名稱（可能是從舊版本遺留）
3. 沒有測試完整的同步流程

**為什麼之前沒發現**：
- 訂單創建 API 的修復只涉及 API 層
- 沒有測試到 Firestore 同步這一步
- Edge Function 的錯誤沒有顯示在客戶端

## ✅ 解決方案

### 修復內容

**檔案**：`supabase/functions/sync-to-firestore/index.ts`

#### 修復 1：aggregate_type 檢查（第 99-104 行）

```typescript
// 修改前
if (event.aggregate_type === 'order') {  // ❌ 錯誤
  await syncOrderToFirestore(event)
}

// 修改後
if (event.aggregate_type === 'booking') {  // ✅ 正確

```

#### 修復 2：函數重命名和 payload 映射（第 129-174 行）

```typescript
// 修改前
async function syncOrderToFirestore(event: OutboxEvent): Promise<void> {
  const orderId = event.aggregate_id
  const orderData = event.payload

  const firestoreData = {
    dropoffAddress: orderData.dropoffAddress,  // ❌ 不存在
    bookingTime: orderData.bookingTime,        // ❌ 不存在
    ...
  }
  ...
}

// 修改後
async function syncBookingToFirestore(event: OutboxEvent): Promise<void> {
  const bookingId = event.aggregate_id
  const bookingData = event.payload

  console.log(`同步訂單到 Firestore: ${bookingId}`, bookingData)

  const firestoreData = {
    id: bookingData.id,
    bookingNumber: bookingData.bookingNumber,      // ✅ 匹配 Trigger
    customerId: bookingData.customerId,
    status: bookingData.status,
    pickupAddress: bookingData.pickupAddress || '',
    destination: bookingData.destination || '',    // ✅ 匹配 Trigger
    startDate: bookingData.startDate,              // ✅ 匹配 Trigger
    startTime: bookingData.startTime,              // ✅ 匹配 Trigger
    durationHours: bookingData.durationHours,      // ✅ 匹配 Trigger
    vehicleType: bookingData.vehicleType,          // ✅ 匹配 Trigger
    specialRequirements: bookingData.specialRequirements || '',
    requiresForeignLanguage: bookingData.requiresForeignLanguage || false,
    basePrice: bookingData.basePrice,
    foreignLanguageSurcharge: bookingData.foreignLanguageSurcharge || 0,
    overtimeFee: bookingData.overtimeFee || 0,
    tipAmount: bookingData.tipAmount || 0,
    totalAmount: bookingData.totalAmount,
    depositAmount: bookingData.depositAmount,
    pickupLocation: bookingData.pickupLocation || { latitude: 0, longitude: 0 },
    createdAt: bookingData.createdAt,
    updatedAt: bookingData.updatedAt,
    driverId: bookingData.driverId || null,
    actualStartTime: bookingData.actualStartTime || null,
    actualEndTime: bookingData.actualEndTime || null,
  }

  if (event.event_type === 'deleted') {
    await deleteFirestoreDocument(bookingId)
  } else {
    await upsertFirestoreDocument(bookingId, firestoreData)
  }
}
```

## 💡 經驗教訓

### 技術層面

1. **代碼同步的重要性**
   - Edge Function 和 Trigger 必須使用相同的欄位名稱
   - 修改 schema 或 Trigger 時，必須同步更新 Edge Function
   - 需要建立代碼審查流程

2. **類型安全**
   - 應該使用共享的 TypeScript 類型定義
   - 可以從 Supabase schema 自動生成類型
   - 避免硬編碼欄位名稱

3. **完整測試的重要性**
   - 不能只測試單個組件
   - 必須測試完整的端到端流程
   - 包括所有異步操作（Trigger, Cron Job, Edge Function）

### 除錯層面

1. **診斷工具的價值**
   - 創建專門的診斷查詢可以快速定位問題
   - 結構化的檢查清單避免遺漏
   - 清晰的結果解讀指南節省時間

2. **日誌的重要性**
   - 詳細的日誌可以快速定位問題
   - 使用視覺指標（✅ ❌）提高可讀性
   - 記錄關鍵步驟和錯誤詳情

3. **分層診斷**
   - 從客戶端開始，逐層往下檢查
   - 確認每一層的輸入和輸出
   - 找到第一個失敗的環節

### 流程層面

1. **文檔的重要性**
   - 詳細的測試指南幫助用戶自助
   - 故障排除指南減少重複問題
   - 開發歷程記錄避免重複錯誤

2. **自動化的價值**
   - 部署腳本減少人為錯誤
   - 診斷腳本標準化檢查流程
   - 驗證腳本確保修復有效

3. **用戶溝通**
   - 清晰的步驟說明
   - 明確的成功標準
   - 及時的進度更新
