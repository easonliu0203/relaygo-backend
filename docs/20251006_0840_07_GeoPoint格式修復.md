# GeoPoint 格式修復

**日期**：2025-10-06 08:40  
**編號**：07  
**主旨**：修復 Firestore GeoPoint 格式錯誤導致客戶端解析失敗

---

## 📋 問題描述

### 症狀
1. ❌ 客戶端 App 顯示「載入訂單失敗」
2. ❌ 錯誤訊息：`type '_Map<String, dynamic>' is not a subtype of type 'GeoPoint'`
3. ❌ 有時顯示「訂單不存在」

### 錯誤截圖分析

**錯誤訊息 1**：
```
載入訂單失敗
type '_Map<String, dynamic>' is not a subtype of type 'GeoPoint'
```

**錯誤訊息 2**：
```
訂單不存在
```

---

## 🔍 問題診斷過程

### 1. 分析錯誤訊息

**關鍵線索**：
```
type '_Map<String, dynamic>' is not a subtype of type 'GeoPoint'
```

**診斷**：
- 客戶端期望 `pickupLocation` 和 `dropoffLocation` 是 **Firestore GeoPoint** 類型
- 但實際收到的是 **Map<String, dynamic>** 類型
- Dart 無法將 Map 轉換為 GeoPoint，導致解析失敗

---

### 2. 檢查客戶端代碼

**查看代碼**：`mobile/lib/core/models/booking_order.dart`

```dart
factory BookingOrder.fromFirestore(DocumentSnapshot doc) {
  final data = doc.data() as Map<String, dynamic>;
  
  return BookingOrder(
    ...
    pickupLocation: LocationPoint.fromGeoPoint(data['pickupLocation']),  // ✅ 期望 GeoPoint
    dropoffLocation: LocationPoint.fromGeoPoint(data['dropoffLocation']), // ✅ 期望 GeoPoint
    ...
  );
}
```

**發現**：
- 客戶端使用 `LocationPoint.fromGeoPoint()` 解析
- 這個方法期望接收 Firestore 的 `GeoPoint` 類型
- 不是普通的 Map 或 Object

---

### 3. 檢查 Edge Function 寫入的格式

**修復前的代碼**：

```typescript
const firestoreData = {
  ...
  pickupLocation: bookingData.pickupLocation || { latitude: 0, longitude: 0 },  // ❌ 錯誤
  dropoffLocation: { latitude: 0, longitude: 0 },  // ❌ 錯誤
  ...
}
```

**問題**：
- 直接寫入 `{ latitude: 0, longitude: 0 }` 格式
- Firestore REST API 會將其存儲為普通的 Map
- 不是 GeoPoint 類型

---

### 4. 檢查 Firestore REST API 文檔

**GeoPoint 的正確格式**：

在 Firestore REST API 中，GeoPoint 必須使用特定的格式：

```json
{
  "geoPointValue": {
    "latitude": 25.0330,
    "longitude": 121.5654
  }
}
```

**在 `convertToFirestoreFields` 函數中**：
- 需要檢測 GeoPoint 格式（包含 `_latitude` 和 `_longitude`）
- 轉換為 `geoPointValue` 格式

---

## 🔧 修復方案

### 修改 1：更新 `syncBookingToFirestore` 函數

**檔案**：`supabase/functions/sync-to-firestore/index.ts`

**修改前**：
```typescript
const firestoreData = {
  ...
  pickupLocation: bookingData.pickupLocation || { latitude: 0, longitude: 0 },
  dropoffLocation: { latitude: 0, longitude: 0 },
  ...
}
```

**修改後**：
```typescript
// 處理 GeoPoint（從 Supabase 的 location 格式轉換）
const pickupLocation = bookingData.pickupLocation || { latitude: 25.0330, longitude: 121.5654 }  // 預設台北
const dropoffLocation = { latitude: 25.0330, longitude: 121.5654 }  // 預設台北

const firestoreData = {
  ...
  pickupLocation: {
    _latitude: pickupLocation.latitude,
    _longitude: pickupLocation.longitude,
  },
  dropoffLocation: {
    _latitude: dropoffLocation.latitude,
    _longitude: dropoffLocation.longitude,
  },
  ...
}
```

**關鍵變更**：
- ✅ 使用 `_latitude` 和 `_longitude` 作為欄位名稱
- ✅ 這是 Firestore GeoPoint 的內部格式
- ✅ `convertToFirestoreFields` 函數會識別並轉換為 `geoPointValue`

---

### 修改 2：更新 `convertToFirestoreFields` 函數

**修改前**：
```typescript
function convertToFirestoreFields(data: any): any {
  const fields: any = {}

  for (const [key, value] of Object.entries(data)) {
    if (value === null || value === undefined) {
      fields[key] = { nullValue: null }
    } else if (typeof value === 'string') {
      fields[key] = { stringValue: value }
    } else if (typeof value === 'number') {
      fields[key] = { doubleValue: value }
    } else if (typeof value === 'boolean') {
      fields[key] = { booleanValue: value }
    } else if (typeof value === 'object') {
      // 處理嵌套對象（例如 location）
      fields[key] = { mapValue: { fields: convertToFirestoreFields(value) } }  // ❌ 所有對象都當作 Map
    }
  }

  return fields
}
```

**修改後**：
```typescript
function convertToFirestoreFields(data: any): any {
  const fields: any = {}

  for (const [key, value] of Object.entries(data)) {
    if (value === null || value === undefined) {
      fields[key] = { nullValue: null }
    } else if (typeof value === 'string') {
      fields[key] = { stringValue: value }
    } else if (typeof value === 'number') {
      fields[key] = { doubleValue: value }
    } else if (typeof value === 'boolean') {
      fields[key] = { booleanValue: value }
    } else if (typeof value === 'object') {
      // ✅ 檢查是否是 GeoPoint 格式（包含 _latitude 和 _longitude）
      if ('_latitude' in value && '_longitude' in value) {
        fields[key] = {
          geoPointValue: {
            latitude: value._latitude,
            longitude: value._longitude,
          }
        }
      } else {
        // 處理其他嵌套對象
        fields[key] = { mapValue: { fields: convertToFirestoreFields(value) } }
      }
    }
  }

  return fields
}
```

**關鍵變更**：
- ✅ 檢測包含 `_latitude` 和 `_longitude` 的對象
- ✅ 轉換為 Firestore REST API 的 `geoPointValue` 格式
- ✅ 其他對象仍然使用 `mapValue` 格式

## 💡 開發心得和經驗總結

### 1. Firestore 資料類型的重要性

**教訓**：
- Firestore 有特定的資料類型（string, number, boolean, geopoint, timestamp, map, array）
- 不能隨意使用普通的 JavaScript 對象
- 必須使用 Firestore REST API 的正確格式

**GeoPoint 的正確格式**：
```json
{
  "geoPointValue": {
    "latitude": 25.0330,
    "longitude": 121.5654
  }
}
```

---

### 2. 錯誤訊息的價值

**有效的診斷**：
- 錯誤訊息 `type '_Map<String, dynamic>' is not a subtype of type 'GeoPoint'` 非常明確
- 直接指出了問題：收到的是 Map，期望的是 GeoPoint
- 節省了大量診斷時間

**改進**：
- 在 Edge Function 中添加更多日誌
- 記錄寫入 Firestore 的資料格式
- 方便診斷和驗證

---

### 3. 遇到的困難

**困難 1：不熟悉 Firestore REST API 格式**
- 最初以為 `{ latitude: 0, longitude: 0 }` 就是 GeoPoint
- 實際上需要使用 `geoPointValue` 格式
- 解決：查閱 Firestore REST API 文檔

**困難 2：如何在 Edge Function 中表示 GeoPoint**
- 不能直接使用 Firestore SDK（Deno 環境）
- 需要使用 REST API 的格式
- 解決：使用 `_latitude` 和 `_longitude` 作為標記，在 `convertToFirestoreFields` 中轉換

---

### 4. 犯過的錯誤

**錯誤 1：假設普通對象就是 GeoPoint**
- 最初直接寫入 `{ latitude: 0, longitude: 0 }`
- 導致 Firestore 存儲為 Map 類型
- 教訓：必須使用正確的 Firestore 資料類型

**錯誤 2：沒有檢查 Firestore 中的實際資料類型**
- 最初只檢查了欄位是否存在
- 沒有檢查資料類型是否正確
- 教訓：在 Firebase Console 中檢查資料類型
