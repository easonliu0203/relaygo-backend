## 📋 執行摘要

### ✅ 符合 CQRS 原則的部分

1. **讀取操作正確**：
   - ✅ 客戶端從 Firestore `orders_rt` 集合讀取訂單（Read Model）
   - ✅ 使用 Stream 監聽即時更新
   - ✅ Firestore 規則禁止客戶端寫入 `orders_rt` 集合

2. **同步機制正確**：
   - ✅ Supabase Trigger 監聽 `bookings` 表變更
   - ✅ 寫入 `outbox` 表（Outbox Pattern）
   - ✅ Edge Function 消費 `outbox` 事件
   - ✅ Edge Function 雙寫到 Firestore（`orders_rt` + `bookings`）

3. **寫入操作部分正確**：
   - ✅ 新版本使用 Supabase API（`createBookingWithSupabase`）
   - ✅ 新版本使用 Supabase API（`payDepositWithSupabase`）

#### 🚨 問題 1：舊版 Firestore 直接寫入方法仍然存在

**位置**：`mobile/lib/core/services/booking_service.dart`

**違規代碼**：

      .add(order.toFirestore());

</augment_code_snippet>

**問題**：
- ❌ 直接寫入 Firestore `bookings` 集合
- ❌ 繞過 Supabase（單一真實源）
- ❌ 違反 CQRS 原則

**影響**：
- 如果客戶端調用此方法，資料只會寫入 Firestore
- Supabase 不會有記錄
- 無法觸發 Outbox Pattern 同步
- 造成資料不一致

#### 🚨 問題 4：Firestore 規則允許客戶端寫入 `bookings` 集合

**位置**：`firebase/firestore.rules`

**違規規則**：

<augment_code_snippet path="firebase/firestore.rules" mode="EXCERPT">
````javascript
// 訂單規則（舊版 bookings 集合）
match /bookings/{bookingId} {
  allow read: if request.auth != null &&
    (request.auth.uid == resource.data.customerId ||
     request.auth.uid == resource.data.driverId);
  allow create: if request.auth != null &&
    request.auth.uid == request.resource.data.customerId;  // ❌ 允許創建
  allow update: if request.auth != null &&
    (request.auth.uid == resource.data.customerId ||
     request.auth.uid == resource.data.driverId);  // ❌ 允許更新
}
````
</augment_code_snippet>

**問題**：
- ❌ 允許客戶端創建訂單（`allow create`）
- ❌ 允許客戶端更新訂單（`allow update`）
- ❌ 違反「Firestore 只作為 Read Model」的原則

**正確的規則**（參考 `orders_rt`）：

<augment_code_snippet path="firebase/firestore.rules" mode="EXCERPT">
````javascript
// 訂單即時鏡像規則（orders_rt 集合）
match /orders_rt/{orderId} {
  // 允許用戶讀取自己的訂單
  allow read: if request.auth != null
              && resource.data.customerId == request.auth.uid;

  // 禁止客戶端寫入（由 Supabase 寫入）
  allow write: if false;  // ✅ 正確
}
````
</augment_code_snippet>

## 🎯 當前架構狀態

### 實際資料流（混合模式）⚠️

```
客戶端 App
    ├─→ (新版) POST /api/bookings → Supabase ✅ 正確
    │       ↓
    │   Supabase/PostgreSQL (單一真實源)
    │       ↓ (Trigger)
    │   Outbox 表
    │       ↓ (Cron Job)
    │   Edge Function
    │       ↓ (雙寫)
    │   Firestore (orders_rt + bookings)
    │       ↑ (讀取)
    │   客戶端 App ✅ 正確
    │
    └─→ (舊版) 直接寫入 Firestore ❌ 錯誤
            ↓
        Firestore bookings 集合
            ↑ (讀取)
        客戶端 App
```

**問題**：
- 存在兩條寫入路徑（新版 Supabase API + 舊版 Firestore 直接寫入）
- 如果客戶端調用舊版方法，會造成資料不一致
- Firestore 規則允許客戶端寫入，無法強制使用 Supabase API

---

### 預期的正確架構 ✅

```
客戶端 App
    ↓ (寫入) POST /api/bookings
Supabase/PostgreSQL ← 單一真實源（Single Source of Truth）
    ↓ (Trigger)
Outbox 表
    ↓ (Cron Job 每 30 秒)
Edge Function (sync-to-firestore)
    ↓ (雙寫)
Firestore (orders_rt + bookings) ← 讀模型（Read Model）
    ↑ (讀取)
客戶端 App
```

**特點**：
- ✅ 只有一條寫入路徑（Supabase API）
- ✅ Firestore 規則禁止客戶端寫入
- ✅ 強制所有寫入操作通過 Supabase

## 🛠️ 修復建議

### 優先級 1：刪除舊版 Firestore 直接寫入方法（高風險）

**需要刪除的方法**：
1. `BookingService.createBooking()` - 直接寫入 Firestore
2. `BookingService.payDeposit()` - 直接更新 Firestore
3. `BookingService.cancelBooking()` - 直接更新 Firestore

**原因**：
- 這些方法違反 CQRS 原則
- 可能被誤用導致資料不一致
- 應該強制使用 Supabase API

**替代方案**：
- 保留方法簽名，但內部調用 Supabase API
- 或直接刪除，強制使用新版方法

---

### 優先級 2：修改 Firestore 規則禁止寫入 `bookings` 集合（高風險）

**當前規則**（錯誤）：
```javascript
match /bookings/{bookingId} {
  allow read: if request.auth != null &&
    (request.auth.uid == resource.data.customerId ||
     request.auth.uid == resource.data.driverId);
  allow create: if request.auth != null &&
    request.auth.uid == request.resource.data.customerId;  // ❌
  allow update: if request.auth != null &&
    (request.auth.uid == resource.data.customerId ||
     request.auth.uid == resource.data.driverId);  // ❌
}
```

**修復後規則**（正確）：
```javascript
match /bookings/{bookingId} {
  // 允許用戶讀取自己的訂單
  allow read: if request.auth != null &&
    (request.auth.uid == resource.data.customerId ||
     request.auth.uid == resource.data.driverId);

  // 禁止客戶端寫入（由 Supabase Edge Function 寫入）
  allow write: if false;  // ✅ 強制使用 Supabase API
}
```
