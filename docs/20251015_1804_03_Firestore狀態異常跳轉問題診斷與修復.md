## 🔍 問題診斷過程

### 診斷假設

基於問題描述，我提出以下假設：

1. **假設 1**: Edge Function 處理了多個 outbox 事件，後面的事件是舊狀態
2. **假設 2**: Outbox 表中有重複的事件
3. **假設 3**: Edge Function 未部署最新代碼
4. **假設 4**: Trigger 被觸發了多次，寫入了多個 outbox 事件

### 診斷步驟

#### 步驟 1: 檢查 Edge Function 事件處理邏輯

**檢查文件**: `supabase/functions/sync-to-firestore/index.ts`

**發現的問題**:
```typescript
// 第 169-175 行
const { data: events } = await supabase
  .from('outbox')
  .select('*')
  .is('processed_at', null)
  .lt('retry_count', 3)
  .order('created_at', { ascending: true })  // ⚠️  按創建時間升序
  .limit(10)
```

**問題分析**:
- Edge Function 按創建時間**升序**處理事件
- 如果有多個未處理事件，會先處理舊事件，再處理新事件
- 但由於使用 `Promise.allSettled` 並行處理（第 191-193 行），可能導致舊事件後寫入 Firestore
- 結果：新狀態被舊狀態覆蓋

#### 步驟 2: 分析可能的事件順序

**正常情況**（只有一個事件）:
```
1. 客戶點擊「開始行程」
2. Backend 更新 Supabase: status = 'trip_started'
3. Trigger 寫入 outbox: event_1 (status = 'trip_started')
4. Edge Function 處理 event_1
5. Firestore 狀態: 'inProgress' ✅
```

**異常情況**（有多個事件）:
```
1. 客戶點擊「開始行程」
2. Backend 更新 Supabase: status = 'trip_started'
3. Trigger 寫入 outbox: event_1 (status = 'trip_started')
4. 【某種原因】又寫入 outbox: event_2 (status = 'driver_confirmed')
5. Edge Function 同時處理 event_1 和 event_2
6. event_1 寫入 Firestore: 'inProgress' ✅
7. event_2 寫入 Firestore: 'matched' ❌（覆蓋了 event_1）
8. 最終 Firestore 狀態: 'matched' 或 'pending' ❌
```

#### 步驟 3: 確定根本原因

**最可能的原因**:
1. **Outbox 表中有多個未處理事件**
   - 可能是 Trigger 被觸發多次
   - 可能是有重複的 Trigger
   - 可能是訂單被多次更新

2. **Edge Function 並行處理導致競態條件**
   - 多個事件同時寫入 Firestore
   - 舊事件可能後寫入，覆蓋新狀態

## 💡 解決方案設計

### 短期解決方案（緊急修復）

#### 方案 1: 清理舊的未處理事件

**目的**: 立即解決當前問題

**實施步驟**:
1. 查詢所有有多個未處理事件的訂單
2. 只保留每個訂單的最新事件
3. 將舊事件標記為已處理
4. 手動觸發 Edge Function 同步

**SQL 腳本**: `supabase/緊急修復-清理舊事件並恢復正確狀態.sql`

#### 方案 2: 檢查並修復重複 Trigger

**目的**: 防止未來產生重複事件

**實施步驟**:
1. 檢查 bookings 表上的所有 Trigger
2. 確認只有一個 `bookings_outbox_trigger`
3. 如果有重複，刪除並重新創建

**SQL 腳本**: `supabase/檢查並修復重複Trigger.sql`

### 長期解決方案（根本修復）

#### 方案 1: 創建 RPC 函數，只查詢最新事件

**目的**: 確保 Edge Function 只處理每個訂單的最新事件

**實施內容**:
```sql
-- 創建 RPC 函數
CREATE OR REPLACE FUNCTION get_latest_unprocessed_events(event_limit INT DEFAULT 10)
RETURNS SETOF outbox AS $$
BEGIN
  RETURN QUERY
  SELECT DISTINCT ON (aggregate_id) *
  FROM outbox
  WHERE processed_at IS NULL
    AND retry_count < 3
    AND aggregate_type = 'booking'
  ORDER BY aggregate_id, created_at DESC  -- 每個訂單取最新事件
  LIMIT event_limit;
END;
$$ LANGUAGE plpgsql;
```

**SQL 文件**: `supabase/migrations/20251015_create_get_latest_unprocessed_events.sql`

#### 方案 2: 修改 Edge Function 使用新的 RPC 函數

**目的**: 應用新的查詢邏輯

**修改內容**:
```typescript
// 修改前
const { data: events } = await supabase
  .from('outbox')
  .select('*')
  .is('processed_at', null)
  .lt('retry_count', 3)
  .order('created_at', { ascending: true })
  .limit(10)

// 修改後
const { data: events } = await supabase
  .rpc('get_latest_unprocessed_events', { event_limit: 10 })
```

#### 方案 3: 添加 Firestore 時間戳檢查（可選但推薦）

**目的**: 在 Firestore 層面防止狀態倒退

**實施內容**:
- 在寫入 Firestore 前，檢查事件的創建時間
- 如果新事件的時間早於現有文檔，跳過更新
- 確保只有更新的事件才能覆蓋舊狀態

WITH latest_events AS (
  SELECT DISTINCT ON (aggregate_id) 
    id, aggregate_id, created_at
  FROM outbox
  WHERE processed_at IS NULL
    AND aggregate_type = 'booking'
  ORDER BY aggregate_id, created_at DESC
)
UPDATE outbox o
SET 
  processed_at = NOW(),
  error_message = '自動清理：舊事件'
FROM latest_events le
WHERE o.aggregate_id = le.aggregate_id
  AND o.id != le.id
  AND o.processed_at IS NULL
  AND o.aggregate_type = 'booking';

### 遇到的困難

1. **無法直接查看用戶數據**
   - 問題：無法直接查看用戶的 Supabase 和 Firestore 數據
   - 解決：創建診斷腳本，讓用戶自行執行
   - 改進：建議添加監控和告警系統

2. **並行處理的競態條件**
   - 問題：`Promise.allSettled` 並行處理可能導致順序混亂
   - 解決：使用 RPC 函數只查詢最新事件
   - 改進：考慮使用串行處理或添加鎖機制

3. **狀態倒退難以發現**
   - 問題：狀態倒退可能在幾秒或幾分鐘後發生
   - 解決：添加詳細的日誌輸出
   - 改進：添加狀態變更監控和告警

### 犯過的錯誤與教訓

1. **假設事件處理順序正確**
   - 錯誤：假設按創建時間升序處理就能保證順序
   - 教訓：並行處理會打亂順序，需要額外的保護機制
   - 改進：只處理最新事件，或添加時間戳檢查

2. **未考慮重複事件的情況**
   - 錯誤：假設每個訂單只有一個未處理事件
   - 教訓：實際可能有多個事件（Trigger 重複、訂單多次更新等）
   - 改進：使用 DISTINCT ON 只選擇最新事件

3. **日誌輸出不足**
   - 錯誤：Edge Function 沒有輸出事件處理的詳細信息
   - 教訓：關鍵邏輯必須有詳細的日誌
   - 改進：添加事件 ID、創建時間、處理順序等日誌
