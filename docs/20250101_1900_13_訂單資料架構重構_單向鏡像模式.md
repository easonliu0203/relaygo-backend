# 訂單資料架構重構：單向鏡像模式

**日期**: 2025-01-01 19:00  
**編號**: 13  
**主旨**: 從雙向同步改為單向鏡像模式，實現 Supabase 作為唯一真實來源  
**開發者**: AI Assistant  

## 📋 重構概述

### 重構目標

將訂單資料架構從「雙向同步」模式重構為「單向鏡像」模式，實現：
- ✅ Supabase 作為唯一的資料寫入來源（Single Source of Truth）
- ✅ Firestore 只作為即時資料的鏡像（Read-Only）
- ✅ 解耦資料寫入和即時展示的關注點
- ✅ 提高資料一致性和系統可維護性

### 重構前的問題

**雙向同步模式的缺陷**：

1. **資料一致性風險**：
   - 客戶端同時寫入 Supabase 和 Firebase
   - 如果其中一個寫入失敗，資料不一致
   - 難以確定哪個資料庫是真實來源

2. **耦合度高**：
   - 客戶端需要同時處理兩個資料庫
   - 業務邏輯和資料同步邏輯混在一起
   - 難以維護和測試

3. **錯誤處理複雜**：
   - 需要處理兩個資料庫的錯誤
   - 同步失敗時的回滾邏輯複雜
   - 重試機制難以實作

4. **違反架構原則**：
   - Firebase 應該只負責即時功能
   - Supabase 應該是業務資料的唯一來源
   - 混合使用導致職責不清

## 🎯 新架構設計

### 架構原則

**Single Source of Truth**：
- Supabase 是訂單資料的唯一真實來源
- 所有寫入操作只發生在 Supabase
- Firebase 只作為即時資料的鏡像

**單向資料流**：
```
App → Supabase API → orders 表（唯一寫入）
                        ↓
                   Trigger 監聽
                        ↓
                   outbox 表（事件佇列）
                        ↓
                Edge Function 消費
                        ↓
              Firestore orders_rt 集合（鏡像）
                        ↓
                   App 即時讀取
```

**關注點分離**：
- **Supabase**：負責訂單金額、結帳、退款、獎金、報表
- **Firebase**：負責登入、推播、聊天即時、檔案、定位、即時訂單狀態

### Outbox Pattern

使用 Outbox Pattern 實現可靠的資料鏡像：

**1. Trigger 監聽變更**：
```sql
CREATE TRIGGER orders_outbox_trigger
AFTER INSERT OR UPDATE ON orders
FOR EACH ROW
EXECUTE FUNCTION orders_to_outbox();
```

**2. 寫入事件佇列**：
```sql
INSERT INTO outbox (
  aggregate_type,
  aggregate_id,
  event_type,
  payload
) VALUES (
  'order',
  NEW.id,
  'created' | 'updated' | 'deleted',
  jsonb_build_object(...)
);
```

**3. Edge Function 消費事件**：
- 每 30 秒執行一次
- 批次處理未處理事件（每次最多 10 個）
- 推送到 Firestore `orders_rt` 集合
- 標記事件為已處理

**4. 重試機制**：
- 失敗事件自動重試（最多 3 次）
- 記錄錯誤訊息便於診斷
- 超過重試次數的事件需要手動處理

**5. 自動清理**：
- 每天凌晨 2 點清理舊事件
- 保留最近 7 天的已處理事件

### 資料鏡像策略

**只鏡像前端需要即時展示的欄位**：

```typescript
{
  id: string,              // 訂單 ID
  customerId: string,      // 客戶 ID（Firebase UID）
  status: string,          // 訂單狀態
  pickupAddress: string,   // 上車地點
  dropoffAddress: string,  // 下車地點
  pickupLocation: {        // 上車座標
    latitude: number,
    longitude: number
  },
  dropoffLocation: {       // 下車座標
    latitude: number,
    longitude: number
  },
  bookingTime: string,     // 預約時間
  passengerCount: number,  // 乘客人數
  luggageCount: number,    // 行李數量
  notes: string,           // 備註
  estimatedFare: number,   // 預估費用
  depositAmount: number,   // 訂金金額
  depositPaid: boolean,    // 訂金是否已支付
  createdAt: string,       // 建立時間
  driverId: string | null, // 司機 ID（如果已配對）
  matchedAt: string | null,   // 配對時間
  startedAt: string | null,   // 開始時間
  completedAt: string | null  // 完成時間
}
```

**不鏡像的欄位**（只存在於 Supabase）：
- 支付交易詳情
- 退款記錄
- 獎金計算
- 詳細的財務資料
- 審計日誌

### 讀取策略

**場景 1：需要最準確、完整的資料**
- 產品/營運頁面（公司端 web-admin）
- 報表、分析、對帳
- **資料來源**：直接讀取 Supabase API

**場景 2：需要即時更新的畫面**
- App 即時畫面（配對中、司機已到、倒數計時）
- 訂單狀態追蹤
- **資料來源**：讀取 Firestore 鏡像 `orders_rt/{orderId}`

### 3. 創建 Outbox 表和 Trigger

**文件**: `supabase/migrations/20250101_create_outbox_table.sql`

**outbox 表結構**：
```sql
CREATE TABLE outbox (
  id UUID PRIMARY KEY,
  aggregate_type VARCHAR(50),  -- 'order'
  aggregate_id VARCHAR(255),   -- 訂單 ID
  event_type VARCHAR(50),      -- 'created', 'updated', 'deleted'
  payload JSONB,               -- 事件資料
  created_at TIMESTAMP,
  processed_at TIMESTAMP,      -- NULL = 未處理
  retry_count INTEGER,
  error_message TEXT
);
```

**Trigger 函數**：
```sql
CREATE FUNCTION orders_to_outbox()
RETURNS TRIGGER AS $$
BEGIN
  INSERT INTO outbox (...)
  VALUES ('order', NEW.id, 'created', jsonb_build_object(...));
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

### 4. 創建 Edge Function

**文件**: `supabase/functions/sync-to-firestore/index.ts`

**主要功能**：
1. 查詢未處理的事件（批次處理）

3. 標記事件為已處理
4. 錯誤處理和重試

**核心邏輯**：
```typescript
// 1. 查詢未處理事件
const { data: events } = await supabase
  .from('outbox')
  .select('*')
  .is('processed_at', null)
  .lt('retry_count', 3)
  .limit(10)

// 2. 處理每個事件
for (const event of events) {
  await syncOrderToFirestore(event)
  await markAsProcessed(event.id)
}

// 3. 推送到 Firestore
await fetch(`https://firestore.googleapis.com/.../orders_rt/${orderId}`, {
  method: 'PATCH',
  body: JSON.stringify({ fields: firestoreFields })
})
```

## 💡 架構設計決策

### 決策 1：為什麼選擇 Outbox Pattern？

**原因**：
1. **可靠性**：事件不會丟失，即使 Firestore 暫時不可用
2. **解耦**：業務邏輯和資料同步完全分離
3. **可追蹤**：所有事件都有記錄，便於審計和調試
4. **可重試**：失敗事件自動重試，提高成功率
5. **可擴展**：未來可以輕鬆添加其他消費者（例如：發送通知、更新快取）

**替代方案**：
- ❌ **直接同步**：失敗時難以處理，耦合度高
- ❌ **Message Queue**：增加系統複雜度，需要額外的基礎設施
- ✅ **Outbox Pattern**：簡單、可靠、易於維護

### 決策 2：為什麼每 30 秒同步一次？

**原因**：
1. **即時性**：30 秒的延遲對大多數場景可接受
2. **性能**：避免過於頻繁的查詢和寫入
3. **成本**：減少 Edge Function 調用次數
4. **可調整**：可以根據實際需求調整頻率

**替代方案**：
- ❌ **即時同步**：成本高，性能壓力大
- ❌ **每分鐘同步**：延遲過高，用戶體驗差
- ✅ **每 30 秒同步**：平衡即時性和成本

### 決策 3：為什麼只鏡像部分欄位？

**原因**：
1. **性能**：減少資料傳輸量
2. **安全**：敏感財務資料不暴露在 Firestore
3. **關注點分離**：Firestore 只負責即時展示
4. **成本**：減少 Firestore 儲存成本

**鏡像的欄位**：
- 訂單基本資訊（ID、地點、時間）
- 訂單狀態（pending, matched, in_progress）
- 司機資訊（如果已配對）
- 即時更新欄位（matchedAt, startedAt）

**不鏡像的欄位**：
- 支付交易詳情
- 退款記錄
- 獎金計算
- 詳細的財務資料

### 決策 4：為什麼使用 Firestore REST API？

**原因**：
1. **簡單**：不需要 Firebase Admin SDK
2. **輕量**：減少 Edge Function 的依賴
3. **可控**：直接控制 HTTP 請求
4. **跨平台**：Deno 環境中易於使用

**替代方案**：
- ❌ **Firebase Admin SDK**：Deno 支援不完整
- ❌ **gRPC API**：過於複雜
- ✅ **REST API**：簡單、可靠、易於使用

## 🚧 實作過程中的挑戰

### 挑戰 1：Firestore 欄位格式轉換

**問題**：
- Firestore REST API 需要特定的欄位格式
- 需要將 JSON 轉換為 Firestore 格式

**解決方法**：
```typescript
function convertToFirestoreFields(data: any): any {
  const fields: any = {}
  for (const [key, value] of Object.entries(data)) {
    if (typeof value === 'string') {
      fields[key] = { stringValue: value }
    } else if (typeof value === 'number') {
      fields[key] = { doubleValue: value }
    }
    // ... 其他類型
  }
  return fields
}
```

### 挑戰 2：處理嵌套對象

**問題**：
- Location 對象需要特殊處理
- Firestore 使用 GeoPoint 類型

**解決方法**：
```typescript
pickupLocation: {
  mapValue: {
    fields: {
      latitude: { doubleValue: data.pickupLocation.latitude },
      longitude: { doubleValue: data.pickupLocation.longitude }
    }
  }
}
```

### 挑戰 3：獲取 Firebase UID

**問題**：
- Supabase 儲存的是內部 user ID
- Firestore 需要 Firebase UID

**解決方法**：
```sql
SELECT firebase_uid FROM users WHERE id = NEW.customer_id
```

在 trigger 中查詢 users 表獲取 Firebase UID

```typescript
    }
  )
  .subscribe()
```
