## 📋 問題描述

用戶回報包車服務管理系統中的兩個關鍵頁面出現載入問題：

### 問題現象
1. **封測訂單測試頁面** (http://localhost:3001/orders/create-test) 點擊進入後顯示空白頁面
2. **價格設定頁面** (http://localhost:3001/settings/pricing) 點擊進入後顯示空白頁面
3. 在空白頁面點擊瀏覽器的「上一頁」按鈕無法正常返回，仍然顯示空白頁面

### 技術環境
- Next.js 14 管理後台運行在 http://localhost:3001
- 使用 Ant Design 5 UI 組件庫
- TypeScript 開發環境

## 📚 開發心得和經驗總結

### 1. 環境變數管理經驗

#### 問題根源
- **客戶端環境**: Next.js 在客戶端環境中，`process.env` 可能不包含所有環境變數
- **SSR/CSR 不一致**: 服務端和客戶端環境變數載入時機不同
- **構建時優化**: Next.js 在構建時會優化掉未使用的環境變數

#### 最佳實踐
- **提供預設值**: 所有環境變數都應該有合理的預設值
- **錯誤處理**: 使用 try-catch 包裝所有外部服務初始化
- **降級策略**: 為每個外部依賴提供降級方案

### 2. 服務層設計經驗

#### 依賴注入模式
- **避免直接依賴**: 不要在服務層直接導入外部客戶端
- **介面抽象**: 使用 API 服務層統一管理外部調用
- **降級機制**: 每個服務調用都應該有降級方案

#### 錯誤處理策略
- **分層錯誤處理**: 在不同層級實作不同的錯誤處理策略
- **用戶友好**: 對用戶隱藏技術細節，提供有意義的錯誤訊息
- **開發友好**: 在開發環境中提供詳細的錯誤資訊

### 3. 前端架構經驗

#### 組件設計原則
- **容錯性**: 組件應該能夠處理資料載入失敗的情況
- **載入狀態**: 提供清晰的載入和錯誤狀態提示
- **降級顯示**: 當資料不可用時，提供合理的預設顯示

#### 狀態管理
- **初始狀態**: 為所有狀態提供合理的初始值
- **錯誤狀態**: 明確區分載入中、成功、失敗狀態
- **恢復機制**: 提供重試和恢復功能

## 🚨 遇到的困難和解決方法

### 1. 環境變數載入問題

**困難**: Next.js 客戶端環境中環境變數不可用
**解決**: 
- 提供硬編碼的預設值作為後備
- 實作運行時檢查和警告機制
- 使用 try-catch 包裝所有初始化代碼

**教訓**: 永遠不要假設環境變數在所有環境中都可用

### 2. Supabase 客戶端初始化失敗

**困難**: 當 API 金鑰無效時，Supabase 客戶端建立失敗導致整個應用崩潰
**解決**:
- 使用 try-catch 包裝客戶端建立過程
- 建立模擬客戶端作為降級方案
- 實作統一的錯誤處理機制

**教訓**: 所有外部服務初始化都應該有錯誤處理和降級方案

### 3. 服務層依賴管理

**困難**: 直接導入 Supabase 客戶端導致循環依賴和初始化問題
**解決**:
- 移除直接的 Supabase 依賴
- 統一使用 API 服務層
- 實作服務層降級機制

**教訓**: 避免在服務層直接依賴外部客戶端，應該通過抽象層管理

## 🔄 犯過的錯誤和教訓

### 1. 過度依賴環境變數

**錯誤**: 假設環境變數在所有環境中都可用
**後果**: 客戶端環境中應用無法啟動
**教訓**: 
- 為所有環境變數提供預設值
- 實作運行時檢查機制
- 不要使用 TypeScript 的 `!` 斷言符號

### 2. 缺乏錯誤邊界

**錯誤**: 沒有在組件層級實作錯誤邊界
**後果**: 單個組件錯誤導致整個頁面崩潰
**教訓**:
- 在關鍵組件周圍實作錯誤邊界
- 提供有意義的錯誤回饋
- 實作錯誤恢復機制

### 3. 忽略降級策略

**錯誤**: 沒有為外部服務提供降級方案
**後果**: 外部服務不可用時應用完全無法使用
**教訓**:
- 為每個外部依賴設計降級方案
- 實作模擬資料和模擬服務
- 確保核心功能在離線狀態下仍可使用
