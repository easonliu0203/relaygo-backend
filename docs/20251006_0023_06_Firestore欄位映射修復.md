## 📋 問題描述

### 症狀
1. ✅ 手動觸發 Edge Function 返回 `success: 7, failure: 0`
2. ✅ Supabase `bookings` 表有新訂單

**查看代碼**：`supabase/functions/sync-to-firestore/index.ts`（修復前）

**Edge Function 寫入的欄位**：

- 缺少 `dropoffLocation`、`passengerCount`、`luggageCount`、`estimatedFare`、`depositPaid`

---

### 4. 根本原因

**欄位映射不匹配**：
- Edge Function 使用的欄位名稱與客戶端期望的不一致
- Edge Function 缺少客戶端必需的欄位
- 導致客戶端無法正確解析 Firestore 文檔

**為什麼會發生**：
- Edge Function 的欄位映射是基於 Supabase Trigger 的 payload 格式
- 但客戶端 App 的欄位格式是基於原始的 Firestore 設計
- 兩者沒有對齊

---

## 🔧 修復方案

### 修改 Edge Function 的欄位映射

**檔案**：`supabase/functions/sync-to-firestore/index.ts`

**修改內容**：

```typescript
async function syncBookingToFirestore(event: OutboxEvent): Promise<void> {
  const bookingId = event.aggregate_id
  const bookingData = event.payload

  console.log(`同步訂單到 Firestore: ${bookingId}`, bookingData)

  // 組合 bookingTime（從 startDate 和 startTime）
  let bookingTime: string
  if (bookingData.startDate && bookingData.startTime) {
    bookingTime = `${bookingData.startDate}T${bookingData.startTime}`
  } else {
    bookingTime = bookingData.createdAt
  }

  // 轉換資料格式為客戶端 App 期望的格式
  const firestoreData = {
    // 基本資訊
    customerId: bookingData.customerId,
    driverId: bookingData.driverId || null,
    
    // 地點資訊
    pickupAddress: bookingData.pickupAddress || '',
    pickupLocation: bookingData.pickupLocation || { latitude: 0, longitude: 0 },
    dropoffAddress: bookingData.destination || '',  // ✅ destination -> dropoffAddress
    dropoffLocation: { latitude: 0, longitude: 0 },  // ✅ 添加預設值
    
    // 時間資訊
    bookingTime: bookingTime,  // ✅ 組合 startDate + startTime
    
    // 乘客資訊
    passengerCount: 1,  // ✅ 添加預設值
    luggageCount: null,  // ✅ 添加
    notes: bookingData.specialRequirements || null,  // ✅ 映射
    
    // 費用資訊
    estimatedFare: bookingData.totalAmount || 0,  // ✅ 映射
    depositAmount: bookingData.depositAmount || 0,
    depositPaid: false,  // ✅ 添加預設值
    
    // 狀態
    status: bookingData.status || 'pending',
    
    // 時間戳記
    createdAt: bookingData.createdAt,
    matchedAt: bookingData.actualStartTime || null,
    completedAt: bookingData.actualEndTime || null,
  }

  console.log(`轉換後的 Firestore 資料:`, firestoreData)

  // 根據事件類型執行不同操作
  if (event.event_type === 'deleted') {
    await deleteFirestoreDocument(bookingId)
  } else {
    await upsertFirestoreDocument(bookingId, firestoreData)
  }
}
```

### 關鍵修改

1. **`destination` → `dropoffAddress`**
   - 匹配客戶端期望的欄位名稱

2. **組合 `bookingTime`**
   - 從 `startDate` 和 `startTime` 組合為 ISO 8601 格式
   - 例如：`2025-10-05T09:00:00`

3. **添加缺少的欄位**
   - `dropoffLocation`: 預設值 `{ latitude: 0, longitude: 0 }`
   - `passengerCount`: 預設值 `1`
   - `luggageCount`: `null`
   - `estimatedFare`: 映射自 `totalAmount`
   - `depositPaid`: 預設值 `false`

4. **映射欄位**
   - `notes` ← `specialRequirements`
   - `estimatedFare` ← `totalAmount`

## 💡 開發心得和經驗總結

### 1. 欄位映射的重要性

**教訓**：
- 在設計資料同步時，必須確保兩端的欄位格式一致
- 不能假設 Supabase 的欄位名稱與 Firestore 的一致
- 需要明確的欄位映射文檔

**改進**：
- 創建欄位映射對照表
- 在代碼中添加詳細的註解
- 使用 TypeScript 類型定義確保一致性

---

### 2. 診斷流程

**有效的診斷步驟**：
1. 確認資料來源（客戶端從哪裡讀取）
2. 檢查客戶端期望的欄位格式
3. 檢查 Edge Function 寫入的欄位格式
4. 對比差異，找出不匹配的欄位

**工具**：
- 代碼搜尋（codebase-retrieval）
- Firebase Console（查看實際資料）
- Edge Function 日誌（查看寫入的資料）

---

### 3. 遇到的困難

**困難 1：欄位名稱不一致**
- Supabase 使用 `destination`
- 客戶端期望 `dropoffAddress`
- 解決：在 Edge Function 中進行映射

**困難 2：時間格式不一致**
- Supabase 分開存儲 `startDate` 和 `startTime`
- 客戶端期望單一的 `bookingTime`
- 解決：在 Edge Function 中組合為 ISO 8601 格式

**困難 3：缺少必需欄位**
- Edge Function 沒有寫入 `passengerCount`、`estimatedFare` 等
- 客戶端解析時會失敗
- 解決：添加預設值或從其他欄位映射

---

### 4. 犯過的錯誤

**錯誤 1：假設欄位名稱一致**
- 最初以為 Supabase 和 Firestore 的欄位名稱相同
- 導致客戶端無法解析資料
- 教訓：永遠不要假設，要明確驗證

**錯誤 2：沒有檢查客戶端代碼**
- 最初只檢查了 Edge Function 和 Firestore
- 沒有檢查客戶端如何解析資料
- 教訓：要從客戶端的角度思考問題

**錯誤 3：沒有添加日誌**
- 最初沒有在 Edge Function 中添加 `console.log`
- 導致難以診斷問題
- 教訓：添加詳細的日誌，特別是資料轉換的部分
