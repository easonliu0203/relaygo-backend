## 📋 任務概述

用戶在 Android 模擬器上測試完整支付流程時，發現支付成功後訂單無法正常顯示。

### 核心問題
1. **預約成功頁面顯示異常**：頁面顯示「訂單不存在」
2. **我的訂單頁面問題**：剛創建並支付的訂單沒有顯示
3. **資料不一致**：訂單創建成功但無法查詢

### 3. 根本原因確認

**資料流分析**：

```
創建訂單
    ↓
調用 Supabase API
    ↓
POST /api/bookings
    ↓
後端返回模擬訂單資料
{
  id: "mock_booking_1234567890",
  status: "pending",
  totalAmount: 50.0,
  depositAmount: 15.0
}
    ↓
前端收到訂單 ID
    ↓
支付訂金
    ↓
調用 Supabase API
    ↓
POST /api/bookings/:id/pay-deposit
    ↓
支付成功
    ↓
跳轉到「預約成功」頁面
    ↓
查詢訂單
    ↓
從 Firebase Firestore 查詢
    ↓
❌ 找不到訂單（因為只存在於 Supabase 模擬資料中）
    ↓
顯示「訂單不存在」
```

**根本原因**：

🎯 **雙資料庫架構衝突**

1. **訂單創建**：使用 `createBookingWithSupabase()` → 調用 Supabase API → 只存在於後端模擬資料
2. **訂單查詢**：使用 `watchBooking()` → 從 Firebase Firestore 查詢 → 找不到資料
3. **資料不同步**：訂單只存在於 Supabase，Firebase 中沒有

### 4. 問題確認

**測試驗證**：

1. 創建訂單後，檢查 Firebase Console
   - ❌ `bookings` 集合中沒有新訂單

2. 查看控制台日誌
   - ✅ 創建訂單成功：`API 返回訂單 ID: mock_booking_xxx`
   - ❌ 沒有同步到 Firebase 的日誌

3. 查看「預約成功」頁面
   - ❌ 顯示「訂單不存在」

**結論**：
- 訂單創建成功（Supabase API）
- 訂單沒有同步到 Firebase
- 查詢時從 Firebase 查詢，找不到資料

## 🛠️ 具體修復方案

### 修復策略

**方案選擇**：添加資料同步機制

**原因**：
1. 保持現有的查詢邏輯不變（從 Firebase 查詢）
2. 利用 Firebase 的實時更新特性
3. 封閉測試階段的快速解決方案
4. 未來可以選擇完全遷移到 Supabase 或保持雙資料庫

### 修復步驟

#### 步驟 1：添加訂單同步方法

在 `booking_service.dart` 中添加 `_syncBookingToFirestore()` 方法：

```dart
/// 同步訂單到 Firebase Firestore
Future<void> _syncBookingToFirestore(
  Map<String, dynamic> bookingData,
  BookingRequest request,
  String userId,
) async {
  try {
    debugPrint('[BookingService] 同步訂單到 Firebase: ${bookingData['id']}');
    
    final order = BookingOrder(
      id: bookingData['id'] ?? '',
      customerId: userId,
      pickupAddress: request.pickupAddress ?? '',
      pickupLocation: request.pickupLocation,
      dropoffAddress: request.dropoffAddress ?? '',
      dropoffLocation: request.dropoffLocation,
      bookingTime: request.bookingTime,
      passengerCount: request.passengerCount,
      luggageCount: request.luggageCount,
      notes: request.notes,
      estimatedFare: (bookingData['totalAmount'] ?? 0.0).toDouble(),
      depositAmount: (bookingData['depositAmount'] ?? 0.0).toDouble(),
      depositPaid: false,
      status: BookingStatus.pending,
      createdAt: DateTime.now(),
    );

    // 使用 API 返回的 ID 作為文檔 ID
    await _firestore
        .collection('bookings')
        .doc(bookingData['id'])
        .set(order.toFirestore());
    
    debugPrint('[BookingService] 訂單已同步到 Firebase');
  } catch (e) {
    debugPrint('[BookingService] 同步到 Firebase 失敗: $e');
    // 不拋出異常，因為主要資料已經在 Supabase 中
  }
}
```

**關鍵點**：
- 使用 API 返回的訂單 ID 作為 Firebase 文檔 ID
- 將 Supabase API 返回的資料轉換為 BookingOrder 對象
- 保存到 Firebase Firestore
- 錯誤不影響主流程

#### 步驟 3：添加支付狀態同步方法

在 `booking_service.dart` 中添加 `_syncPaymentStatusToFirestore()` 方法：

```dart
/// 同步支付狀態到 Firebase
Future<void> _syncPaymentStatusToFirestore(String bookingId) async {
  try {
    debugPrint('[BookingService] 更新 Firebase 訂單狀態: $bookingId');
    
    await _firestore
        .collection('bookings')
        .doc(bookingId)
        .update({
          'depositPaid': true,
          'status': BookingStatus.matched.name, // 支付成功後變為已配對狀態
        });
    
    debugPrint('[BookingService] Firebase 訂單狀態已更新');
  } catch (e) {
    debugPrint('[BookingService] 更新 Firebase 狀態失敗: $e');
    // 不拋出異常，因為主要資料已經在 Supabase 中
  }
}
```

**關鍵點**：
- 更新 `depositPaid` 為 `true`
- 更新訂單狀態為 `matched`（已配對）
- 錯誤不影響主流程

## 💡 開發心得和經驗總結

### 1. 技術收穫

**雙資料庫架構的挑戰**：
- 使用多個資料庫時，必須確保資料同步
- 需要明確定義哪個資料庫是主要資料來源
- 同步失敗不應該影響主流程

**資料流設計**：
- 創建資料時：主資料庫 → 同步到其他資料庫
- 查詢資料時：從最適合的資料庫查詢
- 更新資料時：同步更新所有相關資料庫

**錯誤處理策略**：
- 主要操作失敗應該拋出異常
- 同步操作失敗只記錄日誌
- 不讓次要操作影響主流程

### 2. 問題診斷技巧

**系統性診斷流程**：
1. 收集用戶報告的現象
2. 分析資料流和代碼邏輯
3. 找出資料流斷裂的地方
4. 確認根本原因
5. 設計修復方案

**使用日誌追蹤資料流**：
- 在關鍵步驟添加日誌
- 記錄資料的流向和狀態
- 便於快速定位問題

**驗證假設**：
- 不要假設資料已經同步
- 實際檢查資料庫中的資料
- 使用工具（Firebase Console）驗證

### 3. 架構設計經驗

**封閉測試階段的權衡**：
- 快速實作 vs 完美架構
- 選擇了添加同步機制（快速）
- 未來可以重構為單一資料庫

**資料一致性**：
- 使用相同的 ID 確保資料對應
- 同步時機很重要（創建後、更新後）
- 錯誤處理不能影響主流程

**可維護性**：
- 將同步邏輯封裝在獨立方法中
- 添加詳細的日誌記錄
- 代碼結構清晰易懂

## 🚧 遇到的困難和解決方法

### 困難 1：診斷資料流問題

**問題描述**：
- 用戶只報告「訂單不存在」
- 不清楚是創建失敗還是查詢失敗
- 需要逐步追蹤資料流

**解決方法**：
- 審查創建訂單的代碼
- 審查查詢訂單的代碼
- 發現創建和查詢使用不同的資料庫
- 確認是資料同步問題

**經驗**：
- 理解完整的資料流很重要
- 不要假設資料會自動同步
- 使用日誌追蹤資料流向

### 困難 2：設計同步機制

**問題描述**：
- 需要在不影響主流程的情況下同步資料
- 同步失敗不應該導致訂單創建失敗
- 需要確保資料一致性

**解決方法**：
- 將同步邏輯封裝在獨立方法中
- 使用 try-catch 捕獲同步錯誤
- 只記錄日誌，不拋出異常
- 主要資料已經在 Supabase 中
