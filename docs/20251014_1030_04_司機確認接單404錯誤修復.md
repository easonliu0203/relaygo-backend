## 🔧 修復方案

### 修復 1: 改進錯誤處理和日誌

**文件**: `backend/src/routes/bookingFlow-minimal.ts`

**修改前**（第 27-42 行）：
```typescript
// 1. 查詢訂單資料
const { data: booking, error: bookingError } = await supabase
  .from('bookings')
  .select('*')
  .eq('id', bookingId)
  .single();

if (bookingError || !booking) {
  console.error('[API] 查詢訂單失敗:', bookingError);
  res.status(404).json({
    success: false,
    error: '訂單不存在'
  });
  return;
}
```

**修改後**（第 28-64 行）：
```typescript
// 1. 查詢訂單資料
let booking;
let bookingError;

try {
  const result = await supabase
    .from('bookings')
    .select('*')
    .eq('id', bookingId)
    .single();
  
  booking = result.data;
  bookingError = result.error;
} catch (error: any) {
  console.error('[API] ❌ Supabase 查詢異常:', {
    message: error.message,
    stack: error.stack,
    cause: error.cause,
    code: error.code,
    errno: error.errno,
    syscall: error.syscall
  });
  res.status(500).json({
    success: false,
    error: 'Supabase 連接失敗，請檢查網絡或配置'
  });
  return;
}

if (bookingError || !booking) {
  console.error('[API] 查詢訂單失敗:', bookingError);
  res.status(404).json({
    success: false,
    error: '訂單不存在'
  });
  return;
}
```

**改進點**：
1. ✅ 添加 `try-catch` 捕獲異常
2. ✅ 詳細記錄錯誤信息（message, stack, cause, code, errno, syscall）
3. ✅ 區分「連接失敗」（500）和「訂單不存在」（404）
4. ✅ 返回更準確的錯誤訊息

## 🎯 根本原因總結

### 問題根源

**不是訂單不存在，而是 Supabase 連接失敗！**

1. **錯誤處理不當**：
   - Supabase 查詢拋出 `TypeError: fetch failed` 異常
   - 異常被 catch 後只記錄錯誤，然後返回 404
   - 用戶看到的錯誤訊息是「訂單不存在」，而不是真實的「連接失敗」

2. **日誌不足**：
   - 原始代碼只記錄 `bookingError`
   - 沒有捕獲和記錄異常的詳細信息
   - 無法診斷真實問題

3. **錯誤碼不準確**：
   - 連接失敗應該返回 500（服務器錯誤）
   - 但代碼返回 404（資源不存在）
   - 誤導用戶和開發者

### 為什麼重啟後就好了？

**推測原因**：
1. **舊進程問題**：
   - 舊的 Backend 進程可能有網絡連接問題
   - 或者 Supabase 客戶端初始化失敗
   - 重啟後重新初始化，問題解決

2. **環境變數問題**：
   - 舊進程可能沒有正確載入 `.env` 文件
   - 或者 `SUPABASE_URL` / `SUPABASE_SERVICE_ROLE_KEY` 為空
   - 重啟後重新載入環境變數

3. **Node.js 內部問題**：
   - undici（Node.js 內建 fetch）可能有暫時性問題
   - 重啟後重新初始化網絡堆疊

## 💡 經驗教訓

### 1. 錯誤處理的重要性

**錯誤**：
```typescript
const { data, error } = await supabase.from('table').select();
if (error) {
  return res.status(404).json({ error: '資源不存在' });
}
```

**正確**：
```typescript
let data, error;
try {
  const result = await supabase.from('table').select();
  data = result.data;
  error = result.error;
} catch (exception) {
  console.error('連接異常:', exception);
  return res.status(500).json({ error: '服務器錯誤' });
}

if (error) {
  console.error('查詢錯誤:', error);
  return res.status(404).json({ error: '資源不存在' });
}
```

### 2. 日誌的重要性

**不足**：
```typescript
console.error('[API] 查詢失敗:', error);
```

**充分**：
```typescript
console.error('[API] ❌ 查詢異常:', {
  message: error.message,
  stack: error.stack,
  cause: error.cause,
  code: error.code,
  errno: error.errno,
  syscall: error.syscall
});
```

### 3. HTTP 狀態碼的準確性

| 狀態碼 | 含義 | 使用場景 |
|--------|------|----------|
| 200 | 成功 | 請求成功 |
| 400 | 客戶端錯誤 | 請求參數錯誤 |
| 404 | 資源不存在 | 訂單不存在 |
| 500 | 服務器錯誤 | 數據庫連接失敗 |
| 503 | 服務不可用 | 外部服務不可用 |

### 4. 診斷流程

1. ✅ **檢查日誌** - 查看 Backend 終端機輸出
2. ✅ **測試連接** - 驗證外部服務是否可訪問
3. ✅ **添加日誌** - 增加詳細的錯誤記錄
4. ✅ **重現問題** - 使用相同的請求重現錯誤
5. ✅ **修復驗證** - 確認修復後問題解決
