## 📋 需求分析

用戶要求實作包車服務管理系統的價格配置和封測階段自動支付功能：

### 價格配置需求
- **8-9人座車型價格表**
  - 8小時包車：原價$85，優惠價$75，超時費$8/小時
  - 6小時包車：原價$70，優惠價$60，超時費$8/小時
- **3-4人座車型價格表**
  - 8小時包車：原價$60，優惠價$50，超時費$5/小時
  - 6小時包車：原價$50，優惠價$40，超時費$5/小時

### 封測階段特殊需求
- 新訂單自動從「待支付」轉為「已支付」
- 僅封測階段啟用，正式上線前移除
- 確保自動支付後能正常進入派單流程

## 🔍 技術分析過程

### 1. 現有架構分析
通過檢查現有資料庫結構發現：
- `bookings` 表已有基本價格欄位：`base_price`, `hourly_rate`, `total_amount`, `deposit_amount`
- `system_settings` 表可用於儲存價格配置
- `vehicle_type` 使用 A, B, C, D 區分車型

### 2. 設計決策
- 使用 JSONB 格式儲存價格配置，提供靈活性
- 建立 PostgreSQL 函數進行價格計算
- 使用觸發器實作自動支付功能
- 建立完整的管理介面

export class PricingService {
  // 獲取價格配置
  static async getPricingConfig(): Promise<PricingConfig> {
    const response = await ApiService.get('/api/admin/system-settings?key=pricing_config');
    return response.data.value;
  }

  // 計算訂單價格
  static async calculatePrice(
    vehicleType: string,
    duration: number,
    useDiscount: boolean = false

    const response = await ApiService.post('/api/admin/calculate-price', {
      vehicle_type: vehicleType,

## 🚨 遇到的困難和解決方法

### 1. TypeScript 類型問題
**困難**: 模擬服務的用戶類型與介面不匹配
**解決**: 修正 MOCK_USERS 的類型定義，使用正確的欄位名稱
**教訓**: 在建立模擬資料時要嚴格遵循類型定義

### 2. 價格計算複雜性
**困難**: 需要支援多種車型、時長、優惠的組合計算
**解決**: 建立統一的計算函數，使用配置驅動的方式
**教訓**: 複雜的業務邏輯需要良好的抽象設計

### 3. 前端狀態管理
**困難**: 多步驟表單的狀態管理
**解決**: 使用 Steps 組件和狀態機模式
**教訓**: 複雜的用戶流程需要清晰的狀態管理
