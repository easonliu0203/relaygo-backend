- **自動化方式**：`supabase db push`

- **用戶操作**：執行部署腳本即可

#### 步驟 3：部署 Edge Functions
- **自動化方式**：`supabase functions deploy`
- **腳本支援**：✅ 已包含在 `deploy.sh` 和 `deploy.ps1` 中
- **用戶操作**：執行部署腳本即可

#### 步驟 5：更新 Firestore 安全規則
- **自動化方式**：`firebase deploy --only firestore:rules`
- **腳本支援**：✅ 已包含在 `deploy.sh` 和 `deploy.ps1` 中
- **用戶操作**：執行部署腳本即可（需要先安裝 Firebase CLI）

### 必須手動操作的步驟 ⚠️

- **無法自動化**：需要用戶登入 Firebase 和 Supabase Dashboard
- **提供的協助**：
  - ✅ 詳細的 UI 操作路徑（`MANUAL_STEPS_GUIDE.md`）

- **原因**：需要在 Supabase Dashboard 的 SQL Editor 中執行 SQL
- **無法自動化**：Supabase CLI 目前不支援直接執行 SQL

## 💡 關鍵設計決策

### 1. 為什麼不能完全自動化？

**步驟 2（環境變數）**：
- 需要從 Firebase Console 獲取敏感憑證
- 需要用戶登入並授權
- 無法通過 CLI 自動獲取

**步驟 4（Cron Job）**：
- Supabase CLI 目前不支援直接執行 SQL
- 需要在 Dashboard 的 SQL Editor 中執行
- 可以提供 SQL 腳本，但無法自動執行

### 2. 為什麼創建兩個部署腳本？

- Windows 用戶通常使用 PowerShell
- Linux/macOS 用戶通常使用 Bash
- 提供兩個版本確保所有用戶都能使用

### 3. 為什麼創建多個文檔？

- **DEPLOYMENT_GUIDE.md**：完整參考，適合首次部署
- **MANUAL_STEPS_GUIDE.md**：專注於手動步驟，適合快速查閱
- **DEPLOYMENT_CHECKLIST.md**：追蹤進度，適合打印使用
- **DEPLOYMENT_STATUS.md**：快速了解狀態，適合決策
