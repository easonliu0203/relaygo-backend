# 訂單狀態同步問題診斷和修復

## 🐛 問題描述

### 問題 1：客戶端未顯示「待司機確認」狀態
- **預期**：公司端手動派單後，客戶端應該顯示「待司機確認」（`awaitingDriver`，淺橙色）
- **實際**：客戶端仍然顯示「待配對」（`pending`，橙色）

### 問題 2：司機端「確認接單」按鈕消失
- **預期**：司機端訂單詳情頁面應該顯示「確認接單」按鈕
- **實際**：「確認接單」按鈕不見了

---

## 🔍 根本原因分析

### 原因 1：Edge Function 部署時機問題

**時間線**：
1. 用戶創建訂單並支付訂金（Supabase 狀態：`paid_deposit`）
2. 公司端手動派單（Supabase 狀態：`matched`）
3. **舊版本 Edge Function** 處理 outbox 記錄，將 Firestore 狀態設為 `"pending"`
4. 我們修改了 Edge Function 的狀態映射邏輯（`'matched': 'awaitingDriver'`）
5. 重新部署 Edge Function
6. **但是**：outbox 記錄已經被處理過了（`processed = true`），不會再次觸發

**結果**：Firestore 中的狀態仍然是舊的 `"pending"`，而不是新的 `"awaitingDriver"`

### 原因 2：Flutter APP 未重新啟動

即使 Freezed 程式碼已經重新生成，Flutter APP 也需要完全重新啟動（不是熱重載）才能載入新的枚舉值。

## 🔧 待執行的修復步驟

### 步驟 3：重新觸發 Outbox 記錄

由於舊的 outbox 記錄已經被處理過了，我們需要重新觸發這些記錄，讓新版本的 Edge Function 重新處理它們。

#### 方法 1：使用 Supabase SQL Editor（推薦）

1. 打開 Supabase Dashboard
2. 進入 SQL Editor
3. 執行以下 SQL：

```sql
-- 查看當前 outbox 表中 status = 'matched' 的記錄
SELECT id, aggregate_id, event_type, payload->>'status' as status, processed, created_at
FROM outbox
WHERE payload->>'status' = 'matched'
ORDER BY created_at DESC
LIMIT 10;

-- 重新觸發這些記錄（將 processed 設為 false）
UPDATE outbox
SET processed = false
WHERE payload->>'status' = 'matched'
  AND processed = true;

-- 驗證更新結果
SELECT COUNT(*) as reprocessed_count
FROM outbox
WHERE payload->>'status' = 'matched'
  AND processed = false;
```

#### 方法 2：手動觸發 Edge Function（備選）

如果方法 1 不可行，可以手動調用 Edge Function：


## 🚨 常見問題

### Q1：為什麼 Firestore 的狀態沒有更新？

**A1**：可能的原因：
1. Edge Function 沒有重新部署 → 執行步驟 1
2. Outbox 記錄已經被處理過了 → 執行步驟 3
3. Edge Function 執行時出現錯誤 → 檢查 Supabase Dashboard 的 Edge Function 日誌

### Q2：為什麼司機端「確認接單」按鈕不見了？

**A2**：可能的原因：
1. Flutter APP 沒有重新啟動 → 執行步驟 4
2. Firestore 的狀態仍然是 `"pending"` 而不是 `"awaitingDriver"` → 執行步驟 3
3. 按鈕顯示條件錯誤 → 檢查 `driver_order_detail_page.dart` 第 411 行

### Q3：如何確認 Edge Function 是否正確執行？

**A3**：
1. 檢查 outbox 表的 `processed` 欄位是否為 `true`
2. 檢查 Firestore 的 `status` 欄位是否已更新
3. 檢查 Supabase Dashboard 的 Edge Function 日誌（如果有錯誤會顯示）
