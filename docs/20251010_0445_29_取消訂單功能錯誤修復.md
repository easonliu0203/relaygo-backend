# 取消訂單功能錯誤修復

**日期**: 2025-10-10  
**時間**: 04:45  
**編號**: 29  
**主旨**: 修復取消訂單功能中 cancelled_by 欄位不存在的錯誤  
**狀態**: ✅ 已完成

---

## 📋 問題描述

### 當前狀況

1. 在「所有訂單」頁面（`web-admin/src/app/orders/page.tsx`）點擊「取消訂單」按鈕後，取消訂單失敗
2. 在「待處理訂單」頁面（`web-admin/src/app/orders/pending/page.tsx`）點擊「取消訂單」按鈕後，取消訂單失敗

### 錯誤訊息

**終端機顯示**:
```
❌ 更新訂單失敗: {
  code: 'PGRST204',
  details: null,
  hint: null,
  message: "Could not find the 'cancelled_by' column of 'bookings' in the schema cache"
}
```

### 根本原因

- 資料庫 `bookings` 表中不存在 `cancelled_by` 欄位
- 取消訂單 API（`web-admin/src/app/api/admin/bookings/[id]/cancel/route.ts`）嘗試更新不存在的欄位
- 前端代碼傳遞了 `cancelledBy` 參數，但資料庫無法接受

---

## 🔍 問題診斷過程

### 1. 錯誤分析

**錯誤代碼**: `PGRST204`  
**錯誤訊息**: "Could not find the 'cancelled_by' column of 'bookings' in the schema cache"

**分析**:
- PostgREST 錯誤代碼 PGRST204 表示嘗試訪問不存在的欄位
- 錯誤明確指出 `cancelled_by` 欄位在 `bookings` 表中不存在

### 2. 代碼檢查

**取消訂單 API** (`web-admin/src/app/api/admin/bookings/[id]/cancel/route.ts`):

```typescript
// 問題代碼（第 100-102 行）
if (cancelledBy) {
  updateData.cancelled_by = cancelledBy;
}
```

**前端代碼** (`web-admin/src/app/orders/pending/page.tsx` 和 `page.tsx`):

```typescript
// 問題代碼
body: JSON.stringify({ 
  reason: cancelReason.trim(),
  cancelledBy: 'admin' // TODO: 使用實際的管理員 ID
}),
```

### 3. 資料庫檢查

檢查 `bookings` 表的結構，確認以下欄位存在：
- ✅ `status` - 訂單狀態
- ✅ `cancelled_at` - 取消時間
- ✅ `cancellation_reason` - 取消原因
- ✅ `updated_at` - 更新時間
- ❌ `cancelled_by` - **不存在**

---

## 🔧 修復方案

### 方案選擇

**方案 1: 移除 cancelled_by 欄位（推薦）** ✅
- 修改取消訂單 API，移除對 `cancelled_by` 欄位的更新
- 只保留必要的欄位：`status`, `cancelled_at`, `cancellation_reason`, `updated_at`
- 優點：簡單、不需要修改資料庫結構
- 缺點：無法記錄取消者

**方案 2: 添加 cancelled_by 欄位到資料庫**
- 創建資料庫遷移腳本，添加 `cancelled_by` 欄位到 `bookings` 表
- 欄位類型：`UUID REFERENCES users(id)`
- 優點：可以記錄取消者
- 缺點：需要修改資料庫結構，更複雜

**選擇**: 方案 1（推薦）

## 💡 開發心得和經驗總結

### 1. 資料庫欄位檢查的重要性

**學習要點**:
- 在開發 API 時，應該先檢查資料庫表結構
- 確認所有要更新的欄位都存在於資料庫中
- 避免假設欄位存在而直接使用

**最佳實踐**:
```typescript
// ❌ 不好的做法：假設欄位存在
if (cancelledBy) {
  updateData.cancelled_by = cancelledBy;
}

// ✅ 好的做法：只更新確認存在的欄位
const updateData = {
  status: 'cancelled',
  cancelled_at: new Date().toISOString(),
  cancellation_reason: reason.trim(),
  updated_at: new Date().toISOString(),
};
```

### 2. 錯誤訊息的解讀

**學習要點**:
- PostgREST 錯誤代碼 PGRST204 表示欄位不存在
- 錯誤訊息通常會明確指出問題所在
- 仔細閱讀錯誤訊息可以快速定位問題

**錯誤訊息分析**:
```
"Could not find the 'cancelled_by' column of 'bookings' in the schema cache"
                      ^^^^^^^^^^^^^^        ^^^^^^^^
                      欄位名稱              表名稱
```

### 3. 簡化代碼的好處

**學習要點**:
- 移除不必要的欄位可以簡化代碼
- 減少潛在的錯誤來源
- 提高代碼的可維護性

**代碼簡化**:
```typescript
// 修改前：使用 any 類型，動態添加欄位
const updateData: any = {
  status: 'cancelled',
  // ...
};
if (cancelledBy) {
  updateData.cancelled_by = cancelledBy;
}

// 修改後：固定結構，類型安全
const updateData = {
  status: 'cancelled',
  cancelled_at: new Date().toISOString(),
  cancellation_reason: reason.trim(),
  updated_at: new Date().toISOString(),
};
```

### 4. 前後端一致性

**學習要點**:
- 前端和後端的參數應該保持一致
- 移除後端不需要的參數時，也要更新前端代碼
- 確保 API 文檔與實際實現一致

### 困難 1: 確定問題根源

**問題**: 錯誤訊息指向資料庫欄位，但不確定是資料庫問題還是代碼問題

**解決**:
1. 檢查錯誤訊息，確認是欄位不存在的問題
2. 查看資料庫表結構，確認欄位確實不存在
3. 決定修復方案（添加欄位 vs 移除代碼）

### 困難 2: 選擇修復方案

**問題**: 有兩種修復方案，需要選擇最合適的

**解決**:
1. 評估兩種方案的優缺點
2. 考慮實際需求（是否真的需要記錄取消者）
3. 選擇更簡單、風險更低的方案（方案 1）

### 困難 3: 確保完整修復

**問題**: 需要確保所有相關代碼都被修復

**解決**:
1. 使用搜索功能查找所有使用 `cancelledBy` 的地方
2. 逐一修改所有相關文件
3. 運行語法檢查確保沒有遺漏

## 📝 犯過的錯誤和教訓

### 錯誤 1: 假設欄位存在

**錯誤**:
- 在開發時假設 `cancelled_by` 欄位存在
- 沒有檢查資料庫表結構就直接使用

**教訓**:
- 開發前應該先檢查資料庫表結構
- 確認所有要使用的欄位都存在
- 或者創建遷移腳本添加新欄位

### 錯誤 2: 使用 any 類型

**錯誤**:
```typescript
const updateData: any = {
  // ...
};
```

**教訓**:
- 避免使用 `any` 類型
- 使用明確的類型定義
- 提高類型安全性

### 錯誤 3: 前後端不一致

**錯誤**:
- 後端接受 `cancelledBy` 參數但無法使用
- 前端傳遞了不必要的參數

**教訓**:
- 保持前後端參數一致
- 移除不必要的參數
- 更新 API 文檔
