# 診斷：客戶端開始行程狀態同步問題

**問題描述**: 客戶點擊「開始行程」後，訂單狀態跳回「待配對」(pending) 而不是「進行中」(inProgress)

**創建時間**: 2025-10-15

---

## 🔍 診斷步驟

### 步驟 1: 檢查 Backend API 是否正確更新 Supabase

在 Supabase SQL Editor 中執行：

```sql
-- 查看最近更新的訂單
SELECT 
  id,
  booking_number,
  status,
  actual_start_time,
  actual_end_time,
  updated_at
FROM bookings
ORDER BY updated_at DESC
LIMIT 10;
```

**預期結果**: 
- 點擊「開始行程」後，`status` 應該是 `trip_started`
- `actual_start_time` 應該有值

**如果狀態不是 `trip_started`**:
- 檢查 Backend 日誌，查看 API 調用是否成功
- 檢查是否有錯誤訊息

### 步驟 2: 檢查 Outbox 表是否記錄事件

```sql
-- 查看最近的 outbox 事件
SELECT 
  id,
  aggregate_type,
  aggregate_id,
  event_type,
  payload->>'status' as status,
  payload->>'bookingNumber' as booking_number,
  created_at,
  processed_at
FROM outbox
WHERE aggregate_type = 'booking'
ORDER BY created_at DESC
LIMIT 10;
```

**預期結果**:
- 應該看到一個 `event_type = 'updated'` 的事件
- `payload->>'status'` 應該是 `trip_started`
- `processed_at` 應該有值（表示已被 Edge Function 處理）

**如果沒有事件**:
- 檢查 Trigger 是否正常工作
- 執行以下 SQL 檢查 Trigger：

```sql
SELECT 
  tgname as trigger_name,
  proname as function_name,
  tgenabled as enabled
FROM pg_trigger t
JOIN pg_proc p ON t.tgfoid = p.oid
WHERE tgname = 'bookings_outbox_trigger';
```

### 步驟 3: 檢查 Edge Function 日誌

1. 打開 Supabase Dashboard
2. 進入 Edge Functions → sync-to-firestore
3. 查看 Logs

**查找關鍵日誌**:
```
[狀態映射] Supabase 狀態: trip_started
[狀態映射] Firestore 狀態: inProgress
✅ 雙寫成功: orders_rt/{bookingId} 和 bookings/{bookingId}
```

**如果看到錯誤**:
- 記錄錯誤訊息
- 檢查 Firebase Service Account 配置
- 檢查 Firestore 安全規則

### 步驟 4: 檢查 Firestore 實際狀態

1. 打開 Firebase Console
2. 進入 Firestore Database
3. 查看 `orders_rt/{bookingId}` 文檔

**檢查 `status` 欄位**:
- 應該是 `inProgress`
- 如果是 `pending`，說明狀態映射有問題

**檢查 `bookings/{bookingId}` 文檔**:
- 也應該是 `inProgress`
- 兩個集合的狀態應該一致

## 🐛 可能的根本原因

### 原因 1: 狀態映射鍵值不匹配

**問題**: `bookingData.status` 的值可能不是 `trip_started`，而是其他格式

**診斷方法**:
1. 查看 Edge Function 日誌中的 `[狀態映射] Supabase 狀態:` 輸出
2. 如果輸出不是 `trip_started`，說明 payload 中的狀態欄位有問題

**解決方案**:
- 檢查 Trigger 函數中的 payload 構建邏輯
- 確認 `'status', NEW.status` 正確

### 原因 2: Edge Function 未部署最新代碼

**問題**: Edge Function 仍在使用舊的狀態映射邏輯

**診斷方法**:
1. 檢查 Edge Function 的部署時間
2. 確認最新的代碼變更是否已部署

**解決方案**:
```bash
# 重新部署 Edge Function
cd supabase
supabase functions deploy sync-to-firestore
```

### 原因 3: Firestore 安全規則阻止寫入

**問題**: Firestore 安全規則可能阻止了狀態更新

**診斷方法**:
1. 查看 Edge Function 日誌中的錯誤訊息
2. 檢查 Firestore 安全規則

**解決方案**:
- 確認 Service Account 有寫入權限
- 檢查 Firestore 安全規則是否允許更新 status 欄位

### 原因 4: 多次同步覆蓋狀態

**問題**: Edge Function 可能處理了多個事件，後面的事件覆蓋了前面的狀態

**診斷方法**:
```sql
-- 查看同一訂單的所有 outbox 事件
SELECT 
  id,
  event_type,
  payload->>'status' as status,
  created_at,
  processed_at
FROM outbox
WHERE aggregate_id = '{BOOKING_ID}'
ORDER BY created_at DESC;
```

**解決方案**:
- 確保只有最新的事件被處理
- 檢查是否有重複的 Trigger 執行

## 🔧 臨時修復方案

如果無法立即修復，可以使用以下臨時方案：

### 方案 1: 手動更新 Firestore

在 Firebase Console 中手動更新 `orders_rt/{bookingId}` 文檔：
- 將 `status` 欄位改為 `inProgress`

### 方案 2: 重新觸發同步

在 Supabase SQL Editor 中執行：

```sql
-- 手動觸發訂單更新（會重新寫入 outbox）
UPDATE bookings
SET updated_at = NOW()
WHERE id = '{BOOKING_ID}';
```

等待 5-30 秒，Edge Function 會重新同步。
