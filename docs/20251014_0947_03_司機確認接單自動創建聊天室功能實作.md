## 📋 需求背景

### 問題描述

司機端確認接單功能已修復完成（載入對話框不再卡住），但聊天室需要手動創建，無法自動顯示在客戶端和司機端的聊天頁面中。

### 業務需求

當司機確認接單成功後：
1. ✅ Backend API 自動創建聊天室並寫入 Firestore
2. ✅ 客戶端和司機端的聊天頁面自動顯示新創建的聊天室
3. ✅ 雙方可以在聊天室中即時發送和接收訊息
4. ✅ 發送系統歡迎訊息到聊天室

### 架構設計原則

採用**混合架構**，根據操作特性選擇不同的處理方式：

#### 1. 聊天室創建（Creation）- Backend API 處理

**原因**: 涉及核心業務邏輯，需要安全性優先

**優勢**:
- ✅ **業務流程連動**: 創建聊天室是訂單狀態變更的一部分，需要與 Supabase/Postgres 中的訂單 ID、用戶 ID 等數據進行嚴格驗證和綁定
- ✅ **資料一致性**: 確保只有在 Postgres 訂單狀態確認後，Firestore 才寫入對應的聊天室，保證雙資料庫同步
- ✅ **權限控制**: 避免客戶端或司機端透過惡意請求隨意創建聊天室，後端使用高權限 Service Account 寫入

#### 2. 聊天訊息發送（Messaging）- 客戶端直接寫入 Firestore

**原因**: 追求即時性和高效能，性能優先

**優勢**:
- ✅ **高頻寫入**: 聊天訊息是高頻操作，直連 Firestore 避免 API 延遲和伺服器負載
- ✅ **輕量化安全規則**: 使用 Firestore Security Rules 控制權限
- ✅ **即時傳輸**: 最大化利用 Firebase 的即時同步和擴展能力

```typescript
export async function createChatRoomInFirestore(chatRoomData: {
  bookingId: string;
  customerId: string;
  driverId: string;
  customerName?: string;
  driverName?: string;
  pickupAddress?: string;
  bookingTime?: string;
}): Promise<string> {
  const firestore = getFirestore();
  const { bookingId } = chatRoomData;

  const chatRoom = {
    bookingId,
    customerId: chatRoomData.customerId,
    driverId: chatRoomData.driverId,
    customerName: chatRoomData.customerName || '客戶',
    driverName: chatRoomData.driverName || '司機',
    pickupAddress: chatRoomData.pickupAddress || '',
    bookingTime: chatRoomData.bookingTime 
      ? admin.firestore.Timestamp.fromDate(new Date(chatRoomData.bookingTime))
      : admin.firestore.Timestamp.now(),
    lastMessage: null,
    lastMessageTime: null,
    customerUnreadCount: 0,
    driverUnreadCount: 0,
    createdAt: admin.firestore.Timestamp.now(),
    updatedAt: admin.firestore.Timestamp.now(),
  };

  await firestore.collection('chat_rooms').doc(bookingId).set(chatRoom);
  return bookingId;
}
```
