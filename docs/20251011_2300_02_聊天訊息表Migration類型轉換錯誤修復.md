## 📋 問題描述

在 Supabase SQL Editor 中執行聊天訊息表的 migration 文件（`supabase/migrations/20251011_create_chat_messages_table.sql`）時，遇到了 PostgreSQL 類型轉換錯誤。

### 錯誤訊息

```
ERROR: 42883: operator does not exist: uuid = character varying
HINT: No operator matches the given name and argument types. You might need to add explicit type casts.
```

### 錯誤原因

PostgreSQL 不允許直接比較 UUID 類型和 VARCHAR 類型。在 `chat_messages_to_outbox()` trigger 函數中，當將 UUID 類型的值插入到 JSONB 對象時，沒有顯式轉換為 TEXT 類型，導致類型不匹配錯誤。

---

## 🔍 問題診斷

### 問題位置 1: RLS 策略中的 auth.uid() 使用錯誤 ⚠️ **根本原因**

**文件**: `supabase/migrations/20251011_create_chat_messages_table.sql`
**行號**: Line 60-87

**問題代碼**:
```sql
CREATE POLICY "Users can view their own messages"
  ON chat_messages
  FOR SELECT
  USING (
    auth.uid() IN (  -- ❌ auth.uid() 返回 UUID 類型
      SELECT firebase_uid FROM users WHERE id = sender_id  -- firebase_uid 是 VARCHAR 類型
      UNION
      SELECT firebase_uid FROM users WHERE id = receiver_id
    )
  );
```

**問題分析**:
- `auth.uid()` 在 Supabase 中返回的是 **UUID 類型**（Supabase 內部用戶 ID）
- `firebase_uid` 是 **VARCHAR(128) 類型**（Firebase Authentication UID）
- PostgreSQL 不允許直接比較 UUID 和 VARCHAR 類型
- **正確做法**: 使用 `auth.jwt() ->> 'sub'` 獲取 Firebase UID（TEXT 類型）

### 問題位置 2: booking_data JSONB 對象

**文件**: `supabase/migrations/20251011_create_chat_messages_table.sql`
**行號**: Line 123-141

**問題代碼**:
```sql
SELECT jsonb_build_object(
  'bookingId', b.id,  -- ❌ UUID 類型未轉換
  'customerId', (SELECT firebase_uid FROM users WHERE id = b.customer_id),
  'driverId', (SELECT firebase_uid FROM users WHERE id = b.driver_id),
  -- ...
)
INTO booking_data
FROM bookings b
WHERE b.id = NEW.booking_id;
```

**問題分析**:
- `b.id` 是 UUID 類型
- 直接插入 JSONB 對象時，PostgreSQL 無法自動轉換
- 需要顯式轉換為 TEXT 類型

### 問題位置 3: payload JSONB 對象

**文件**: `supabase/migrations/20251011_create_chat_messages_table.sql`
**行號**: Line 158-172

**問題代碼**:
```sql
jsonb_build_object(
  'id', NEW.id,          -- ❌ UUID 類型未轉換
  'bookingId', NEW.booking_id,  -- ❌ UUID 類型未轉換
  'senderId', sender_firebase_uid,
  'receiverId', receiver_firebase_uid,
  -- ...
)
```

**問題分析**:
- `NEW.id` 和 `NEW.booking_id` 都是 UUID 類型
- 需要顯式轉換為 TEXT 類型

## 🔧 修復方案

### 修復 1: RLS 策略 - 使用 auth.jwt() ->> 'sub' 替代 auth.uid() ⭐ **關鍵修復**

**修改前**:
```sql
CREATE POLICY "Users can view their own messages"
  ON chat_messages
  FOR SELECT
  USING (
    auth.uid() IN (  -- ❌ 錯誤：auth.uid() 返回 UUID
      SELECT firebase_uid FROM users WHERE id = sender_id
      UNION
      SELECT firebase_uid FROM users WHERE id = receiver_id
    )
  );
```

**修改後**:
```sql
CREATE POLICY "Users can view their own messages"
  ON chat_messages
  FOR SELECT
  USING (
    sender_id IN (  -- ✅ 正確：比較 UUID 類型
      SELECT id FROM users WHERE firebase_uid = auth.jwt() ->> 'sub'  -- ✅ 使用 Firebase UID
    )
    OR
    receiver_id IN (
      SELECT id FROM users WHERE firebase_uid = auth.jwt() ->> 'sub'
    )
  );
```

**修改位置**: Line 60-72

**關鍵變更**:
1. `auth.uid()` → `auth.jwt() ->> 'sub'`（獲取 Firebase UID）
2. `auth.uid() IN (SELECT firebase_uid ...)` → `sender_id IN (SELECT id ...)`（比較 UUID 而不是混合類型）
3. 使用 `OR` 替代 `UNION`（更清晰的邏輯）

**同樣的修復應用於其他兩個策略**:
- Line 74-80: "Users can send messages" 策略
- Line 82-92: "Users can mark messages as read" 策略

## ✅ 修復後的完整代碼

### chat_messages_to_outbox() 函數（關鍵部分）

```sql
CREATE OR REPLACE FUNCTION chat_messages_to_outbox()
RETURNS TRIGGER AS $$
DECLARE
  sender_firebase_uid VARCHAR(128);
  receiver_firebase_uid VARCHAR(128);
  sender_name VARCHAR(200);
  receiver_name VARCHAR(200);
  booking_data JSONB;
BEGIN
  -- 獲取發送者資訊
  SELECT 
    u.firebase_uid,
    TRIM(CONCAT(up.first_name, ' ', up.last_name))
  INTO 
    sender_firebase_uid,
    sender_name
  FROM users u
  LEFT JOIN user_profiles up ON u.id = up.user_id
  WHERE u.id = NEW.sender_id;

  -- 獲取接收者資訊
  SELECT 
    u.firebase_uid,
    TRIM(CONCAT(up.first_name, ' ', up.last_name))
  INTO 
    receiver_firebase_uid,
    receiver_name
  FROM users u
  LEFT JOIN user_profiles up ON u.id = up.user_id
  WHERE u.id = NEW.receiver_id;

  -- 獲取訂單資訊
  SELECT jsonb_build_object(
    'bookingId', b.id::TEXT,  -- ✅ 修復：顯式轉換為 TEXT
    'customerId', (SELECT firebase_uid FROM users WHERE id = b.customer_id),
    'driverId', (SELECT firebase_uid FROM users WHERE id = b.driver_id),
    'customerName', (
      SELECT TRIM(CONCAT(up.first_name, ' ', up.last_name))
      FROM user_profiles up
      WHERE up.user_id = b.customer_id
    ),
    'driverName', (
      SELECT TRIM(CONCAT(up.first_name, ' ', up.last_name))
      FROM user_profiles up
      WHERE up.user_id = b.driver_id
    ),
    'pickupAddress', b.pickup_location,
    'bookingTime', b.start_date || 'T' || b.start_time
  )
  INTO booking_data
  FROM bookings b
  WHERE b.id = NEW.booking_id;

  -- 插入 outbox 事件
  INSERT INTO outbox (
    aggregate_type,
    aggregate_id,
    event_type,
    payload
  ) VALUES (
    'chat_message',
    NEW.id::TEXT,
    CASE
      WHEN TG_OP = 'INSERT' THEN 'created'
      WHEN TG_OP = 'UPDATE' THEN 'updated'
      WHEN TG_OP = 'DELETE' THEN 'deleted'
    END,
    jsonb_build_object(
      'id', NEW.id::TEXT,          -- ✅ 修復：顯式轉換為 TEXT
      'bookingId', NEW.booking_id::TEXT,  -- ✅ 修復：顯式轉換為 TEXT
      'senderId', sender_firebase_uid,
      'receiverId', receiver_firebase_uid,
      'senderName', sender_name,
      'receiverName', receiver_name,
      'messageText', NEW.message_text,
      'translatedText', NEW.translated_text,
      'createdAt', NEW.created_at,
      'readAt', NEW.read_at,
      'bookingData', booking_data
    )
  );
  
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

## 💡 經驗總結

### 學到的教訓

1. **PostgreSQL 類型系統嚴格**:
   - PostgreSQL 不允許隱式轉換 UUID 和 VARCHAR
   - 在 JSONB 中使用 UUID 時，必須顯式轉換為 TEXT
   - 使用 `::TEXT` 或 `CAST(value AS TEXT)` 進行轉換

2. **JSONB 類型轉換規則**:
   - JSONB 只支持基本類型（string, number, boolean, null）
   - UUID、TIMESTAMP 等特殊類型需要轉換為字符串
   - 使用 `jsonb_build_object()` 時要特別注意類型

3. **Trigger 函數調試**:
   - Trigger 函數的錯誤訊息可能不夠明確
   - 需要仔細檢查所有類型轉換
   - 建議在開發時先測試函數，再創建 trigger

### 最佳實踐

1. **顯式類型轉換**:
   ```sql
   -- ❌ 錯誤：隱式轉換
   'id', uuid_column
   
   -- ✅ 正確：顯式轉換
   'id', uuid_column::TEXT
   ```

2. **JSONB 構建**:
   ```sql
   -- 所有 UUID 都轉換為 TEXT
   jsonb_build_object(
     'id', id_column::TEXT,
     'userId', user_id::TEXT,
     'bookingId', booking_id::TEXT
   )
   ```

3. **Timestamp 轉換**:
   ```sql
   -- Timestamp 可以直接使用，PostgreSQL 會自動轉換為 ISO 8601 字符串
   jsonb_build_object(
     'createdAt', created_at,  -- 自動轉換
     'updatedAt', updated_at   -- 自動轉換
   )
   ```
