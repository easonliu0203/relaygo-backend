## 📋 問題描述

### 問題現象

**受影響的頁面**:

無錯誤訊息，但頁面只顯示硬編碼的模擬資料，無法顯示 Supabase 中的真實訂單。

#### 步驟 1: 創建 `/api/admin/bookings` API 端點

**文件**: `web-admin/src/app/api/admin/bookings/route.ts`

**功能**:
- 從 Supabase 查詢訂單
- 支持狀態篩選 (`status`)
- 支持搜尋功能 (`search`)
- 支持日期範圍篩選 (`startDate`, `endDate`)
- 支持分頁 (`limit`, `offset`)
- 格式化訂單資料（客戶、司機、價格等）

**關鍵代碼**:
```typescript
export async function GET(request: NextRequest) {
  const { searchParams } = new URL(request.url);
  const status = searchParams.get('status');
  const search = searchParams.get('search');
  const limit = parseInt(searchParams.get('limit') || '100');
  const offset = parseInt(searchParams.get('offset') || '0');

  const db = new DatabaseService();
  
  let query = db.supabase
    .from('bookings')
    .select(`
      *,
      customer:customer_id (
        id,
        firebase_uid,
        email,
        user_profiles (first_name, last_name, phone)
      ),
      driver:driver_id (
        id,
        firebase_uid,
        email,
        user_profiles (first_name, last_name, phone),
        drivers (vehicle_type, vehicle_plate)
      )
    `, { count: 'exact' })
    .order('created_at', { ascending: false });

  // 狀態篩選
  if (status && status !== 'all') {
    query = query.eq('status', status);
  }

  // 搜尋功能
  if (search) {
    query = query.ilike('booking_number', `%${search}%`);
  }

  // 分頁
  query = query.range(offset, offset + limit - 1);

  const { data: bookings, error, count } = await query;

  // 格式化訂單資料
  const formattedBookings = (bookings || []).map(booking => ({
    id: booking.id,
    bookingNumber: booking.booking_number,
    status: booking.status,
    customer: { /* ... */ },
    driver: booking.driver ? { /* ... */ } : null,
    vehicleType: booking.vehicle_type,
    pickupLocation: booking.pickup_location,
    dropoffLocation: booking.destination,
    // ...
  }));

  return NextResponse.json({
    success: true,
    data: formattedBookings,
    total: count || 0,
    limit,
    offset,
  });
}
```

## 🚧 遇到的困難和解決方法

### 困難 1: Supabase 關聯查詢的資料結構

**問題**: 
- Supabase 的 `.select()` 返回的關聯資料可能是陣列或物件
- `user_profiles` 可能是 `[{...}]` 或 `{...}`

**解決**:
```typescript
const customerProfile = customer?.user_profiles?.[0] || customer?.user_profiles;
```

**教訓**: 
- 需要處理多種可能的資料格式
- 使用可選鏈 (`?.`) 避免錯誤

### 困難 2: 訂單狀態的多樣性

**問題**:
- 訂單狀態有很多種 (`pending`, `pending_payment`, `confirmed`, `assigned`, etc.)
- 不同頁面需要顯示不同狀態的訂單

**解決**:
- 在 API 端點支持狀態篩選
- 在前端頁面傳遞正確的狀態參數

**教訓**:
- 需要清楚了解業務邏輯中的狀態定義
- 統一狀態命名和使用

### 困難 3: 模擬資料的移除

**問題**:
- 原有代碼使用模擬資料
- 需要確保移除後不影響頁面功能

**解決**:
- 逐步替換，先添加 API 調用
- 測試通過後再移除模擬資料
- 添加錯誤處理和空狀態顯示

## ❌ 犯過的錯誤和教訓

### 錯誤 1: 忘記處理空資料情況

**錯誤代碼**:
```typescript
const stats = {
  total: orders.length,  // ❌ 如果 API 失敗，orders 可能是 undefined
};
```

**正確代碼**:
```typescript
const stats = {
  total: total,  // ✅ 使用 API 返回的 total
  pending: orders.filter(o => ...).length,  // ✅ orders 已經確保是陣列
};
```

**教訓**: 
- 始終處理可能的 null/undefined 情況
- 使用 TypeScript 類型檢查

### 錯誤 2: 忘記添加 loading 狀態

**問題**: 
- 初始版本沒有在表格中顯示 loading 狀態
- 用戶不知道資料正在載入

**解決**:
```typescript
<Table
  loading={loading}
  locale={{
    emptyText: loading ? '載入中...' : '暫無訂單資料',
  }}
/>
```

**教訓**: 
- 良好的用戶體驗需要清晰的狀態反饋
- loading、empty、error 狀態都要處理

### 錯誤 3: 統計數據不一致

**問題**:
- 統計卡片使用 `orders.length`
- 但實際總數應該是 API 返回的 `total`（考慮分頁）

**解決**:
```typescript
<Statistic title="總訂單" value={total} />  // ✅ 使用 API 返回的 total
```

**教訓**:
- 分頁情況下，當前頁的資料數量 ≠ 總數量
- 需要使用 API 返回的 count
