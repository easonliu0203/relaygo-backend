**查詢條件**:
- 集合: `orders_rt`（訂單即時鏡像）

## 🔍 診斷過程

### 步驟 1: 檢查 Firestore 安全規則

**文件**: `firebase/firestore.rules`

**發現問題**:
```javascript
// 第 36-53 行（修復前）
match /orders_rt/{orderId} {
  allow read: if request.auth != null
              && (
                !exists(/databases/$(database)/documents/orders_rt/$(orderId))
                ||
                // ❌ 只檢查 customerId，沒有檢查 driverId
                resource.data.customerId == request.auth.uid
              );
  allow write: if false;
}
```

**問題分析**:
- ❌ 安全規則只允許客戶（customerId）讀取訂單
- ❌ 沒有允許司機（driverId）讀取訂單
- ❌ 司機端查詢 `driverId == CMfTxhJFlUVDkosJPyUoJvKjCQk1` 被拒絕

### 步驟 2: 檢查 bookings 集合規則

**文件**: `firebase/firestore.rules`

**發現**:
```javascript
// 第 15-34 行
match /bookings/{bookingId} {
  allow read: if request.auth != null &&
    (
      !exists(/databases/$(database)/documents/bookings/$(bookingId))
      ||
      // ✅ 正確：同時檢查 customerId 和 driverId
      (request.auth.uid == resource.data.customerId ||
       request.auth.uid == resource.data.driverId)
    );
  allow write: if false;
}
```

**對比分析**:
- ✅ `bookings` 集合的規則是正確的
- ❌ `orders_rt` 集合的規則缺少 `driverId` 檢查
- 🎯 需要將 `orders_rt` 規則修改為與 `bookings` 一致

## 🔧 修復方案

### 修復 orders_rt 集合的安全規則

**文件**: `firebase/firestore.rules`

**修改前**（第 36-53 行）:
```javascript
match /orders_rt/{orderId} {
  // 允許用戶讀取自己的訂單
  allow read: if request.auth != null
              && (
                !exists(/databases/$(database)/documents/orders_rt/$(orderId))
                ||
                // ❌ 只檢查 customerId
                resource.data.customerId == request.auth.uid
              );
  allow write: if false;
}
```

**修改後**（第 36-55 行）:
```javascript
match /orders_rt/{orderId} {
  // 允許用戶讀取自己的訂單（客戶或司機）
  allow read: if request.auth != null
              && (
                !exists(/databases/$(database)/documents/orders_rt/$(orderId))
                ||
                // ✅ 同時檢查 customerId 和 driverId
                (resource.data.customerId == request.auth.uid ||
                 resource.data.driverId == request.auth.uid)
              );
  // 禁止客戶端寫入（由 Supabase Edge Function 寫入）
  // 所有寫入操作必須通過 Supabase API
  allow write: if false;
}
```

**關鍵變更**:
- ✅ 添加 `resource.data.driverId == request.auth.uid` 檢查
- ✅ 使用 `||` 運算符允許客戶或司機讀取
- ✅ 保持 `allow write: if false` 確保安全性
- ✅ 添加更詳細的註解說明

## 💡 開發心得

### 1. Firestore 安全規則的重要性

**重點**:
- Firestore 安全規則是資料安全的第一道防線
- 必須同時考慮所有使用者角色（客戶、司機、管理員）
- 使用 `||` 運算符允許多個條件

**最佳實踐**:
```javascript
// ✅ 正確：同時檢查多個角色
allow read: if request.auth != null &&
  (resource.data.customerId == request.auth.uid ||
   resource.data.driverId == request.auth.uid);

// ❌ 錯誤：只檢查一個角色
allow read: if request.auth != null &&
  resource.data.customerId == request.auth.uid;
```

### 2. 處理同步延遲

**問題**:
- Supabase → Firestore 同步有延遲（最多 30 秒）
- 客戶端可能在文檔創建前嘗試讀取

**解決方案**:
```javascript
allow read: if request.auth != null &&
  (
    // 允許讀取不存在的文檔（返回 null）
    !exists(/databases/$(database)/documents/orders_rt/$(orderId))
    ||
    // 文檔存在時檢查權限
    (resource.data.customerId == request.auth.uid ||
     resource.data.driverId == request.auth.uid)
  );
```

### 3. 安全規則測試

**建議**:
- 在部署前使用 Firebase Emulator 測試規則
- 測試所有使用者角色的讀寫權限
- 測試邊界情況（不存在的文檔、null 值等）

**測試指令**:
```bash
# 啟動 Firestore Emulator
firebase emulators:start --only firestore

# 在另一個終端機運行測試
firebase emulators:exec --only firestore "npm test"
```
