## 💡 根本原因分析

### 問題根本原因

**Backend 服務未啟動**

**詳細分析**:
1. Flutter 客戶端嘗試連接 `http://10.0.2.2:3000/api/bookings`
2. Android 模擬器將 `10.0.2.2` 映射到主機的 `localhost`
3. 但主機上沒有進程監聽端口 3000
4. 連接超時（Connection timed out）
5. 拋出 `SocketException`

### 為什麼會發生這個問題？

**可能的原因**:
1. **Backend 進程被意外終止**
   - 用戶可能關閉了 Terminal
   - 系統重啟
   - 進程崩潰

2. **用戶忘記啟動 Backend**
   - 測試時只啟動了 Flutter 應用
   - 沒有啟動 Backend 服務

3. **端口被其他進程佔用**
   - 其他應用佔用了端口 3000
   - Backend 啟動失敗但沒有提示

### 錯誤訊息中的 `port = 433332` 是什麼？

**分析**:
- 這是 Dart 的 `SocketException` 調試輸出的一個 bug 或格式問題
- 實際連接的端口是 3000（從 URI 可以看出）
- `433332` 可能是內部調試信息的錯誤顯示
- **不影響實際問題診斷**

**目的**: 在應用啟動時檢查 Backend 是否可訪問

**實施內容**:
```dart
// 文件：mobile/lib/core/services/health_check_service.dart

class HealthCheckService {
  static const String _healthCheckUrl = 'http://10.0.2.2:3000/health';
  
  /// 檢查 Backend 是否可訪問
  static Future<bool> checkBackendHealth() async {
    try {
      final response = await http.get(
        Uri.parse(_healthCheckUrl),
      ).timeout(const Duration(seconds: 5));
      
      if (response.statusCode == 200) {
        final data = json.decode(response.body);
        return data['status'] == 'OK';
      }
      return false;
    } catch (e) {
      debugPrint('[HealthCheck] Backend 不可訪問: $e');
      return false;
    }
  }
  
  /// 顯示 Backend 不可訪問的提示
  static void showBackendUnavailableDialog(BuildContext context) {
    showDialog(
      context: context,
      builder: (context) => AlertDialog(
        title: const Text('⚠️  Backend 服務不可訪問'),
        content: const Text(
          '無法連接到 Backend 服務。\n\n'
          '請確認：\n'
          '1. Backend 服務已啟動（npm run dev）\n'
          '2. Backend 運行在端口 3000\n'
          '3. 網絡連接正常'
        ),
        actions: [
          TextButton(
            onPressed: () => Navigator.pop(context),
            child: const Text('確定'),
          ),
        ],
      ),
    );
  }
}
```

   - 錯誤：看到錯誤訊息後，首先懷疑 URL 格式問題
   - 教訓：應該先檢查最基本的問題（Backend 是否運行）
   - 改進：診斷時從最簡單的問題開始檢查

2. **被 `port = 433332` 誤導**
   - 錯誤：花時間分析這個奇怪的端口號
   - 教訓：調試輸出可能有 bug，應該專注於實際的錯誤類型
   - 改進：優先分析錯誤類型（SocketException）和 URI

3. **沒有提供自動化的服務啟動方案**
   - 錯誤：用戶需要手動啟動多個服務
   - 教訓：應該提供一鍵啟動腳本
