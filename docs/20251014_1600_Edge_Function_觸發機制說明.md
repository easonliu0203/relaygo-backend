## 1️⃣ 當前觸發機制詳解

### 架構設計：Outbox Pattern

```
┌─────────────┐
│   Bookings  │  1. 資料變更（INSERT/UPDATE）
│    Table    │
└──────┬──────┘
       │
       │ Trigger: bookings_outbox_trigger
       ▼
┌─────────────┐
│   Outbox    │  2. 寫入事件佇列
│    Table    │
└──────┬──────┘
       │
       │ Cron Job: 每 30 秒
       ▼
┌─────────────┐
│ Edge Func   │  3. 消費事件佇列
│sync-to-fire │
│   store     │
└──────┬──────┘
       │
       │ REST API
       ▼
┌─────────────┐
│  Firestore  │  4. 同步到 Firestore
│             │
└─────────────┘
```

### 觸發流程

#### 步驟 1: Database Trigger 監聽變更

**文件**: `supabase/migrations/20251011_update_outbox_trigger_with_user_info.sql`

```sql
-- 創建 Trigger 監聽 bookings 表的 INSERT 和 UPDATE
CREATE TRIGGER bookings_outbox_trigger
AFTER INSERT OR UPDATE ON bookings
FOR EACH ROW
EXECUTE FUNCTION bookings_to_outbox();
```

**觸發條件**:
- ✅ `bookings` 表的 `INSERT` 事件
- ✅ `bookings` 表的 `UPDATE` 事件

**執行動作**:
- 將變更寫入 `outbox` 表（事件佇列）

#### 步驟 2: Outbox 表存儲事件

**表結構**:
```sql
CREATE TABLE outbox (
  id UUID PRIMARY KEY,
  aggregate_type VARCHAR(50),      -- 'booking' 或 'chat_message'
  aggregate_id VARCHAR(255),       -- 訂單 ID
  event_type VARCHAR(50),          -- 'created', 'updated', 'deleted'
  payload JSONB,                   -- 事件資料
  created_at TIMESTAMP,            -- 創建時間
  processed_at TIMESTAMP,          -- 處理時間（NULL = 未處理）
  retry_count INTEGER,             -- 重試次數
  error_message TEXT               -- 錯誤訊息
);
```

**事件範例**:
```json
{
  "id": "uuid-123",
  "aggregate_type": "booking",
  "aggregate_id": "booking-456",
  "event_type": "updated",
  "payload": {
    "id": "booking-456",
    "status": "driver_confirmed",
    "customerId": "firebase-uid-789",
    ...
  },
  "created_at": "2025-10-14T12:00:00Z",
  "processed_at": null,
  "retry_count": 0
}
```

#### 步驟 3: Cron Job 定時觸發 Edge Function

**配置文件**: `supabase/functions/sync-to-firestore/cron.yaml`

```yaml
# 每 30 秒執行一次
- name: sync-orders-to-firestore
  schedule: "*/30 * * * * *"  # Cron 表達式
  function: sync-to-firestore
  description: "同步訂單變更到 Firestore（Outbox Pattern）"
```

**Cron 表達式說明**:
- `*/30` = 每 30 秒
- `*` = 每分鐘
- `*` = 每小時
- `*` = 每天
- `*` = 每月
- `*` = 每星期

**執行動作**:
- 調用 `sync-to-firestore` Edge Function
- 傳遞空的 request body（函數會自動查詢 outbox 表）

#### 步驟 4: Edge Function 消費事件

**文件**: `supabase/functions/sync-to-firestore/index.ts`

**執行邏輯**:
```typescript
// 1. 查詢未處理的事件（批次處理，每次最多 10 個）
const { data: events } = await supabase
  .from('outbox')
  .select('*')
  .is('processed_at', null)      // 未處理
  .lt('retry_count', 3)          // 重試次數 < 3
  .order('created_at', { ascending: true })
  .limit(10);

// 2. 處理每個事件
for (const event of events) {
  if (event.aggregate_type === 'booking') {
    await syncBookingToFirestore(event);
  }
  
  // 3. 標記為已處理
  await supabase
    .from('outbox')
    .update({ processed_at: new Date().toISOString() })
    .eq('id', event.id);
}
```

#### 步驟 5: 同步到 Firestore

**雙寫策略**:
- ✅ 寫入 `orders_rt/{bookingId}` 集合（客戶端即時訂單）
- ✅ 寫入 `bookings/{bookingId}` 集合（完整訂單記錄）

## 2️⃣ 為什麼需要手動測試？

### 可能的原因

#### 原因 1: Cron Job 未啟用

**檢查方法**:
```sql
-- 查詢 Cron Job 狀態
SELECT 
  jobid,
  jobname,
  schedule,
  active,
  command
FROM cron.job
WHERE jobname LIKE '%sync%';
```

**預期結果**:
```
jobname: sync-orders-to-firestore
schedule: */30 * * * * *
active: true  ← 應該是 true
```

**如果 `active = false`**:
- Cron Job 未啟用
- 需要重新部署或手動啟用

#### 原因 2: Cron Job 配置錯誤

**常見錯誤**:
1. **Service Role Key 錯誤**: Cron Job 調用 Edge Function 時需要正確的 API Key
2. **URL 錯誤**: Edge Function 的 URL 不正確
3. **Cron 表達式錯誤**: 執行頻率配置錯誤

#### 原因 3: Edge Function 未部署

**檢查方法**:
- 前往 Supabase Dashboard: https://supabase.com/dashboard/project/vlyhwegpvpnjyocqmfqc/functions
- 檢查 `sync-to-firestore` 是否存在

**如果不存在**:
- 需要重新部署 Edge Function

#### 原因 4: 環境變數未設置

**必需的環境變數**:
- `FIREBASE_SERVICE_ACCOUNT`: Firebase Service Account JSON
- `SUPABASE_URL`: Supabase 專案 URL
- `SUPABASE_SERVICE_ROLE_KEY`: Supabase Service Role Key

## 3️⃣ 如何改為自動觸發？

### 方案 1: 確保 Cron Job 正確配置（推薦）

#### 步驟 1: 檢查 Cron Job 狀態

```sql
-- 執行此查詢檢查 Cron Job
SELECT 
  jobid,
  jobname,
  schedule,
  active,
  command
FROM cron.job
WHERE jobname = 'sync-orders-to-firestore';
```

#### 步驟 2: 如果 Cron Job 不存在，創建它

**文件**: `supabase/setup_cron_jobs_fixed.sql`

```sql
-- 創建同步任務（每 30 秒執行一次）
SELECT cron.schedule(
  'sync-orders-to-firestore',
  '*/30 * * * * *',
  $$
  SELECT extensions.http_post(
    url := 'https://vlyhwegpvpnjyocqmfqc.supabase.co/functions/v1/sync-to-firestore',
    headers := '{"Authorization": "Bearer YOUR_SERVICE_ROLE_KEY", "Content-Type": "application/json"}'::jsonb,
    body := '{}'::jsonb
  ) as request_id;
  $$
);
```

**⚠️ 重要**: 將 `YOUR_SERVICE_ROLE_KEY` 替換為實際的 Service Role Key

**獲取 Service Role Key**:
1. 前往 Supabase Dashboard
2. Settings > API
3. 複製 `service_role` key（secret）

#### 步驟 3: 驗證 Cron Job 執行

```sql
-- 查看最近的 Cron Job 執行記錄
SELECT 
  jobid,
  runid,
  job_pid,
  database,
  username,
  command,
  status,
  return_message,
  start_time,
  end_time
FROM cron.job_run_details
WHERE jobid = (
  SELECT jobid FROM cron.job WHERE jobname = 'sync-orders-to-firestore'
)
ORDER BY start_time DESC
LIMIT 10;
```

### 方案 2: 使用 Database Webhook（替代方案）

**優點**:
- 即時觸發（無需等待 30 秒）
- 更低的延遲

**缺點**:
- 高頻率變更時可能產生大量請求
- 需要額外配置

**配置步驟**:
1. 前往 Supabase Dashboard > Database > Webhooks
2. 創建新 Webhook:
   - **Table**: `outbox`
   - **Events**: `INSERT`
   - **Type**: `HTTP Request`
   - **Method**: `POST`
   - **URL**: `https://vlyhwegpvpnjyocqmfqc.supabase.co/functions/v1/sync-to-firestore`
   - **Headers**: `Authorization: Bearer <SERVICE_ROLE_KEY>`

## 5️⃣ 故障排除

### 問題 1: Outbox 事件未被處理

**症狀**: `processed_at` 一直是 NULL

**可能原因**:
1. Cron Job 未啟用
2. Edge Function 執行失敗
3. Service Role Key 錯誤

**解決方法**:
```sql
-- 1. 檢查 Cron Job 狀態
SELECT * FROM cron.job WHERE jobname = 'sync-orders-to-firestore';

-- 2. 檢查錯誤訊息
SELECT 
  id,
  error_message,
  retry_count
FROM outbox
WHERE processed_at IS NULL
  AND error_message IS NOT NULL
ORDER BY created_at DESC
LIMIT 5;

-- 3. 手動觸發測試
SELECT extensions.http_post(
  url := 'https://vlyhwegpvpnjyocqmfqc.supabase.co/functions/v1/sync-to-firestore',
  headers := '{"Authorization": "Bearer <SERVICE_ROLE_KEY>", "Content-Type": "application/json"}'::jsonb,
  body := '{}'::jsonb
);
```

### 問題 2: Cron Job 執行失敗

**症狀**: Cron Job 執行記錄顯示錯誤

**解決方法**:
```sql
-- 查看執行記錄
SELECT 
  status,
  return_message,
  start_time,
  end_time
FROM cron.job_run_details
WHERE jobid = (SELECT jobid FROM cron.job WHERE jobname = 'sync-orders-to-firestore')
ORDER BY start_time DESC
LIMIT 10;
```

### 問題 3: Edge Function 執行失敗

**症狀**: Edge Function 日誌顯示錯誤

**常見錯誤**:
1. **401 Unauthorized**: Service Role Key 錯誤
2. **500 Internal Server Error**: Edge Function 代碼錯誤
3. **Timeout**: 處理時間過長

**解決方法**:
- 檢查環境變數
- 查看 Edge Function 日誌
- 檢查 Firestore 權限

```sql
-- 定期檢查未處理事件數量
SELECT 
  COUNT(*) FILTER (WHERE processed_at IS NULL) as 未處理,
```
