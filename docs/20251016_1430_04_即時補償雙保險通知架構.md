## 專案背景

### 現有架構

在實施本次改進之前，系統使用單一的 Cron Job 機制進行數據同步：

```
Supabase Trigger (bookings_to_outbox)
    ↓
寫入 outbox 表
    ↓
Cron Job (每 5-30 秒執行)
    ↓
調用 Edge Function (sync-to-firestore)
    ↓
同步到 Firestore
```

**問題**：
- 延遲時間：5-30 秒
- 用戶體驗：訂單狀態更新不夠即時
- 實時性：無法滿足即時通知需求

### 改進目標

建立「即時 + 補償」雙保險機制：
- **即時通知**：1-3 秒內同步（Trigger + pg_net）
- **補償機制**：5-30 秒補償（Cron Job）
- **可控性**：管理後台一鍵開關
- **可靠性**：雙保險，不丟失數據

## 需求分析

### 功能需求

1. **即時通知機制**
   - 當 bookings 表變更時，立即發送 HTTP 請求
   - 使用 pg_net 擴展進行異步 HTTP 調用
   - 目標延遲：1-3 秒

2. **補償機制**
   - 保留現有 Cron Job
   - 處理即時通知失敗的情況
   - 確保數據最終一致性

3. **退場機制**
   - 一鍵停用即時通知
   - 快速退回 Cron Job 模式
   - 不影響數據同步

4. **管理界面**
   - 派單設定頁面
   - 即時同步開關
   - 狀態監控和統計

5. **冪等性保證**
   - Edge Function 可重複執行
   - 不會重複處理同一事件
   - 數據一致性保證

### 非功能需求

1. **性能**
   - 即時通知延遲 < 3 秒
   - 不影響數據庫寫入性能
   - 支持高並發場景

2. **可靠性**
   - 雙保險機制
   - 失敗自動重試
   - 錯誤日誌記錄

3. **可維護性**
   - 清晰的代碼結構
   - 完整的文檔
   - 易於調試和監控

### 核心組件

#### 1. 即時通知 Trigger

**文件**: `supabase/migrations/20251016_create_realtime_sync_trigger.sql`

**功能**:
- 監聽 bookings 表的 INSERT 和 UPDATE 事件
- 使用 pg_net.http_post 發送異步 HTTP 請求
- 失敗不阻塞 Trigger 執行

**關鍵代碼**:
```sql
CREATE OR REPLACE FUNCTION notify_edge_function_realtime()
RETURNS TRIGGER AS $$
DECLARE
  edge_function_url TEXT;
  service_role_key TEXT;
  request_id BIGINT;
BEGIN
  edge_function_url := 'https://vlyhwegpvpnjyocqmfqc.supabase.co/functions/v1/sync-to-firestore';
  service_role_key := current_setting('app.settings.service_role_key', true);
  
  SELECT net.http_post(
    url := edge_function_url,
    headers := jsonb_build_object(
      'Authorization', 'Bearer ' || service_role_key,
      'Content-Type', 'application/json'
    ),
    body := jsonb_build_object(
      'trigger', 'realtime',
      'booking_id', NEW.id,
      'event_type', CASE
        WHEN TG_OP = 'INSERT' THEN 'created'
        WHEN TG_OP = 'UPDATE' THEN 'updated'
        ELSE 'unknown'
      END
    )
  ) INTO request_id;
  
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

## 技術細節

### pg_net 擴展

**特性**:
- 異步 HTTP 請求
- 不阻塞 Trigger 執行
- 自動記錄請求和響應

**使用方式**:
```sql
SELECT net.http_post(
  url := 'https://example.com/api',
  headers := '{"Content-Type": "application/json"}'::jsonb,
  body := '{"key": "value"}'::jsonb
) as request_id;
```

**查詢請求記錄**:
```sql
SELECT * FROM net._http_response 
WHERE id = request_id;
```

### Service Role Key 配置

**方法 1**: 使用 current_setting()
```sql
service_role_key := current_setting('app.settings.service_role_key', true);
```

**方法 2**: 環境變數（需要在 Supabase Dashboard 配置）

### Edge Function 冪等性

現有 Edge Function 已經具備冪等性：
- 只處理 `processed_at IS NULL` 的事件
- 處理後立即標記 `processed_at`
- Firestore 使用 PATCH（upsert）操作

### 錯誤處理

**Trigger 中的錯誤處理**:
```sql
BEGIN
  -- HTTP 請求
EXCEPTION
  WHEN OTHERS THEN
    RAISE WARNING 'HTTP request failed: %', SQLERRM;
    -- 不阻塞 Trigger 執行
END;
```

**API 中的錯誤處理**:
- Try-catch 包裹所有操作
- 返回明確的錯誤信息
- 記錄錯誤日誌

## 問題與解決

### 問題 1: pg_net 擴展未啟用

**症狀**: 執行 migration 時報錯 "extension pg_net does not exist"

**解決方法**:
1. 前往 Supabase Dashboard
2. Database → Extensions
3. 搜尋 "pg_net"
4. 點擊 "Enable"

### 問題 2: Service Role Key 無法獲取

**症狀**: Trigger 執行時報錯 "Unable to get service_role_key"

**解決方法**:
在 Supabase Dashboard 配置環境變數，或在 Trigger 中硬編碼（不推薦）

### 問題 3: HTTP 請求失敗

**症狀**: net._http_response 顯示 status_code 非 200

**排查步驟**:
1. 檢查 Edge Function 是否正常運行
2. 檢查 URL 是否正確
3. 檢查 Authorization header
4. 查看 Edge Function 日誌

### 問題 4: 延遲仍然很高

**可能原因**:
- 即時同步未啟用
- pg_net 請求排隊
- Edge Function 性能問題

**解決方法**:
1. 確認 Trigger 已創建並啟用
2. 檢查 net._http_response 表
3. 優化 Edge Function 代碼
