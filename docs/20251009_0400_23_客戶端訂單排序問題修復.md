## 📋 問題描述

### 問題現象

**受影響的頁面**:
1. 我的訂單 > 進行中訂單頁面
2. 我的訂單 > 歷史訂單頁面

**當前問題**:
- 訂單列表的排序順序不正確
- 可能是按照其他欄位排序，或是由舊到新排序

**預期行為**:
- 訂單應該按照「建立日期」（createdAt）排序
- 排序方向：由新至舊（降序排列）
- 最新建立的訂單顯示在列表最上方
- 最舊的訂單顯示在列表最下方

## 🔍 問題診斷過程

### 1. 檢查客戶端代碼

**文件**: `mobile/lib/core/services/booking_service.dart`

**getUserBookings() 方法** (第 298-312 行):
```dart
Stream<List<BookingOrder>> getUserBookings() {
  if (currentUserId == null) {
    return Stream.value([]);
  }

  return _firestore
      .collection('orders_rt')
      .where('customerId', isEqualTo: currentUserId)
      .orderBy('createdAt', descending: true)  // ✅ 代碼正確
      .snapshots()
      .map((snapshot) => snapshot.docs
          .map((doc) => BookingOrder.fromFirestore(doc))
          .toList());
}
```

**getActiveBookings() 方法** (第 344-362 行):
```dart
Stream<List<BookingOrder>> getActiveBookings() {
  if (currentUserId == null) {
    return Stream.value([]);
  }

  return _firestore
      .collection('orders_rt')
      .where('customerId', isEqualTo: currentUserId)
      .where('status', whereIn: [
        BookingStatus.pending.name,
        BookingStatus.matched.name,
        BookingStatus.inProgress.name,
      ])
      .orderBy('createdAt', descending: true)  // ✅ 代碼正確
      .snapshots()
      .map((snapshot) => snapshot.docs
          .map((doc) => BookingOrder.fromFirestore(doc))
          .toList());
}
```

**發現 1**: 代碼中的排序邏輯是正確的！
- 都使用了 `.orderBy('createdAt', descending: true)`
- 這應該會按照建立日期降序排列（由新至舊）

### 2. 檢查 Firestore 資料同步

**文件**: `supabase/migrations/20250101_create_outbox_table.sql`

**Trigger Function** (第 39-73 行):
```sql
jsonb_build_object(
  'id', NEW.id,
  'bookingNumber', NEW.booking_number,
  'customerId', (SELECT firebase_uid FROM users WHERE id = NEW.customer_id),
  'status', NEW.status,
  ...
  'createdAt', NEW.created_at,  -- ✅ 有包含 createdAt
  'updatedAt', NEW.updated_at,
  ...
)
```

**發現 2**: Firestore 同步邏輯包含 `createdAt` 欄位

### 3. 檢查資料模型

**文件**: `mobile/lib/core/models/booking_order.dart`

**fromFirestore() 方法** (第 114-149 行):
```dart
factory BookingOrder.fromFirestore(DocumentSnapshot doc) {
  final data = doc.data() as Map<String, dynamic>;

  // 解析 createdAt (必填欄位)
  final createdAt = _parseTimestamp(data['createdAt']);  // ✅ 正確解析

  return BookingOrder(
    id: doc.id,
    customerId: data['customerId'] ?? '',
    ...
    createdAt: createdAt,  // ✅ 正確賦值
    ...
  );
}
```

**發現 3**: 資料模型正確解析 `createdAt` 欄位

### 4. 根本原因分析

**問題確認**:
- ✅ 代碼邏輯正確
- ✅ 資料同步正確
- ✅ 資料解析正確
- ❌ **缺少 Firestore 複合索引**

**關鍵發現**:
Firestore 查詢使用了複合條件：
1. `where('customerId', isEqualTo: ...)`
2. `where('status', whereIn: [...])`（僅進行中訂單）
3. `orderBy('createdAt', descending: true)`

**這種複合查詢需要創建 Firestore 複合索引！**

如果沒有索引：
- 查詢可能會失敗
- 或返回錯誤的排序
- 或性能極差

## 💡 開發心得和經驗總結

### 1. Firestore 複合索引的重要性

**學習要點**:
- Firestore 的複合查詢（多個 where + orderBy）需要複合索引
- 沒有索引會導致查詢失敗或性能極差
- 索引配置應該版本化管理（firestore.indexes.json）

**最佳實踐**:
```json
{
  "indexes": [
    {
      "collectionGroup": "collection_name",
      "queryScope": "COLLECTION",
      "fields": [
        { "fieldPath": "field1", "order": "ASCENDING" },
        { "fieldPath": "field2", "order": "DESCENDING" }
      ]
    }
  ]
}
```

### 2. 查詢優化

**查詢模式**:
```dart
// 需要索引: customerId (ASC) + createdAt (DESC)
_firestore
  .collection('orders_rt')
  .where('customerId', isEqualTo: userId)
  .orderBy('createdAt', descending: true)
  .snapshots()

// 需要索引: customerId (ASC) + status (ASC) + createdAt (DESC)
_firestore
  .collection('orders_rt')
  .where('customerId', isEqualTo: userId)
  .where('status', whereIn: ['pending', 'matched'])
  .orderBy('createdAt', descending: true)
  .snapshots()
```

      .collection('orders_rt')
      .where('customerId', isEqualTo: currentUserId)
      .orderBy('createdAt', descending: true)
