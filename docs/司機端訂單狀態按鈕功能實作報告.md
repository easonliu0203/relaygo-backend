# 司機端訂單狀態按鈕與自動通知功能實作報告

## 實作日期
2025-10-15

## 功能概述
在司機確認接單後（訂單狀態為 `driver_confirmed`），在司機端的訂單詳情頁面添加兩個狀態更新按鈕，點擊後會：
1. 更新訂單狀態
2. 在聊天室自動發送通知訊息給客戶
3. 同步狀態到 Firestore

## 實作內容

### 1. Backend API 端點

#### 1.1 司機出發 API
- **路徑**: `POST /api/booking-flow/bookings/:bookingId/depart`
- **文件**: `backend/src/routes/bookingFlow-minimal.ts` (第 266-374 行)
- **功能**:
  1. 驗證司機權限（是否為該訂單的司機）
  2. 檢查訂單狀態（必須為 `driver_confirmed`）
  3. 更新 Supabase `bookings` 表的 `status` 為 `driver_departed`
  4. 寫入 `outbox` 表（觸發 Firestore 同步）
  5. 在聊天室自動發送訊息：「司機已出發，正在前往上車地點 🚗」

**請求範例**:
```json
POST /api/booking-flow/bookings/{bookingId}/depart
{
  "driverUid": "Firebase UID"
}
```

**響應範例**:
```json
{
  "success": true,
  "data": {
    "bookingId": "xxx",
    "status": "driver_departed",
    "nextStep": "driver_arrive"
  },
  "message": "已出發"
}
```

#### 1.2 司機到達 API
- **路徑**: `POST /api/booking-flow/bookings/:bookingId/arrive`
- **文件**: `backend/src/routes/bookingFlow-minimal.ts` (第 376-484 行)
- **功能**:
  1. 驗證司機權限
  2. 檢查訂單狀態（必須為 `driver_departed`）
  3. 更新 Supabase `bookings` 表的 `status` 為 `driver_arrived`
  4. 寫入 `outbox` 表（觸發 Firestore 同步）
  5. 在聊天室自動發送訊息：「司機已到達上車地點，請準備上車 📍」

**請求範例**:
```json
POST /api/booking-flow/bookings/{bookingId}/arrive
{
  "driverUid": "Firebase UID"
}
```

**響應範例**:
```json
{
  "success": true,
  "data": {
    "bookingId": "xxx",
    "status": "driver_arrived",
    "nextStep": "start_trip"
  },
  "message": "已到達"
}
```

### 3. Flutter UI 實作

#### 3.1 司機訂單詳情頁面
- **文件**: `mobile/lib/apps/driver/presentation/pages/driver_order_detail_page.dart`
- **修改位置**: `_buildActionButtons` 方法 (第 502-737 行)

#### 3.2 按鈕顯示邏輯

**按鈕 1: 「出發前往載客 🚗」**
- **顯示條件**: `order.status == BookingStatus.matched`
- **說明**: 
  - 司機確認接單後，Supabase 狀態為 `driver_confirmed`
  - Edge Function 同步到 Firestore 時，映射為 `matched`
  - 點擊後，Supabase 狀態變為 `driver_departed`
  - Edge Function 再次同步，Firestore 狀態變為 `inProgress`
- **按鈕顏色**: 藍色 (`0xFF2196F3`)

**按鈕 2: 「抵達上車地點 📍」**
- **顯示條件**: `order.status == BookingStatus.inProgress`
- **說明**: 
  - 司機出發後，Supabase 狀態為 `driver_departed`
  - Edge Function 同步到 Firestore 時，映射為 `inProgress`
  - 點擊後，Supabase 狀態變為 `driver_arrived`
  - Edge Function 再次同步，Firestore 狀態仍為 `inProgress`
- **按鈕顏色**: 綠色 (`0xFF4CAF50`)

#### 3.3 用戶體驗設計
1. **確認對話框**: 點擊按鈕前會顯示確認對話框，避免誤觸
2. **載入指示器**: API 調用期間顯示載入對話框，防止重複點擊
3. **成功提示**: 操作成功後顯示 SnackBar 提示
4. **錯誤處理**: 操作失敗時顯示錯誤訊息
5. **自動刷新**: 操作成功後自動刷新訂單資料

### 4. 聊天室自動訊息

#### 4.1 系統訊息格式
- **發送者**: `senderId: 'system'`
- **接收者**: `receiverId: 'all'`
- **發送者名稱**: `senderName: '系統'`
- **訊息類型**: 系統訊息（可通過 `senderId === 'system'` 判斷）

#### 4.2 訊息內容
- **出發訊息**: 「司機已出發，正在前往上車地點 🚗」
- **到達訊息**: 「司機已到達上車地點，請準備上車 📍」

#### 4.3 實作方式
- 使用 Firebase Admin SDK 直接寫入 Firestore
- 調用 `sendSystemMessage()` 函數（`backend/src/config/firebase.ts`）
- 同時更新聊天室的 `lastMessage` 和 `lastMessageTime`

### 5. 資料庫狀態

#### 5.1 Supabase 狀態
- `driver_confirmed`: 司機已確認接單
- `driver_departed`: 司機已出發
- `driver_arrived`: 司機已到達

#### 5.2 Firestore 狀態映射
```typescript
const statusMapping = {
  'driver_confirmed': 'matched',    // 已配對
  'driver_departed': 'inProgress',  // 進行中
  'driver_arrived': 'inProgress',   // 進行中
};
```

#### 5.3 狀態流程
```
Supabase: matched → driver_confirmed → driver_departed → driver_arrived
Firestore: pending → matched → inProgress → inProgress
```

### 6. Firestore 同步

#### 6.1 同步機制
- **觸發器**: Supabase `bookings` 表的 `INSERT` 或 `UPDATE` 觸發器
- **Outbox 表**: 寫入事件到 `outbox` 表
- **Edge Function**: `sync-to-firestore` 函數每 5 秒（測試環境）消費 outbox 事件
- **雙寫策略**: 同時寫入 `orders_rt` 和 `bookings` 集合

#### 6.2 同步延遲
- **測試環境**: 最多 5 秒延遲
- **正式環境**: 最多 30 秒延遲（需修改 cron 配置）

## 測試步驟

### 1. 前置條件
1. 確保 Backend 服務正在運行
2. 確保 Supabase Edge Function 正在運行
3. 確保 Firebase Admin SDK 已正確配置
4. 司機已登入 Flutter 應用
5. 訂單狀態為 `matched`（Firestore）或 `driver_confirmed`（Supabase）

### 2. 測試「出發前往載客」功能
1. 打開司機端訂單詳情頁面
2. 確認顯示「出發前往載客 🚗」按鈕（藍色）
3. 點擊按鈕
4. 確認顯示確認對話框
5. 點擊「確認出發」
6. 確認顯示載入指示器
7. 等待 API 響應
8. 確認顯示成功提示：「✅ 已出發！客戶已收到通知」
9. 確認訂單狀態更新（按鈕變為「抵達上車地點 📍」）
10. 打開聊天室，確認收到系統訊息：「司機已出發，正在前往上車地點 🚗」

### 3. 測試「抵達上車地點」功能
1. 確認顯示「抵達上車地點 📍」按鈕（綠色）
2. 點擊按鈕
3. 確認顯示確認對話框
4. 點擊「確認到達」
5. 確認顯示載入指示器
6. 等待 API 響應
7. 確認顯示成功提示：「✅ 已到達！客戶已收到通知」
8. 確認訂單狀態更新
9. 打開聊天室，確認收到系統訊息：「司機已到達上車地點，請準備上車 📍」

### 4. 測試錯誤處理
1. **網路錯誤**: 關閉網路，點擊按鈕，確認顯示錯誤訊息
2. **權限錯誤**: 使用其他司機帳號，確認顯示「無權限操作此訂單」
3. **狀態錯誤**: 在錯誤的訂單狀態下點擊，確認顯示狀態不正確錯誤

### 5. 驗證 Firestore 同步
1. 打開 Firebase Console
2. 查看 `chat_rooms/{bookingId}/messages` 集合
3. 確認系統訊息已正確寫入
4. 查看 `orders_rt/{bookingId}` 文檔
5. 確認訂單狀態已更新為 `inProgress`

## 注意事項

### 1. 狀態判斷限制
由於 Flutter 的 `BookingStatus` 枚舉較簡化，`inProgress` 狀態包含了 `driver_departed` 和 `driver_arrived` 兩個 Supabase 狀態。目前的實作中，當訂單狀態為 `inProgress` 時，只顯示「抵達上車地點」按鈕。

**改進方案**:
1. 在 `BookingOrder` 模型中添加原始 Supabase 狀態欄位
2. 根據原始狀態判斷顯示哪個按鈕
3. 或者擴展 `BookingStatus` 枚舉，添加更多狀態

### 2. 確認對話框
所有狀態更新操作都添加了確認對話框，避免誤觸。如果覺得影響用戶體驗，可以考慮移除。

### 3. Firestore 同步延遲
目前 Edge Function 每 5 秒執行一次（測試環境），正式環境建議改為 30 秒。如果需要更即時的同步，可以考慮：
1. 減少 cron 間隔（增加成本）
2. 使用 Supabase Realtime（需要額外配置）
3. Backend API 直接寫入 Firestore（雙寫策略）

### 4. 系統訊息樣式
目前系統訊息與用戶訊息使用相同的格式，只是 `senderId` 為 `'system'`。建議在聊天室 UI 中添加特殊樣式來區分系統訊息。
